Advanced Concurrency Control in DBMS

Concurrency control is the mechanism that ensures correct results when multiple transactions execute simultaneously in a database system. While basic concurrency concepts (serializability, simple locking) are covered elsewhere, this article dives deep into advanced concurrency control protocols as presented in Chapter 18 of Korth/Silberschatz's Database System Concepts. This is a heavily tested topic in GATE CSE.

1. Review: Serializability

Before discussing concurrency control protocols, let us review the two notions of serializability.

1.1 Conflict Serializability

Two operations conflict if they are from different transactions, access the same data item, and at least one of them is a write.

Operation 1 Operation 2 Conflict?
Read(A)Read(A)No (both reads)
Read(A)Write(A)Yes (read-write)
Write(A)Read(A)Yes (write-read)
Write(A)Write(A)Yes (write-write)
Read(A)Write(B)No (different items)

A schedule is conflict serializable if it can be transformed into a serial schedule by swapping consecutive non-conflicting operations. Equivalently, a schedule is conflict serializable if and only if its precedence graph (conflict graph) is acyclic.

1.2 View Serializability

A schedule S is view serializable if it is view-equivalent to some serial schedule. Two schedules are view-equivalent if:

  1. For each data item, the same transaction performs the initial read in both schedules.
  2. For each data item, if Ti reads a value written by Tj in one schedule, the same holds in the other.
  3. For each data item, the same transaction performs the final write in both schedules.
Relationship: Every conflict-serializable schedule is view-serializable, but NOT vice versa. The additional view-serializable (but not conflict-serializable) schedules involve blind writes. Testing for view serializability is NP-complete.

2. Lock-Based Protocols

Shared (S) lock-S(Q) Read data item Q Compatible with other S locks
Exclusive (X) lock-X(Q) Read and write data item Q Not compatible with any lock

Lock Compatibility Matrix

S X
S Compatible Not Compatible
X Not Compatible Not Compatible

2.2 Two-Phase Locking (2PL)

The Two-Phase Locking protocol is the most widely used locking protocol. It divides a transaction's execution into two phases:

  1. Growing phase: The transaction may acquire locks but may not release any lock.
  2. Shrinking phase: The transaction may release locks but may not acquire any new lock.

The point where the transaction has acquired all its locks is called the lock point.

flowchart LR A3["Transaction Start"] --> B3["Growing Phase: Acquire locks"] B3 --> C3["Lock Point (max locks)"] C3 --> D3["Shrinking Phase: Release locks"] D3 --> E3["Transaction End"] style C3 fill:#f1c40f
2PL Guarantees: 2PL ensures conflict serializability. The serialization order of transactions is determined by their lock points. However, basic 2PL does NOT prevent deadlocks, and does NOT prevent cascading rollbacks.

2.2.1 Variants of Two-Phase Locking

Variant Rule Prevents Cascading Rollback? Ensures Conflict Serializability?
Basic 2PL Locks released in shrinking phase, possibly before commit No Yes
Strict 2PL All exclusive (X) locks held until commit/abort Yes Yes
Rigorous 2PL ALL locks (both S and X) held until commit/abort Yes Yes
Why Strict 2PL prevents cascading rollback: Since exclusive locks are held until commit, no other transaction can read data modified by an uncommitted transaction. Therefore, if a transaction aborts, no other transaction has read its dirty data, so no cascading rollback is needed.

Worked Example: Basic 2PL Allowing Cascading Rollback

T1:  lock-X(A)  Write(A)  unlock(A)                         abort
T2:                            lock-S(A)  Read(A)  ...  commit?
T2 reads A after T1 releases it (shrinking phase of T1). If T1 later aborts, T2 has read a dirty value and must also be rolled back — a cascading rollback. Strict 2PL prevents this because T1 would hold lock-X(A) until it commits or aborts.

2.3 Lock Conversion (Upgrade/Downgrade)

Lock conversion allows a transaction to change the type of lock it holds on a data item:

  • Upgrade: Convert a shared lock (S) to an exclusive lock (X). Allowed only during the growing phase.
  • Downgrade: Convert an exclusive lock (X) to a shared lock (S). Allowed only during the shrinking phase.

This is useful when a transaction initially reads a data item and later decides to modify it. Without lock conversion, the transaction would need to request an X lock upfront (even if it might not write), reducing concurrency.

Lock Conversion in 2PL

Growing Phase:  lock-S(A)  lock-S(B)  upgrade(A to X)
                  ^--- can upgrade here (still growing)
Lock Point
Shrinking Phase:  downgrade(A to S)  unlock(B)  unlock(A)
                  ^--- can downgrade here (now shrinking)
Deadlock Risk with Upgrades: If T1 and T2 both hold S locks on the same item and both try to upgrade to X, neither can proceed (each waits for the other to release the S lock). This is a form of deadlock called conversion deadlock.

3. Graph-Based Protocols (Tree Protocol)

Graph-based protocols exploit knowledge about the order in which data items are accessed. The most well-known is the tree protocol.

3.1 Assumptions

  • A partial ordering exists among the data items (forming a tree/forest).
  • All data items are organized in a rooted tree (e.g., B+ tree index nodes).
  • Only exclusive locks are used.

3.2 Tree Protocol Rules

  1. The first lock by a transaction may be on any data item.
  2. Subsequently, a data item Q can only be locked if the parent of Q is currently locked by the same transaction.
  3. Data items may be unlocked at any time (no two-phase restriction).
  4. A data item that has been locked and unlocked by a transaction cannot be re-locked by the same transaction.
flowchart TB R["A (root)"] --> B["B"] R --> C["C"] B --> D["D"] B --> E["E"] C --> F["F"] C --> G["G"]

Example: Valid vs Invalid Lock Sequences

Sequence Valid? Reason
lock(B), lock(D), lock(E), unlock(D), unlock(B), unlock(E) Valid B is parent of D and E; first lock can be on any node
lock(A), lock(C), lock(G), unlock(A), unlock(C), unlock(G) Valid A is parent of C, C is parent of G
lock(D), lock(B) Invalid To lock B, its parent A must be locked (D is a child of B, not parent)
lock(B), unlock(B), lock(B) Invalid Cannot re-lock an item after unlocking it (rule 4)

3.3 Properties of Tree Protocol

Property Tree Protocol 2PL
Conflict serializable Yes Yes
Deadlock-free Yes No
Prevents cascading rollback No No (basic), Yes (strict/rigorous)
May lock unnecessary items Yes (must lock parent to reach child) No
Recoverability Not guaranteed (needs additional protocol) Not guaranteed (basic), Yes (strict)
GATE Key Point: The tree protocol is deadlock-free but may require transactions to lock data items they do not actually need (to maintain a path from an ancestor). It does NOT ensure recoverability or prevent cascading rollbacks by itself.

4. Timestamp-Based Protocols

Timestamp-based protocols assign a unique timestamp to each transaction when it begins. The timestamp determines the serialization order. No locks are used, so there are no deadlocks.

4.1 Timestamp Ordering Protocol

Each data item Q maintains two timestamps:

  • W-timestamp(Q): The timestamp of the youngest (latest) transaction that successfully wrote Q.
  • R-timestamp(Q): The timestamp of the youngest transaction that successfully read Q.

Rules for Ti with timestamp TS(Ti):

Read(Q) by Ti:

  • If TS(Ti) < W-timestamp(Q): Ti is trying to read a value that was already overwritten by a younger transaction. Reject and rollback Ti.
  • If TS(Ti) ≥ W-timestamp(Q): Allow the read. Set R-timestamp(Q) = max(R-timestamp(Q), TS(Ti)).

Write(Q) by Ti:

  • If TS(Ti) < R-timestamp(Q): A younger transaction has already read the current value. Ti's write would invalidate that read. Reject and rollback Ti.
  • If TS(Ti) < W-timestamp(Q): A younger transaction has already written Q. Ti's write is obsolete. Reject and rollback Ti (or apply Thomas Write Rule).
  • Otherwise: Allow the write. Set W-timestamp(Q) = TS(Ti).

4.2 Thomas Write Rule

The Thomas Write Rule is a modification to the basic timestamp ordering protocol that handles the case where TS(Ti) < W-timestamp(Q) for a write operation.

Thomas Write Rule: If TS(Ti) < W-timestamp(Q), instead of rolling back Ti, simply ignore the write (suppress it) and let Ti continue. The rationale: Ti's write is obsolete anyway since a younger transaction has already written a newer value that overwrites it.

This allows more schedules to succeed without rollback. However, the Thomas Write Rule produces schedules that are view serializable but may NOT be conflict serializable (because it effectively introduces a blind write skip).

Worked Example: Thomas Write Rule

TS(T1) = 5,  TS(T2) = 10

Schedule:
  T2: Write(A)    --> W-timestamp(A) = 10
  T1: Write(A)    --> TS(T1)=5 < W-timestamp(A)=10

Without Thomas Write Rule: T1 is rolled back.
With Thomas Write Rule: T1's write is simply ignored. T1 continues.
   A retains the value written by T2.

This is safe because in any serial order where T1 comes before T2,
T2's write would overwrite T1's anyway.

4.3 Properties of Timestamp Ordering

Property Basic TO TO + Thomas Write Rule
Conflict serializable Yes Not necessarily (view serializable)
Deadlock-free Yes Yes
Starvation possible Yes (repeated rollbacks) Yes
Cascading rollback Possible Possible

5. Validation-Based (Optimistic) Concurrency Control

Optimistic concurrency control assumes that conflicts are rare. Transactions execute without acquiring locks, and validation is performed before commit to check for conflicts.

5.1 Three Phases

  1. Read phase: The transaction reads from the database and writes to a private (local) workspace. All reads and writes are on local copies.
  2. Validation phase: The system checks whether the transaction's operations conflict with other committed or validating transactions. If validation fails, the transaction is rolled back.
  3. Write phase: If validation succeeds, the local copies are written to the actual database. For read-only transactions, this phase is skipped.
flowchart LR A4["Read Phase"] --> B4["Validation Phase"] B4 --> C4["Write Phase"] B4 -- "Fail" --> D4["Rollback & Restart"] style B4 fill:#fef9e7 style D4 fill:#fdecea

5.2 Validation Conditions

Each transaction Ti is assigned three timestamps:

  • Start(Ti): when Ti started its read phase
  • Validation(Ti): when Ti entered its validation phase
  • Finish(Ti): when Ti finished its write phase

The serialization order is determined by the Validation timestamp. For Ti to be validated successfully against an earlier transaction Tj (where Validation(Tj) < Validation(Ti)), one of the following must hold:

  1. Finish(Tj) < Start(Ti): Tj completed before Ti started. No overlap at all.
  2. Finish(Tj) < Validation(Ti) AND the set of data items written by Tj does not intersect with the set of data items read by Ti.

5.3 When to Use Optimistic CC

  • When conflicts are rare (mostly read-only workloads)
  • When the cost of locking overhead is high relative to conflict rates
  • When transactions are short
Disadvantages: If conflicts are frequent, many transactions will fail validation and be rolled back, wasting work. Also, the write phase may require exclusive access, creating a bottleneck.

6. Multi-Version Concurrency Control (MVCC)

MVCC maintains multiple versions of each data item. When a transaction writes, a new version is created rather than overwriting the old value. Readers can access an appropriate older version without being blocked by writers.

6.1 MVCC with Timestamp Ordering

Each version Qk of data item Q has:

  • W-timestamp(Qk): timestamp of the transaction that created this version
  • R-timestamp(Qk): largest timestamp of any transaction that read this version
  • The data value

Read(Q) by Ti: Find the version Qk with the largest W-timestamp that is ≤ TS(Ti). Return that version's value. Update R-timestamp(Qk) if needed. Reads are never rejected.

Write(Q) by Ti: Let Qk be the version with the largest W-timestamp ≤ TS(Ti).

  • If TS(Ti) < R-timestamp(Qk): Rollback Ti (a younger transaction has already read this version).
  • If TS(Ti) = W-timestamp(Qk): Overwrite Qk's value.
  • Otherwise: Create a new version with W-timestamp = TS(Ti).

6.2 MVCC with 2PL (Snapshot Isolation)

A common practical approach (used in PostgreSQL, Oracle, etc.):

  • Each transaction sees a snapshot of the database as of its start time.
  • Writes create new versions; reads always see the snapshot (never blocked by writers).
  • At commit time, check for write-write conflicts: if another committed transaction wrote the same item after this transaction's snapshot, abort (first-committer-wins).
Key Advantage of MVCC: Readers never block writers, and writers never block readers. This provides significantly higher concurrency than lock-based approaches for read-heavy workloads.

GATE Note: Snapshot isolation does NOT guarantee serializability. It prevents dirty reads, non-repeatable reads, and phantom reads, but can allow write skew anomalies.

7. Multiple Granularity Locking

In practice, data items can be locked at different levels of granularity: the entire database, a table, a page, or a single tuple. Fine granularity (tuple-level) allows more concurrency but has higher overhead. Coarse granularity (table-level) has lower overhead but less concurrency.

7.1 Granularity Hierarchy

flowchart TB DB["Database"] --> T1A["Table A"] DB --> T2A["Table B"] T1A --> P1["Page 1"] T1A --> P2["Page 2"] P1 --> R1["Tuple 1"] P1 --> R2["Tuple 2"] P2 --> R3["Tuple 3"] P2 --> R4["Tuple 4"]

7.2 Intention Locks

To efficiently support multiple granularity, we introduce intention locks. Before locking a node at a fine granularity, a transaction must acquire appropriate intention locks on all ancestor nodes.

Lock Type Notation Meaning
Intention-Shared IS The transaction intends to acquire S locks on some descendant nodes
Intention-Exclusive IX The transaction intends to acquire X locks on some descendant nodes
Shared + Intention-Exclusive SIX The node is locked in S mode AND the transaction intends to acquire X locks on some descendants (S + IX)

7.3 Lock Compatibility Matrix (with Intention Locks)

IS IX S SIX X
IS Yes Yes Yes Yes No
IX Yes Yes No No No
S Yes No Yes No No
SIX Yes No No No No
X No No No No No
GATE Favorite: The multiple granularity lock compatibility matrix is a frequently tested topic. Memorize it. A useful pattern: IS is most compatible, X is least compatible. SIX is compatible only with IS.

7.4 Locking Rules

  1. Lock compatibility must be checked before granting a lock.
  2. The root of the tree must be locked first (in any mode).
  3. A node can be locked in S or IS mode only if the parent is locked in IS or IX mode.
  4. A node can be locked in X, IX, or SIX mode only if the parent is locked in IX or SIX mode.
  5. A transaction can lock a node only if it has not previously unlocked any node (2PL is observed).
  6. A transaction can unlock a node only if none of its children are currently locked by the transaction.

Example: Using Intention Locks

Scenario: T1 wants to read all of Table A. T2 wants to update Tuple 3 in Page 2 of Table A. 
T1: lock-IS(Database) -> lock-S(Table A)     [read all of Table A]
T2: lock-IX(Database) -> lock-IX(Table A) -> lock-IX(Page 2) -> lock-X(Tuple 3)

Can T1 and T2 proceed concurrently?
- Database: T1 has IS, T2 wants IX -> Compatible? Yes (IS-IX = Yes)
- Table A: T1 has S, T2 wants IX -> Compatible? No (S-IX = No)
- T2 must WAIT for T1 to release its S lock on Table A.

Alternative: If T1 also locked at tuple level:
T1: lock-IS(Database) -> lock-IS(Table A) -> lock-IS(Page 1) -> lock-S(Tuple 1), lock-S(Tuple 2)
                                           -> lock-IS(Page 2) -> lock-S(Tuple 3), lock-S(Tuple 4)
Now at Table A: T1 has IS, T2 wants IX -> Compatible? Yes!
But at Page 2: T1 has IS, T2 wants IX -> Compatible? Yes!
But at Tuple 3: T1 has S, T2 wants X -> Not compatible. T2 waits for Tuple 3 only.
This shows how finer granularity locking increases concurrency.

7.5 Lock Escalation

Lock escalation is a technique to dynamically adjust lock granularity. When a transaction acquires many fine-grained locks on the same object, the system may escalate these to a single coarser-grained lock to reduce lock management overhead.

  • Example: If a transaction holds S locks on 80% of the tuples in a table, escalate to a single S lock on the entire table.
  • The threshold for escalation is typically configurable.
  • Escalation reduces lock overhead but may decrease concurrency.

8. Deadlock in Databases

Deadlock occurs when two or more transactions are waiting for each other to release locks, creating a cycle of dependencies.

8.1 Deadlock Prevention

Deadlock prevention schemes use transaction timestamps to decide which transaction should be rolled back when a conflict arises. Let Ti request a lock held by Tj.

Wait-Die Scheme (Non-preemptive)

  • If Ti is older than Tj (TS(Ti) < TS(Tj)): Ti waits.
  • If Ti is younger than Tj: Ti dies (is rolled back).
  • Mnemonic: "Old waits for young; young dies."

Wound-Wait Scheme (Preemptive)

  • If Ti is older than Tj: Ti wounds Tj (forces Tj to roll back).
  • If Ti is younger than Tj: Ti waits.
  • Mnemonic: "Old wounds young; young waits for old."
Scheme Ti older than Tj Ti younger than Tj Type
Wait-Die Ti waits Ti dies (rolls back) Non-preemptive
Wound-Wait Tj is wounded (rolls back) Ti waits Preemptive
GATE Tip: In both schemes, when a rolled-back transaction is restarted, it retains its original timestamp. This prevents starvation: as time passes, a repeatedly rolled-back transaction becomes the oldest and will eventually be allowed to complete.

Worked Example: Wait-Die and Wound-Wait

Given: TS(T1) = 5 (older), TS(T2) = 10 (younger). T2 holds a lock on item A. T1 requests a conflicting lock on A.

Wait-Die: T1 is older than T2, so T1 waits for T2.
Wound-Wait: T1 is older than T2, so T1 wounds T2 (T2 is rolled back and T1 gets the lock).

Now suppose T2 requests a conflicting lock on item B held by T1:
Wait-Die: T2 is younger than T1, so T2 dies (T2 is rolled back).
Wound-Wait: T2 is younger than T1, so T2 waits for T1.

8.2 Deadlock Detection: Wait-For Graph

Instead of preventing deadlocks, the system can allow them to occur and then detect and resolve them.

  • Maintain a wait-for graph: nodes are transactions, edge Ti → Tj means Ti is waiting for Tj to release a lock.
  • Deadlock exists if and only if the wait-for graph contains a cycle.
  • The system periodically checks for cycles (or checks whenever a new edge is added).
  • When a cycle is found, a victim transaction is chosen and rolled back to break the cycle.

Victim Selection Criteria

  • How much work the transaction has already done
  • How much work remains
  • Number of cycles the transaction is involved in
  • Minimize total cost of rollback
  • Starvation prevention: do not always choose the same transaction as victim
flowchart LR T1_2["T1"] --> T2_2["T2"] T2_2 --> T3_2["T3"] T3_2 --> T1_2 style T1_2 fill:#fdecea style T2_2 fill:#fef9e7 style T3_2 fill:#d5f5e3

Wait-for graph with cycle: T1 → T2 → T3 → T1 (deadlock detected)

8.3 Timeout-Based Deadlock Detection

A simpler approach: if a transaction has been waiting for a lock for longer than a predefined timeout period, assume deadlock and roll it back.

  • Advantage: Simple to implement, no overhead of maintaining wait-for graph.
  • Disadvantage: May cause unnecessary rollbacks (false positives) if timeout is too short. May detect deadlocks late if timeout is too long.

9. Phantom Problem and Predicate Locking

9.1 The Phantom Problem

The phantom problem occurs when a transaction reads a set of tuples satisfying some condition, and another transaction inserts a new tuple that also satisfies the condition. The first transaction, if it re-reads, sees a "phantom" tuple that was not there before.

Example: 
T1: SELECT COUNT(*) FROM student WHERE dept = 'CS'  --> returns 100
T2: INSERT INTO student VALUES (..., 'CS', ...)     --> inserts new CS student
T2: COMMIT
T1: SELECT COUNT(*) FROM student WHERE dept = 'CS'  --> returns 101  (PHANTOM!)
Standard row-level locking does not prevent this because the new row did not exist when T1 acquired its locks.

9.2 Predicate Locking

Predicate locking locks all tuples (existing and future) that satisfy a given predicate. For example, lock-S(dept = 'CS') locks all current and future CS students.

  • Prevents the phantom problem because an insert that satisfies the predicate would conflict with the predicate lock.
  • Problem: Predicate locking is extremely expensive to implement because checking whether two predicates conflict (overlap) is computationally hard in general.

9.3 Index Locking (Practical Solution)

A practical approximation to predicate locking: lock the index entries or index leaf pages that correspond to the predicate.

  • If T1 reads using an index on dept, it locks the index leaf node for 'CS'.
  • T2's insert must update the same index leaf node, so it conflicts with T1's lock.
  • If no suitable index exists, the system may lock the entire table (very conservative).

9.4 Next-Key Locking

Used in some systems (e.g., InnoDB in MySQL). Locks the gaps between index entries in addition to the entries themselves. This prevents insertion of new records that would fall in the locked range.

12. Summary of Key Points for GATE

  • 2PL ensures conflict serializability. Strict 2PL also prevents cascading rollbacks.
  • Tree protocol is deadlock-free but may require locking unnecessary items.
  • Timestamp ordering is deadlock-free; the Thomas Write Rule allows ignoring obsolete writes (gives view serializability).
  • Validation-based CC works well for low-conflict workloads (3 phases: read, validate, write).
  • MVCC allows readers and writers to not block each other by maintaining multiple versions.
  • Multiple granularity: Know the IS/IX/SIX/S/X compatibility matrix cold.
  • Deadlock: Wait-Die (old waits, young dies), Wound-Wait (old wounds young, young waits). Both prevent starvation by preserving original timestamps.
  • Phantom problem: Solved by predicate locking (theoretical) or index locking/next-key locking (practical).

Concurrency and Synchronization

⚙️ Section 8: Operating System

1. Critical Section Problem

When multiple processes or threads access shared data concurrently, and the outcome depends on the order of execution, a race condition occurs. The critical section is the portion of code where a process accesses shared resources. A correct solution must satisfy three requirements:

RequirementDescription
Mutual ExclusionIf process Pi is executing in its critical section, no other process can execute in its critical section.
ProgressIf no process is in its critical section and some processes wish to enter, only processes not in their remainder section can participate in the decision, and the decision cannot be postponed indefinitely.
Bounded WaitingThere exists a bound on the number of times other processes can enter their critical sections after a process has made a request and before that request is granted.

1.1 Race Condition Example

Example: Two threads incrementing a shared counter count = 5. 
Thread A: register_A = count     // register_A = 5
Thread A: register_A = register_A + 1  // register_A = 6
Thread B: register_B = count     // register_B = 5
Thread B: register_B = register_B + 1  // register_B = 6
Thread A: count = register_A     // count = 6
Thread B: count = register_B     // count = 6
// Expected: 7, Actual: 6  -- Race condition!

2. Peterson's Solution

Peterson's solution is a classic software-based approach for two processes. It uses two shared variables: flag[2] (indicates willingness to enter) and turn (indicates whose turn it is).

// Process i (j = 1 - i)
flag[i] = true;
turn = j;
while (flag[j] && turn == j)
    ; // busy wait
// --- Critical Section ---
flag[i] = false;
// --- Remainder Section ---
PropertySatisfied?Reasoning
Mutual ExclusionYesBoth cannot have turn equal to their own value simultaneously.
ProgressYesIf Pj does not want to enter (flag[j]=false), Pi enters immediately.
Bounded WaitingYesAfter Pi sets turn=j, Pj enters at most once before Pi.
Note: Peterson's solution assumes sequential consistency. On modern processors with out-of-order execution, memory barriers are needed. It works for only two processes. For n processes, the Bakery Algorithm by Lamport is used.

3. Synchronization Tools

3.1 Mutex Locks

A mutex (mutual exclusion) lock is the simplest synchronization tool. A process must acquire the lock before entering the critical section and release it when exiting.

acquire(lock);
// --- Critical Section ---
release(lock);

Simple implementations use busy waiting (spinlock). On multiprocessor systems, spinlocks can be efficient for short critical sections since they avoid the overhead of context switching.

3.2 Semaphores

A semaphore S is an integer variable accessed through two atomic operations:

wait(S):          // also called P() or down()
    while (S <= 0)
        ; // busy wait
    S--;

signal(S):        // also called V() or up()
    S++;
TypeRangePurpose
Binary Semaphore0 or 1Functions like a mutex lock. Controls access to a single resource.
Counting Semaphore0 to NControls access to a resource with multiple instances. Initialized to the number of available resources.
Busy waiting vs. Blocking: To avoid busy waiting, semaphores can be implemented with a waiting queue. When wait() finds S ≤ 0, the process is placed on the queue and blocked. signal() removes a process from the queue and wakes it.

3.3 Monitors

A monitor is a high-level synchronization construct that encapsulates shared data, operations, and synchronization into a single unit. Only one process can be active within a monitor at any time.

Monitors use condition variables with two operations:

  • x.wait() — the process is suspended until another process calls x.signal()
  • x.signal() — resumes exactly one suspended process; if none is suspended, the signal has no effect (unlike semaphore signal which increments)
monitor BoundedBuffer {
    condition full, empty;
    int count = 0;

    void produce(item) {
        if (count == N) full.wait();
        // add item to buffer
        count++;
        empty.signal();
    }

    void consume() {
        if (count == 0) empty.wait();
        // remove item from buffer
        count--;
        full.signal();
    }
}

4. Classical Synchronization Problems

4.1 Producer-Consumer Problem

A producer creates data items and places them in a bounded buffer. A consumer removes items from the buffer. The challenge is to ensure the producer waits when the buffer is full and the consumer waits when it is empty.

flowchart LR P["Producer"] -->|"produce item"| B["wait(empty)\nwait(mutex)"] B -->|"add to buffer"| BUF[("Bounded Buffer\n(size N)")] BUF -->|"remove from buffer"| C["wait(full)\nwait(mutex)"] C -->|"consume item"| CO["Consumer"] B -->|"signal(mutex)\nsignal(full)"| BUF C -->|"signal(mutex)\nsignal(empty)"| BUF 
// Shared: semaphore mutex = 1, empty = N, full = 0;

// Producer                    // Consumer
while (true) {                 while (true) {
    produce(item);                 wait(full);
    wait(empty);                   wait(mutex);
    wait(mutex);                   item = buffer[out];
    buffer[in] = item;             out = (out+1) % N;
    in = (in+1) % N;              signal(mutex);
    signal(mutex);                 signal(empty);
    signal(full);                  consume(item);
}                              }

4.2 Readers-Writers Problem

Multiple readers can read simultaneously, but a writer needs exclusive access. The first readers-writers problem gives priority to readers (writers may starve); the second gives priority to writers (readers may starve).

// First Readers-Writers Solution
// Shared: semaphore rw_mutex = 1, mutex = 1; int read_count = 0;

// Writer                      // Reader
wait(rw_mutex);                wait(mutex);
// --- write ---               read_count++;
signal(rw_mutex);              if (read_count == 1) wait(rw_mutex);
                               signal(mutex);
                               // --- read ---
                               wait(mutex);
                               read_count--;
                               if (read_count == 0) signal(rw_mutex);
                               signal(mutex);

4.3 Dining Philosophers Problem

Five philosophers sit around a table with five chopsticks (one between each pair). A philosopher needs both adjacent chopsticks to eat. The challenge is to avoid deadlock and starvation.

Allow at most 4 to sitLimits concurrent diners to N−1Reduces concurrency

AsymmetricOdd philosophers pick left first, even pick right firstSimple and deadlock-free

6. Summary and GATE Key Points

Key Points for GATE:
  • A critical section solution must satisfy mutual exclusion, progress, and bounded waiting.
  • Peterson's solution works for two processes; requires sequential consistency.
  • Binary semaphore has value 0 or 1; counting semaphore can go up to N.
  • In the producer-consumer problem, the order of wait() calls matters: always wait on the counting semaphore before the mutex, or deadlock can occur.
  • In the readers-writers problem, the first reader locks the resource, and the last reader unlocks it.
  • Monitors automatically enforce mutual exclusion; signal() on a condition variable has no effect if no process is waiting (unlike semaphore).
  • The dining philosophers problem is a classic test case for deadlock prevention strategies.

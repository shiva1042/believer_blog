TCP Congestion Control — Deep Dive

Based on Kurose & Ross — Computer Networking: A Top-Down Approach, Chapter 3

TCP congestion control is one of the most heavily tested topics in GATE Computer Science. This article covers the full theory, all major TCP variants, worked numerical examples, and the modern algorithms you need to know. Every RTT-by-RTT trace is shown so you can practice the exact style of problems that appear in GATE.

1. Why Congestion Control?

Congestion occurs when the aggregate demand for network resources (bandwidth, buffer space) exceeds the available capacity. Without congestion control, the network can enter a state called congestion collapse:

  • Routers' buffers overflow, causing packet drops
  • Senders retransmit dropped packets, adding more traffic
  • Retransmissions cause more congestion, causing more drops
  • Useful throughput (goodput) drops to near zero while offered load remains high
Key insight: Congestion control is NOT the same as flow control. Flow control prevents a fast sender from overwhelming a slow receiver (receiver-side constraint). Congestion control prevents senders from overwhelming the network itself (network-side constraint). TCP enforces both simultaneously: sending_window = min(cwnd, rwnd).

1.1 Costs of Congestion (Kurose & Ross Scenarios)

TCP traditionally uses end-to-end congestion control. The sender adjusts its sending rate based on two signals:

  1. Timeout: indicates severe congestion (packet likely dropped)
  2. Triple duplicate ACK: indicates moderate congestion (packet lost but subsequent packets arrived)

Convention: In GATE problems, cwnd is often expressed in units of MSS (segments) rather than bytes. When cwnd = 4, it means the sender can send 4 MSS-sized segments before waiting for an ACK. We follow this convention throughout this article.

The effective sending window is:

sending_window = min(cwnd, rwnd)

In congestion control problems, we typically assume rwnd is large enough that it does not constrain the sender, so sending_window = cwnd.

4. TCP Congestion Control Algorithm

The TCP congestion control algorithm consists of four interrelated mechanisms:

4.1 Slow Start

Purpose: Quickly probe for available bandwidth when starting a new connection or after a timeout.

Mechanism:

  • Initial cwnd = 1 MSS (RFC 5681 allows up to 10 MSS for IW10, but GATE typically uses 1)
  • For every ACK received, cwnd = cwnd + 1 MSS
  • This effectively doubles cwnd every RTT (exponential growth)

Growth pattern (cwnd in MSS):

After RTT n: cwnd = 2n MSS. This is exponential growth.

Slow Start ends when:

  1. cwnd ≥ ssthresh → transition to Congestion Avoidance
  2. Timeout occurs → set ssthresh = cwnd/2, reset cwnd = 1 MSS, restart Slow Start
  3. Triple duplicate ACK → (depends on TCP variant: Tahoe resets; Reno enters Fast Recovery)
Common GATE Mistake: Slow Start is NOT slow! The name refers to the fact that TCP starts from 1 MSS (a "slow" beginning) rather than blasting data at full rate. The growth itself is exponential and quite fast.

4.2 Congestion Avoidance

Purpose: Carefully probe for additional bandwidth once cwnd is in the vicinity of the network's capacity.

Mechanism:

  • For every ACK received: cwnd = cwnd + MSS * (MSS / cwnd)
  • Equivalently, cwnd = cwnd + 1 MSS per RTT (if all segments in the window are ACKed)
  • This is linear growth (additive increase)

Growth pattern (cwnd in MSS, starting from ssthresh = 8):

Congestion Avoidance ends when:

  1. Timeout → set ssthresh = cwnd/2, reset cwnd = 1 MSS, enter Slow Start
  2. Triple duplicate ACK → behaviour depends on TCP variant (see Section 5)

4.3 Fast Retransmit

Problem: TCP's timeout period can be relatively long. Waiting for a timeout to detect a lost segment is wasteful.

Solution: If the sender receives 3 duplicate ACKs (i.e., 4 ACKs for the same sequence number), it immediately retransmits the missing segment without waiting for the timeout.

Why 3 duplicate ACKs? One or two duplicate ACKs might be caused by out-of-order delivery. Three duplicate ACKs strongly suggest a packet was lost (the receiver got three subsequent segments but is still requesting the missing one).

4.4 Fast Recovery

Rationale: Triple duplicate ACKs indicate that data is still flowing (subsequent segments are arriving). This is less severe than a timeout. We can recover more gracefully.

Mechanism (TCP Reno):

  1. Set ssthresh = cwnd / 2
  2. Set cwnd = ssthresh + 3 MSS (the 3 accounts for the 3 duplicate ACKs — those segments left the network)
  3. For each additional duplicate ACK: cwnd = cwnd + 1 MSS (inflating the window to allow new data)
  4. When the ACK for the retransmitted segment arrives (new ACK): cwnd = ssthresh, enter Congestion Avoidance
GATE Simplification: In most GATE problems, Fast Recovery is simplified. After triple duplicate ACK: ssthresh = cwnd/2 and cwnd = ssthresh (or cwnd/2), then continue with Congestion Avoidance. The inflation/deflation steps are typically not tested.

5. TCP Variants: Tahoe, Reno, NewReno

5.1 TCP Tahoe (Original)

On timeout:

  1. ssthresh = cwnd / 2
  2. cwnd = 1 MSS
  3. Enter Slow Start

On triple duplicate ACK:

  1. ssthresh = cwnd / 2
  2. cwnd = 1 MSS
  3. Enter Slow Start

Tahoe treats timeout and triple dup ACK identically. This is conservative — it always resets to Slow Start.

5.2 TCP Reno

On timeout: Same as Tahoe — reset to cwnd = 1, Slow Start.

On triple duplicate ACK:

  1. ssthresh = cwnd / 2
  2. cwnd = ssthresh (halve, do NOT reset to 1)
  3. Enter Fast Recovery / Congestion Avoidance

Reno is faster than Tahoe because it avoids restarting from cwnd = 1 when only a single packet is lost.

5.3 TCP NewReno

NewReno improves on Reno's handling of multiple packet losses within the same window.

Problem with Reno: When multiple packets in the same window are lost, Reno exits Fast Recovery when the first retransmitted segment is ACKed, but the ACK might be a partial ACK (not acknowledging all outstanding data). This causes Reno to potentially cut the window multiple times for the same congestion event.

NewReno's fix:

  • A "partial ACK" (ACK that advances the window but does not acknowledge all data) keeps the sender in Fast Recovery
  • The sender retransmits the next presumed lost segment immediately
  • Fast Recovery exits only when all data outstanding at the time of the triple dup ACK has been acknowledged

6. AIMD — Additive Increase Multiplicative Decrease

TCP's congestion control follows the AIMD principle:

  • Additive Increase: During Congestion Avoidance, cwnd grows by 1 MSS per RTT
  • Multiplicative Decrease: On detecting loss (triple dup ACK in Reno), cwnd is halved

6.1 Fairness of AIMD

AIMD converges to fair sharing of bandwidth. Consider two TCP connections sharing a bottleneck link of capacity R:

  • If both connections start at different rates, AIMD will bring them to the fairness line (x1 = x2 = R/2)
  • Additive increase moves the operating point along a 45-degree line toward the capacity boundary
  • Multiplicative decrease moves the operating point toward the origin along a line through the origin
  • The trajectory converges to the equal-share point
Key result: AIMD is fair. Two TCP Reno connections sharing a bottleneck link will converge to equal throughput (assuming equal RTTs). However, connections with smaller RTTs get an advantage because they increase cwnd more frequently — this is known as RTT unfairness.

6.2 TCP Throughput Formula

The average throughput of a TCP connection with loss rate L, MSS W, and round-trip time RTT is:

Throughput = (MSS / RTT) * (C / sqrt(L))

where C is approximately 1.22 (derived from the analysis of the AIMD sawtooth pattern, C = sqrt(3/2)).

A simpler approximation often used in GATE:

Throughput ≈ (MSS / RTT) * (1 / sqrt(L))

This is sometimes called the Mathis formula or the TCP throughput equation.

Alternative AIMD throughput formula: If the maximum window size is W segments (before loss), the average window is 3W/4 (since it oscillates between W/2 and W). Average throughput = (3W/4) * MSS / RTT.

7.3 Example 3: Throughput Calculation

Problem: A TCP connection has MSS = 1460 bytes, RTT = 100 ms, and a packet loss rate of L = 10-4 (one loss per 10,000 segments). Calculate the throughput.

Solution:

Throughput = (MSS / RTT) * (1 / sqrt(L))
           = (1460 bytes / 0.1 s) * (1 / sqrt(10^-4))
           = 14,600 bytes/s * (1 / 0.01)
           = 14,600 * 100
           = 1,460,000 bytes/s
           = 1.46 MB/s
           ≈ 11.68 Mbps

With the more precise constant C = 1.22:
Throughput = 1.22 * 1,460,000 = 1,781,200 bytes/s ≈ 14.25 Mbps

7.4 Example 4: AIMD Sawtooth — Average Window

Problem: A TCP Reno connection experiences loss whenever cwnd reaches W = 20 MSS. What is the average throughput if MSS = 1000 bytes and RTT = 50 ms?

Solution: In AIMD, cwnd oscillates between W/2 and W. After loss, cwnd drops to W/2 = 10, then grows linearly back to W = 20.

Average window = (W/2 + W) / 2 = (10 + 20) / 2 = 15 MSS = 3W/4

Average throughput = average_window * MSS / RTT
                   = 15 * 1000 / 0.05
                   = 300,000 bytes/s
                   = 300 KB/s
                   ≈ 2.4 Mbps

7.6 Example 6: Determining Phase from cwnd Pattern

Problem: The cwnd values at the start of successive RTTs are: 1, 2, 4, 8, 16, 17, 18, 19, 20, 21. At RTT 10, the cwnd drops to 1 with ssthresh = 10. Identify the TCP variant.

Solution:

  • RTT 0-4: cwnd doubles → Slow Start
  • RTT 5-9: cwnd increases by 1 → Congestion Avoidance
  • At RTT 10: cwnd drops to 1 → this could be a timeout (both Tahoe and Reno reset to 1)
  • ssthresh = 21/2 ≈ 10 (rounds down) — consistent with cwnd = 21 before the event
  • Need more information: If this was a triple dup ACK and cwnd went to 1, it is TCP Tahoe. If it was a timeout, could be either.

If the problem states it was a triple dup ACK and cwnd → 1, the answer is Tahoe. Reno would set cwnd = ssthresh = 10.

8. TCP CUBIC

TCP CUBIC is the default congestion control algorithm in Linux (since kernel 2.6.19) and the most widely used algorithm on the Internet today.

8.1 Motivation

TCP Reno's AIMD is too conservative for high-bandwidth, high-delay networks (called "long fat networks" or LFNs). With RTT = 100ms and a 10 Gbps link, the window needs to be ~83,333 segments. After a loss, Reno takes ~41,667 RTTs (~70 minutes!) to recover. This is impractical.

8.2 CUBIC's Approach

CUBIC uses a cubic function of time since the last loss event to determine cwnd:

W(t) = C * (t - K)^3 + W_max

where:
  C     = scaling constant (typically 0.4)
  t     = time since last congestion event
  K     = ∛(W_max * β / C)     [time to reach W_max with no loss]
  W_max = cwnd at the time of last loss
  β     = multiplicative decrease factor (0.7 for CUBIC, vs 0.5 for Reno)

8.3 Key Properties

  • Window-based, not RTT-based: CUBIC's growth is a function of real time, not RTT. This makes it fairer across connections with different RTTs.
  • Concave then convex growth: After a loss, cwnd grows quickly toward W_max (concave region), then slowly probes beyond W_max (convex region). This is efficient and cautious.
  • Less aggressive multiplicative decrease: CUBIC uses β = 0.7 (reduces to 70% of cwnd), while Reno uses β = 0.5 (reduces to 50%).
  • Scales well: Recovers much faster than Reno on high-bandwidth links.

9. Explicit Congestion Notification (ECN)

ECN is a network-assisted congestion control mechanism that allows routers to signal congestion before dropping packets.

9.1 How ECN Works

  1. Negotiation: During TCP handshake, both endpoints indicate ECN support using the ECE and CWR flags in the TCP header.
  2. Router marking: When a router's buffer starts filling up, instead of dropping packets, it sets the CE (Congestion Experienced) bits in the IP header's ECN field (bits changed from 01 or 10 to 11).
  3. Receiver feedback: The receiver sees the CE-marked packet and sets the ECE (ECN-Echo) flag in the next ACK.
  4. Sender response: The sender treats the ECE flag like a congestion signal — it reduces cwnd (just as with a loss event) and sets the CWR (Congestion Window Reduced) flag to acknowledge.

9.2 ECN Field in IP Header

ECN bits (2 bits in IP TOS)Meaning
00Not ECN-Capable Transport (Non-ECT)
01ECN-Capable Transport (ECT(1))
10ECN-Capable Transport (ECT(0))
11Congestion Experienced (CE)
Advantage of ECN: No packet loss is needed to signal congestion. This avoids retransmission delays and improves throughput. It is especially beneficial for short flows and latency-sensitive applications.

13. Additional Worked Examples

13.1 Example 7: Finding ssthresh from cwnd Trace

Problem: The cwnd values over successive RTTs are: 1, 2, 4, 8, 9, 10, 11, 12. Determine ssthresh.

Solution: Cwnd doubles from RTT 0-3 (Slow Start: 1→2→4→8), then increases by 1 from RTT 3-7 (Congestion Avoidance: 8→9→10→11→12). The transition happens at cwnd = 8.

ssthresh = 8 MSS

13.2 Example 8: Bandwidth-Delay Product

Problem: A link has bandwidth = 1 Gbps and RTT = 50 ms. MSS = 1500 bytes. What window size (in segments) is needed to fully utilize the link?

Solution:

BDP = bandwidth * RTT
    = 1 * 10^9 bits/s * 50 * 10^-3 s
    = 50 * 10^6 bits
    = 6,250,000 bytes

Window in segments = BDP / MSS
                   = 6,250,000 / 1500
                   ≈ 4167 segments

The sender needs cwnd ≈ 4167 MSS to fully utilize the 1 Gbps link.

13.3 Example 9: Recovery Time Comparison

Problem: A TCP connection has cwnd = 32 and ssthresh = 32 when a triple duplicate ACK occurs. How many RTTs does it take to return to cwnd = 32 for (a) TCP Tahoe and (b) TCP Reno?

(a) TCP Tahoe:

  • New ssthresh = 32/2 = 16, cwnd = 1
  • Slow Start: 1 → 2 → 4 → 8 → 16 (4 RTTs to reach ssthresh)
  • Congestion Avoidance: 16 → 17 → 18 → ... → 32 (16 RTTs)
  • Total: 4 + 16 = 20 RTTs

(b) TCP Reno:

  • New ssthresh = 32/2 = 16, cwnd = 16
  • Congestion Avoidance: 16 → 17 → 18 → ... → 32 (16 RTTs)
  • Total: 16 RTTs

Reno recovers 4 RTTs faster than Tahoe.

13.4 Example 10: Loss Rate from Throughput

Problem: A TCP connection has MSS = 1460 bytes, RTT = 200 ms, and achieves a throughput of 10 Mbps. Estimate the packet loss rate.

Solution: Using the Mathis formula:

Throughput = (MSS / RTT) * (1.22 / sqrt(L))

10 * 10^6 bits/s = (1460 * 8 bits / 0.2 s) * (1.22 / sqrt(L))
10,000,000 = 58,400 * 1.22 / sqrt(L)
10,000,000 = 71,248 / sqrt(L)
sqrt(L) = 71,248 / 10,000,000
sqrt(L) = 0.0071248
L = (0.0071248)^2
L ≈ 5.08 × 10^-5

The loss rate is approximately 5 × 10-5 (about 1 loss per 20,000 segments).

14. Previous Year GATE Insights

Congestion control is tested almost every year in GATE CSE. Common question formats:

  • cwnd trace: Given initial conditions and loss events, trace cwnd over N RTTs
  • Identify variant: Given a cwnd trace, determine if it is Tahoe or Reno
  • ssthresh determination: From cwnd growth pattern, find ssthresh
  • Throughput calculation: Given MSS, RTT, and loss rate
  • Total data sent: Sum of cwnd values over a series of RTTs
  • Conceptual: "Which phase is TCP in if cwnd = X and ssthresh = Y?"
Critical GATE Rule: When cwnd reaches ssthresh, the convention in most GATE answers is that the sender transitions to Congestion Avoidance. So if ssthresh = 16 and cwnd doubles from 8 to 16, at cwnd = 16 the next increment is additive (+1), not multiplicative.

Deadlock

⚙️ Section 8: Operating System

1. Deadlock Conditions

A deadlock is a situation where a set of processes are blocked, each waiting for a resource held by another process in the set. Deadlock can arise if and only if all four of the following conditions hold simultaneously:

ConditionDescription
Mutual ExclusionAt least one resource must be held in a non-shareable mode. Only one process can use the resource at a time.
Hold and WaitA process is holding at least one resource and waiting to acquire additional resources held by other processes.
No PreemptionResources cannot be forcibly taken from a process; they must be released voluntarily.
Circular WaitThere exists a circular chain of processes, each waiting for a resource held by the next process in the chain.
Important: All four conditions are necessary for deadlock. If any one is broken, deadlock cannot occur. They are also jointly sufficient.

2. Resource Allocation Graph

A Resource Allocation Graph (RAG) is a directed graph used to describe deadlocks. It contains:

  • Process nodes (circles): represent processes
  • Resource nodes (rectangles): represent resource types with dots for instances
  • Request edge: Pi → Rj (process requests resource)
  • Assignment edge: Rj → Pi (resource assigned to process)

2.1 Deadlock Example in RAG

Below, P1 holds R1 and requests R2, while P2 holds R2 and requests R1, forming a circular wait:

flowchart LR P1((P1)) -->|requests| R2[R2] R2 -->|held by| P2((P2)) P2 -->|requests| R1[R1] R1 -->|held by| P1
Graph PropertyConclusion
No cycleNo deadlock
Cycle with single-instance resourcesDeadlock exists
Cycle with multi-instance resourcesDeadlock may exist (not guaranteed)

3. Deadlock Prevention

Deadlock prevention ensures that at least one of the four necessary conditions cannot hold.

Condition NegatedStrategyDrawback
Mutual ExclusionMake resources shareable (e.g., read-only files)Not possible for all resources (printers, etc.)
Hold and WaitRequire a process to request all resources before starting, or release all before requesting new onesLow resource utilization; possible starvation
No PreemptionIf a process holding resources requests more and is denied, release all held resourcesPractical only for resources whose state can be saved (CPU registers, memory)
Circular WaitImpose a total ordering on resource types; each process requests in increasing orderMay be restrictive but is the most practical approach

4. Deadlock Avoidance

Deadlock avoidance requires advance knowledge of the maximum resources each process may need. The system only grants requests that keep it in a safe state.

4.1 Safe State

A state is safe if the system can allocate resources to each process (up to its maximum) in some order and still avoid deadlock. A safe sequence <P1, P2, ..., Pn> exists where each Pi's remaining needs can be satisfied by currently available resources plus resources held by all Pj where j < i.

Key insight: Safe state ⇒ no deadlock. Unsafe state ⇒ deadlock possible (not certain). Deadlocked state ⊂ Unsafe states.

4.2 Banker's Algorithm

The Banker's Algorithm (by Dijkstra) works for multiple resource types and multiple instances. It uses the following data structures (n = number of processes, m = number of resource types):

StructureSizeDescription
AvailablemNumber of available instances of each resource type
Maxn × mMaximum demand of each process
Allocationn × mNumber of resources currently allocated to each process
Needn × mRemaining need: Need[i] = Max[i] − Allocation[i]

4.3 Banker's Algorithm Example

Given: 3 resource types (A, B, C) with total instances (10, 5, 7) and 5 processes.
ProcessAllocation (A B C)Max (A B C)Need (A B C)
P00 1 07 5 37 4 3
P12 0 03 2 21 2 2
P23 0 29 0 26 0 0
P32 1 12 2 20 1 1
P40 0 24 3 34 3 1

Available = (3, 3, 2)

Safety Check (find a safe sequence):

  1. P1: Need (1,2,2) ≤ Available (3,3,2). Execute P1. Available becomes (3,3,2) + (2,0,0) = (5,3,2).
  2. P3: Need (0,1,1) ≤ Available (5,3,2). Execute P3. Available becomes (5,3,2) + (2,1,1) = (7,4,3).
  3. P4: Need (4,3,1) ≤ Available (7,4,3). Execute P4. Available becomes (7,4,3) + (0,0,2) = (7,4,5).
  4. P2: Need (6,0,0) ≤ Available (7,4,5). Execute P2. Available becomes (7,4,5) + (3,0,2) = (10,4,7).
  5. P0: Need (7,4,3) ≤ Available (10,4,7). Execute P0. Available becomes (10,5,7).

Safe sequence: <P1, P3, P4, P2, P0>. System is in a safe state.

4.4 Resource Request Algorithm

When process Pi makes a request Request_i:

  1. If Requesti ≤ Needi, proceed. Otherwise, error (exceeded claim).
  2. If Requesti ≤ Available, proceed. Otherwise, Pi must wait.
  3. Pretend to allocate: Available -= Requesti; Allocationi += Requesti; Needi -= Requesti.
  4. Run safety algorithm. If safe, grant. If unsafe, rollback and make Pi wait.

5. Deadlock Detection

5.1 Single Instance of Each Resource Type

Use a wait-for graph: remove resource nodes from the RAG and collapse edges. If a cycle exists, deadlock is present. Cycle detection takes O(n2) time.

5.2 Multiple Instances

Use an algorithm similar to Banker's but without the Max matrix. Instead, use current requests. If a process's request can be satisfied, assume it will release all resources. Processes that cannot finish are deadlocked.

AspectSingle InstanceMultiple Instances
Detection MethodWait-for graph cycle detectionDetection algorithm (like Banker's)
ComplexityO(n2)O(m × n2)
FrequencyRun periodically or on resource requestSame

6. Deadlock Recovery

Once deadlock is detected, the system must recover:

6.1 Process Termination

StrategyDescription
Abort all deadlocked processesGuaranteed to break deadlock but expensive (loses all work)
Abort one process at a timeLess costly; re-run detection after each abort. Selection criteria: priority, CPU time used, resources held, remaining time, number of processes to terminate.

6.2 Resource Preemption

  • Selecting a victim: Choose the process whose preemption causes minimum cost.
  • Rollback: Roll back the process to a safe state (requires checkpointing).
  • Starvation: Ensure a process is not always chosen as victim by including rollback count in the cost factor.

8. Summary and GATE Key Points

Key Points for GATE:
  • All four conditions (mutual exclusion, hold and wait, no preemption, circular wait) must hold simultaneously for deadlock.
  • In a RAG with single-instance resources, a cycle is both necessary and sufficient for deadlock.
  • In a RAG with multi-instance resources, a cycle is necessary but not sufficient.
  • The Banker's Algorithm has time complexity O(m × n2) per request.
  • Circular wait prevention by resource ordering is the most practical prevention technique.
  • Safe state implies no deadlock. Unsafe state does not guarantee deadlock.
  • With n processes and m identical resources, deadlock-free minimum resources = n × (m−1) + 1, where m is max need per process.

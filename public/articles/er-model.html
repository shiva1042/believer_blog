ER Model

🗃️ Section 9: Databases

1. Entity

An entity is a real-world object or concept that is distinguishable from other objects. An entity set is a collection of entities of the same type (e.g., all students in a university).

Entity TypeDescriptionExample
Strong EntityHas a primary key that uniquely identifies each instanceStudent (with StudentID)
Weak EntityDoes not have a sufficient primary key on its own; depends on an identifying (owner) entityDependent (depends on Employee)

2. Attributes

An attribute is a property that describes an entity. Attributes are classified as follows:

Attribute TypeDescriptionExample
Simple (Atomic)Cannot be divided furtherAge, SSN
CompositeCan be divided into sub-attributesName (FirstName, MiddleName, LastName)
Single-ValuedHas one value for each entityDateOfBirth
MultivaluedCan have multiple values for a single entityPhoneNumbers, EmailAddresses
DerivedValue can be computed from other attributesAge (derived from DateOfBirth)
Key AttributeUniquely identifies an entity within its entity setStudentID, EmployeeID
StoredActually stored in the database (source of derived attributes)DateOfBirth
ER Diagram Notation:
  • Simple attribute: plain oval
  • Composite attribute: oval connected to sub-ovals
  • Multivalued attribute: double oval
  • Derived attribute: dashed oval
  • Key attribute: underlined name inside oval

3. Relationships

A relationship is an association among two or more entities. A relationship set is a set of relationships of the same type.

3.1 Degree of a Relationship

DegreeDescriptionExample
Unary (Recursive)Single entity set participates in multiple rolesEmployee manages Employee
BinaryTwo entity sets are involvedStudent enrolls in Course
TernaryThree entity sets are involvedSupplier supplies Part to Project

3.2 Cardinality Constraints

Cardinality defines the number of entities to which another entity can be associated via a relationship:

CardinalityDescriptionExample
1:1 (One-to-One)Each entity in A is associated with at most one entity in B, and vice versaPerson has Passport
1:N (One-to-Many)Each entity in A can be associated with many in B, but each in B is associated with at most one in ADepartment has Employees
M:N (Many-to-Many)Each entity in A can be associated with many in B, and vice versaStudent enrolls in Course

3.3 Participation Constraints

TypeDescriptionNotation
Total ParticipationEvery entity must participate in the relationshipDouble line in ER diagram
Partial ParticipationNot every entity needs to participateSingle line in ER diagram
Example: Every employee MUST work in a department (total participation of Employee in "works_in"). Not every department must have a manager (partial participation of Department in "managed_by").

5. Weak Entities and Identifying Relationships

A weak entity cannot be uniquely identified by its own attributes alone. It depends on an owner (strong) entity through an identifying relationship.

ConceptDescription
Partial Key (Discriminator)An attribute that distinguishes weak entities associated with the same owner. Represented with a dashed underline.
Identifying RelationshipThe relationship connecting the weak entity to its owner. Represented with a double diamond.
Total ParticipationA weak entity always has total participation in its identifying relationship.
Example: Dependent is a weak entity depending on Employee.
  • Employee has EmployeeID (primary key)
  • Dependent has DependentName (partial key / discriminator)
  • Full key of Dependent = (EmployeeID, DependentName)

6. ER to Relational Model Mapping

Converting an ER diagram to relational tables follows systematic rules:

ER ConstructRelational Mapping
Strong EntityCreate a table with all simple attributes; primary key becomes table PK
Weak EntityCreate a table with partial key + owner's PK as composite primary key; add foreign key to owner
1:1 RelationshipAdd foreign key to either side (prefer the side with total participation); or merge into one table
1:N RelationshipAdd foreign key to the "many" side (N-side) referencing the "one" side
M:N RelationshipCreate a new junction table with PKs of both entities as composite PK; include relationship attributes
Multivalued AttributeCreate a separate table with the entity's PK and the attribute as composite PK
Composite AttributeInclude only the leaf (simple) sub-attributes in the table
Derived AttributeTypically not stored; computed at query time (or stored for performance)
Mapping Example: For the university ER diagram above: 
Student(StudentID, Name, DateOfBirth, Email)
Course(CourseID, Title, Credits, DeptID, InstructorID)
  -- DeptID is FK to Department (1:N "offers")
  -- InstructorID is FK to Instructor (1:N "teaches")
Department(DeptID, DeptName, Location)
Instructor(InstructorID, Name, Rank, DeptID)
  -- DeptID is FK to Department (1:N "employs")
Enrolls(StudentID, CourseID, Grade, EnrollDate)
  -- Composite PK (StudentID, CourseID)
  -- StudentID FK to Student, CourseID FK to Course

6.1 Minimum Number of Tables

GATE Tip: To find the minimum number of tables:
  • Each strong entity = 1 table
  • Each weak entity = 1 table
  • Each M:N relationship = 1 table
  • 1:1 and 1:N relationships can be merged (FK added, no new table needed)
  • Each multivalued attribute = 1 table

7. Generalization, Specialization, and Aggregation

ConceptDescriptionDirection
GeneralizationCombining entity sets that share common attributes into a higher-level entityBottom-up
SpecializationDividing an entity set into subgroups based on distinguishing characteristicsTop-down
AggregationTreating a relationship as a higher-level entity to participate in another relationshipAbstraction
Specialization Example: Person entity specialized into Student and Employee. Student has GPA attribute; Employee has Salary. Both share Name, DOB from Person.

8. Summary and GATE Key Points

Key Points for GATE:
  • A weak entity always has total participation in its identifying relationship and the cardinality is always 1:N (owner:weak).
  • For M:N relationships, a separate table is always required.
  • For 1:N relationships, add the FK on the N-side. No extra table needed.
  • For 1:1 relationships, add the FK on the side with total participation (if any).
  • Minimum tables = strong entities + weak entities + M:N relationships + multivalued attributes.
  • A ternary relationship requires a separate table with the PKs of all three entities.
  • Aggregation is used when a relationship itself needs to participate in another relationship.

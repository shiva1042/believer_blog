1. File Organization

File organization determines how records are physically arranged on disk, affecting search, insertion, and deletion performance.

OrganizationDescriptionSearchInsertDelete
Heap (Unordered)Records stored in insertion orderO(n) linear scanO(1) append to endO(n) search + mark deleted

Hash File Organization: Static hashing uses a fixed number of buckets (overflow chains when buckets are full). Dynamic hashing (extendible hashing, linear hashing) grows the hash table as data grows.

2. Indexing

An index is a data structure that enables fast retrieval of records based on search-key values. It avoids full table scans.

2.1 Types of Indexes

Primary IndexIndex on the ordering field of a sequentially ordered file; one index entry per disk blockSparse; file must be sorted on the index key

Secondary IndexIndex on a non-ordering field; can be on any attributeDense (one entry per record or per unique value)

2.2 Multi-Level Indexing

When a single-level index is too large to fit in memory, build an index on the index (outer index). This creates a hierarchy of indexes. The number of levels = ceil(logf(n)), where f = fan-out (blocking factor of index) and n = number of data blocks.

3. B-Trees and B+ Trees

3.1 B-Tree

A B-tree of order m is a balanced search tree where:

  • Each node has at most m children and m−1 keys
  • Each internal node (except root) has at least ⌈m/2⌉ children
  • The root has at least 2 children (if it is not a leaf)
  • All leaves are at the same level
  • Data pointers exist at both internal nodes and leaves

3.2 B+ Tree

A B+ tree is a variant of B-tree optimized for disk-based databases:

  • All data pointers are in leaf nodes only
  • Internal nodes contain only keys and child pointers (routing)
  • Leaf nodes are linked in a doubly-linked list for efficient range queries
  • Internal nodes can hold more keys (higher fan-out) since they do not store data

4. Transactions

A transaction is a logical unit of work that comprises one or more database operations. It must satisfy the ACID properties.

4.1 ACID Properties

PropertyDescriptionMechanism
AtomicityAll operations execute or none do (all-or-nothing)Undo logging, redo logging
ConsistencyTransaction takes the database from one consistent state to anotherIntegrity constraints, application logic
IsolationConcurrent transactions do not interfere with each otherConcurrency control (locks, timestamps)
DurabilityOnce committed, changes persist even after system failureWrite-ahead logging, checkpointing

5. Concurrency Control

5.1 Serializability

A schedule is serializable if its result is equivalent to some serial schedule of the same transactions.

TypeDefinitionTest
Conflict SerializableCan be transformed into a serial schedule by swapping non-conflicting operationsPrecedence (serialization) graph must be acyclic
View SerializableSame initial reads, final writes, and read-from relationships as some serial scheduleNP-complete to test
Conflict: Two operations conflict if they (1) belong to different transactions, (2) access the same data item, and (3) at least one is a write. The order of conflicting operations determines the precedence graph edges.
Precedence Graph Example:
Schedule: R1(A), W2(A), R2(B), W1(B)
Conflicts: R1(A) before W2(A) ⇒ T1 → T2. R2(B) before W1(B) ⇒ T2 → T1.
Graph has a cycle (T1 → T2 → T1), so the schedule is NOT conflict serializable.

5.2 Lock-Based Concurrency Control

Lock TypeNotationCompatibility
Shared (S)Read lockCompatible with other S locks; not with X locks
Exclusive (X)Write lockNot compatible with any other lock

5.3 Two-Phase Locking (2PL)

flowchart LR subgraph "Growing Phase" G1["Acquire locks"] --> G2["More locks..."] end subgraph "Lock Point" LP["Maximum locks held"] end subgraph "Shrinking Phase" S1["Release locks"] --> S2["More releases..."] end G2 --> LP --> S1

2PL Protocol: A transaction has two phases:

  1. Growing phase: Transaction may acquire locks but may not release any.
  2. Shrinking phase: Transaction may release locks but may not acquire new ones.
2PL VariantDescriptionDeadlock?Cascading Abort?
Basic 2PLStandard growing and shrinking phasesPossiblePossible
Strict 2PLHold all X (exclusive) locks until commit/abortPossibleNo
Rigorous 2PLHold all locks (S and X) until commit/abortPossibleNo
Conservative 2PLAcquire all locks before startingNoPossible
Key Theorem: Any schedule produced by the 2PL protocol is conflict serializable. However, 2PL does not prevent deadlocks (except Conservative 2PL).

5.4 Timestamp-Based Concurrency Control

Each transaction gets a unique timestamp when it starts. For each data item X, the system maintains:

  • W-timestamp(X): Timestamp of the last transaction that wrote X
  • R-timestamp(X): Timestamp of the last transaction that read X
OperationCondition for RejectionAction on Rejection
Ti reads XTS(Ti) < W-timestamp(X)Rollback Ti
Ti writes XTS(Ti) < R-timestamp(X) or TS(Ti) < W-timestamp(X)Rollback Ti (or apply Thomas Write Rule for the W case)

Thomas Write Rule: If TS(Ti) < W-timestamp(X), the write is outdated and can be safely ignored (instead of rolling back). This allows more schedules to succeed.

6. Summary and GATE Key Points

Key Points for GATE:
  • Primary index is always sparse (one entry per block). Secondary index on a key attribute is dense (one entry per record).
  • B+ tree search cost = O(logp n) where p is order and n is number of keys.
  • In B+ trees, all data pointers are at leaf level; leaves are linked for range queries.
  • Conflict serializability is tested by checking for cycles in the precedence graph (polynomial time). View serializability testing is NP-complete.
  • Every conflict-serializable schedule is view-serializable, but not vice versa.
  • 2PL guarantees conflict serializability but not freedom from deadlocks.
  • Strict 2PL prevents cascading rollbacks by holding X locks until commit.
  • Timestamp ordering does not cause deadlocks but may cause starvation (repeated rollbacks).
  • ACID: Atomicity (logging), Consistency (constraints), Isolation (concurrency control), Durability (write-ahead log).
  • For a B+ tree of order p with n records: minimum height = ceil(logp(n)), maximum height = ceil(logceil(p/2)(n)).

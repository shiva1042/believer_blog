I/O Interface

🖥️ Section 3: Computer Organization

1. Introduction

I/O (Input/Output) interfaces connect the CPU and memory system with external devices such as keyboards, disks, displays, and network adapters. The primary challenge is managing the speed mismatch between the fast CPU and slow I/O devices.

1.1 I/O Module Functions

  • Control and timing: Coordinate data flow between CPU and devices
  • CPU communication: Accept commands, report status, transfer data
  • Device communication: Issue commands to device, exchange data
  • Data buffering: Temporarily store data to handle speed differences
  • Error detection: Report device errors to CPU

2. I/O Techniques

2.1 Programmed I/O (Polling)

The CPU continuously checks (polls) the status of the I/O device in a busy-wait loop. The CPU is fully occupied during the transfer.

flowchart TD A["CPU issues I/O command"] --> B["CPU reads device status"] B --> C{"Device ready?"} C -- No --> B C -- Yes --> D["CPU transfers data"] D --> E{"More data?"} E -- Yes --> A E -- No --> F["I/O complete"]
AdvantageDisadvantage
Simple to implementCPU wastes time polling (busy waiting)
No special hardware neededVery inefficient for slow devices
Deterministic timingCPU cannot do other work during transfer

2.2 Interrupt-Driven I/O

The CPU issues an I/O command and continues executing other instructions. When the device is ready, it sends an interrupt signal to the CPU, which then handles the data transfer.

flowchart TD A["CPU issues I/O command"] --> B["CPU continues other work"] B --> C{"Interrupt received?"} C -- No --> B C -- Yes --> D["Save CPU state (push to stack)"] D --> E["Execute ISR (transfer data)"] E --> F["Restore CPU state"] F --> G{"More data?"} G -- Yes --> A G -- No --> H["I/O complete"]
AdvantageDisadvantage
CPU is free between transfersOverhead per interrupt (context save/restore)
Better CPU utilization than pollingCPU still involved in every data word transfer
Good for low-bandwidth devicesNot efficient for bulk data transfers

2.3 Direct Memory Access (DMA)

A DMA controller transfers data directly between I/O devices and memory without CPU intervention. The CPU only initiates the transfer and is interrupted when complete.

flowchart TD A["CPU sends to DMA: device address, memory address, byte count, direction"] --> B["DMA takes control of bus"] B --> C["DMA transfers data between device and memory"] C --> D["DMA decrements byte count"] D --> E{"Count = 0?"} E -- No --> C E -- Yes --> F["DMA sends interrupt to CPU"] F --> G["CPU resumes"]

3.2 Interrupt Priority

When multiple devices request interrupts simultaneously, a priority scheme determines which is serviced first.

MethodDescription
Software PollingCPU polls each device in priority order to find interrupt source
Daisy ChainHardware chain: devices connected in series; nearest gets highest priority
Vectored InterruptDevice provides ISR address directly; fast but needs complex hardware
Priority EncoderParallel priority resolution using hardware encoder

3.3 Interrupt Handling Sequence

flowchart TD A["Device asserts interrupt line"] --> B["CPU finishes current instruction"] B --> C["CPU checks for interrupt"] C --> D["CPU acknowledges interrupt"] D --> E["Save PC and status registers to stack"] E --> F["Determine interrupt source and vector"] F --> G["Load ISR address into PC"] G --> H["Execute Interrupt Service Routine"] H --> I["Restore PC and status from stack"] I --> J["Resume interrupted program"]

3.4 Nested Interrupts

A higher-priority interrupt can preempt a lower-priority ISR. This requires saving the state of the interrupted ISR. The interrupt mask register controls which interrupt levels can preempt.

Non-Maskable Interrupt (NMI): Cannot be disabled by software. Used for critical events like power failure, memory parity errors, or hardware watchdog timers.

5. I/O Bandwidth Calculations

5.1 Polling Overhead

Example: A device transfers data at 1 MB/s, word size = 4 bytes, polling takes 400 clock cycles, CPU clock = 1 GHz.

Transfers per second = 1 MB / 4 B = 250,000

Polling cycles per second = 250,000 × 400 = 100,000,000 = 108

CPU utilization for I/O = 108 / 109 = 10%

5.2 Interrupt Overhead

Same device: interrupt handling takes 500 cycles.

Interrupt cycles per second = 250,000 × 500 = 1.25 × 108

CPU utilization = 12.5%

Interrupts have higher per-transfer overhead than polling but are only invoked when the device is ready, freeing the CPU between transfers.

5.3 DMA Overhead

Same device: DMA transfers 4 KB blocks, DMA setup = 1000 cycles, interrupt at end = 500 cycles.

DMA transfers per second = 1 MB / 4 KB = 256

Total cycles = 256 × (1000 + 500) = 384,000

CPU utilization = 384,000 / 109 = 0.038%

7. GATE Exam Tips

  • Know how to calculate CPU utilization for each I/O technique.
  • DMA is the most efficient for bulk transfers; its overhead is per-block, not per-word.
  • In cycle-stealing DMA, the CPU is slowed but not blocked; in burst mode, it is blocked.
  • Vectored interrupts are faster than polled interrupts because the ISR address is provided directly.
  • Memory-mapped I/O uses regular LOAD/STORE; isolated I/O needs special IN/OUT instructions.
  • Interrupt latency = time from interrupt request to start of ISR execution.
  • For GATE numerical problems, track: device transfer rate, CPU clock rate, cycles per operation, and compute utilization as a fraction.

8. Practice Problems

  1. A CPU running at 2 GHz handles an I/O device at 500 KB/s. Each interrupt takes 200 cycles and transfers 2 bytes. What fraction of CPU time is spent on I/O?
  2. With DMA, the same device transfers 8 KB blocks. DMA setup costs 500 cycles and completion interrupt costs 300 cycles. What is the CPU utilization now?
  3. Compare memory-mapped and isolated I/O in terms of address space and instruction set requirements.

2. Role of the Lexical Analyzer

The lexical analyzer (scanner) reads the source program character by character and groups characters into meaningful sequences called lexemes, producing a stream of tokens.

Primary Functions

  • Read input characters and produce tokens for the parser.
  • Strip out whitespace, comments, and other non-essential characters.
  • Correlate error messages with source line numbers.
  • Interact with the symbol table to store identifiers.
  • Handle macro expansion (in some implementations).

Why Separate Lexical Analysis?

  • Simplicity: Separating scanning from parsing simplifies both phases.
  • Efficiency: Specialized buffering techniques for character reading.
  • Portability: Input-device-specific peculiarities isolated to the scanner.

Tokens are specified using regular expressions, which are then converted to finite automata for efficient recognition.

5. Finite Automata for Pattern Recognition

The lexical analyzer uses finite automata to recognize tokens efficiently.

5.1 From RE to NFA (Thompson's Construction)

Each RE operator maps to a small NFA fragment. These fragments are composed to build an NFA for the entire RE.

5.2 From NFA to DFA (Subset Construction)

The NFA is converted to a DFA using subset construction for efficient (O(1) per character) scanning.

5.3 Combining Multiple Token Patterns

  1. Build an NFA for each token pattern.
  2. Combine all NFAs with a new start state and epsilon transitions.
  3. Convert the combined NFA to a single DFA.
  4. Each accepting state in the DFA is labeled with the corresponding token type.
  5. When multiple patterns match, use longest match and priority rules.
Longest Match Rule: The scanner always returns the longest possible lexeme. For example, "iffy" is scanned as an identifier, not as keyword "if" followed by "fy".
Priority Rule: When two patterns match the same lexeme, the pattern listed first (higher priority) wins. Keywords are typically listed before identifiers.

6. Input Buffering

The lexical analyzer uses buffering techniques to efficiently read source characters.

6.1 Buffer Pairs

Two buffers of size N (e.g., 4096 bytes) are used alternately. Two pointers are maintained:

  • lexemeBegin: Points to the start of the current lexeme.
  • forward: Scans ahead until a match is found.

6.2 Sentinels

A special sentinel character (e.g., EOF) is placed at the end of each buffer to eliminate the need for a separate end-of-buffer check at each character read, reducing the number of comparisons.

7. LEX Tool Concept

LEX (or Flex) is a lexical analyzer generator that automatically creates a scanner from token specifications.

LEX Program Structure

%{
  /* C declarations */
%}

/* Definitions (RE abbreviations) */
digit   [0-9]
letter  [a-zA-Z]

%%

/* Rules: pattern -> action */
if          { return IF; }
{letter}({letter}|{digit})*  { return ID; }
{digit}+    { return NUM; }
[ \t\n]+    { /* skip whitespace */ }

%%

/* User code (auxiliary functions) */

LEX Processing Pipeline

LEX specification (.l file)
    --> LEX compiler
        --> lex.yy.c (C source)
            --> C compiler
                --> Lexical analyzer (a.out)

8. Symbol Table Interaction

The lexical analyzer interacts with the symbol table in several ways:

  • When an identifier is first encountered, it is inserted into the symbol table.
  • On subsequent occurrences, the symbol table is looked up to return the existing entry.
  • Keywords can be pre-loaded into the symbol table to distinguish them from identifiers.
  • The token returned to the parser includes a pointer to the symbol table entry.

9. Error Handling in Lexical Analysis

The lexical analyzer handles a limited set of errors:

  • Illegal characters: Characters not in the language alphabet.
  • Unterminated strings: String literals without closing quotes.
  • Unterminated comments: Block comments without closing delimiters.
  • Numbers with invalid format: Such as "1.2.3".

Panic Mode Recovery

When an error is found, the scanner deletes characters from the remaining input until a valid token is found. This is the simplest error recovery strategy.

10. Key GATE Takeaways

  • The lexical analyzer converts character streams to token streams — the first phase of compilation.
  • Tokens are specified by regular expressions and recognized by finite automata (DFA).
  • Longest match rule and priority rule resolve ambiguities in token recognition.
  • LEX automates scanner construction from RE specifications.
  • The symbol table is shared across compiler phases; the scanner creates initial entries for identifiers.
  • Lexical analysis handles regular language features; anything requiring context-free power (like nested structures) is left to the parser.
  • Buffer pairs with sentinels optimize character-level I/O performance.

Microprogrammed Control Unit Design

Computer Organization

The control unit is the brain of the CPU, generating the sequence of control signals needed to execute each instruction. There are two fundamental approaches to control unit design: hardwired control and microprogrammed control. This article provides a comprehensive treatment of microprogrammed control as covered in Mano Chapter 7 and Stallings Chapter 20.

1. Control Unit Fundamentals

Every machine instruction is executed through a sequence of micro-operations — elementary operations performed on data stored in registers. The control unit must generate the correct sequence of control signals for each instruction.

1.1 Control Signals

Control signals determine:

  • Which registers are loaded, read, or cleared
  • Which ALU operation to perform
  • Whether memory is read or written
  • Which bus paths are active
  • Sequencing decisions (branch, subroutine call)

The control unit takes as input:

  • The opcode from the instruction register (IR)
  • Status flags (zero, carry, sign, overflow)
  • Clock signals for timing
  • External signals (interrupts, bus requests)

2. Hardwired vs Microprogrammed Control

The two approaches differ fundamentally in how the control logic is implemented.

2.1 Hardwired Control

In hardwired control, the control logic is implemented as a combinational circuit (with sequential elements for sequencing). The inputs are the opcode, timing signals, and flags; the outputs are control signals.

graph TD subgraph "Hardwired Control Unit" IR2["Opcode bits"] --> DECODE["Instruction Decoder"] DECODE --> CL["Combinational
Logic Network"] SC["Step Counter
(Timing)"] --> CL FLAGS2["Condition Flags"] --> CL CL --> CS2["Control Signals
C0, C1, ..., Cn"] CLK2["Clock"] --> SC end

Characteristics:

  • Implemented using gates, decoders, encoders, multiplexers
  • Very fast — signals propagate through combinational logic at gate speed
  • Difficult to design, test, and modify for complex instruction sets
  • No regular structure — changes require rewiring
  • Used in RISC processors (ARM, MIPS) where the instruction set is simple and speed is critical

2.2 Microprogrammed Control

In microprogrammed control, the control signals are stored as microinstructions in a special memory called the control memory (CM). Each machine instruction is executed by a sequence of microinstructions called a microprogram (also called firmware).

graph TD subgraph "Microprogrammed Control Unit" IR3["Opcode"] --> MAP["Mapping Logic"] MAP --> CAR["Control Address
Register (CAR)"] CAR --> CM["Control Memory
(ROM/PROM)"] CM --> MIR["Micro-Instruction
Register (MIR)"] MIR --> CS3["Control Signals
to datapath"] MIR --> SEQ["Sequencer
(next address logic)"] FLAGS3["Flags"] --> SEQ SEQ --> CAR end

Characteristics:

  • Control signals are data stored in ROM — the design is systematic and regular
  • Easy to modify: change the content of control memory (like changing software)
  • Slower than hardwired: requires a memory read for each micro-operation
  • Used in CISC processors (x86) where the instruction set is complex
  • Concept introduced by Maurice Wilkes in 1951

2.3 Detailed Comparison Table

Parameter Hardwired Control Microprogrammed Control

Speed Faster (gate delays only) Slower (control memory access time)
Flexibility Inflexible — redesign needed for changes Flexible — reprogram the control memory

GATE Key Point: Modern x86 processors use a hybrid approach — simple instructions are executed via hardwired control for speed, while complex instructions use microcode. Intel calls this approach microcode assist.

3. Control Memory and Microinstructions

3.1 Terminology

Term Definition
Control Memory (CM) A ROM (or writable control store) that holds all microinstructions. Addressed by the Control Address Register.
Microinstruction A single word in control memory. Contains control signal bits and next-address information.
Microprogram A sequence of microinstructions that implements one machine instruction.
Micro-operation An elementary operation triggered by a single control signal (e.g., R1 ← R2, PC ← PC + 1).
Control Address Register (CAR) Holds the address of the next microinstruction to be fetched from CM.
Microinstruction Register (MIR) Holds the current microinstruction being executed. Its bits directly drive control signals.
Control Word The portion of a microinstruction that specifies control signals (synonymous with microinstruction in simple designs).
Firmware The microprograms stored in control memory. Between hardware and software in terms of flexibility.

3.2 Control Memory Organization

If the control memory has 2k words, the CAR has k bits. Each word (microinstruction) has width determined by the encoding scheme used.

A typical microinstruction word contains:

  • Control field: specifies which control signals to activate
  • Condition field: specifies which flag/condition to test
  • Branch address field: address of the next microinstruction if branching
Microinstruction format:
+---------------------------+-------------+-------------------+
| Control Field             | Condition   | Branch Address    |
| (control signals)         | Select      | (next addr)       |
+---------------------------+-------------+-------------------+
       w bits                  c bits          k bits

4. Microinstruction Formats: Horizontal vs Vertical

The key design decision in microprogrammed control is how to encode the control signals within the microinstruction. The two extremes are horizontal and vertical encoding.

4.1 Horizontal Microinstruction

Each control signal has its own dedicated bit in the microinstruction. If there are n control signals, the control field is n bits wide. A bit value of 1 activates the corresponding signal; 0 deactivates it. 

Horizontal format (example with 40 control signals):
+--------------------------------------------+------+----------+
| C0 C1 C2 C3 ... C39                       | Cond | Br Addr  |
| (40 bits, one per control signal)          |      |          |
+--------------------------------------------+------+----------+
                 40 bits                       3 bits  10 bits = 53 bits total

Advantages:

  • Maximum parallelism — multiple independent micro-operations can be specified simultaneously
  • Simple decoding — no decoder needed between MIR and control lines
  • Fast execution — bits directly drive control signals

Disadvantages:

  • Very wide microinstruction (many bits)
  • Many bits are wasted (most are 0 in any given microinstruction)
  • Large control memory required

4.2 Vertical Microinstruction

Control signals are encoded using binary codes. If there are n control signals, only ceil(log2 n) bits are needed. A decoder at the output converts the code to one-hot control signals. 

Vertical format (example with 40 control signals):
+----------------+------+----------+
| Op Code        | Cond | Br Addr  |
| (encoded)      |      |          |
+----------------+------+----------+
    6 bits         3 bits  10 bits = 19 bits total

Advantages:

  • Narrow microinstruction — small control memory
  • Compact representation
  • Resembles a regular machine instruction format

Disadvantages:

  • Only one micro-operation per microinstruction (no parallelism)
  • Requires a decoder, adding to the delay
  • More microinstructions needed to perform the same task, increasing execution time

4.3 Comparison Table

Parameter Horizontal Vertical
Microinstruction width Wide (n bits for n signals) Narrow (log2 n bits)
Parallelism High (many signals at once) Low (one operation at a time)
Control memory size Wider but fewer words Narrower but more words
Decoding No decoding needed Decoder required
Speed Faster (per microinstruction) Slower (decoder delay + more steps)
Resemblance Looks like a control word Looks like a machine instruction

4.4 Practical Approach: Field Encoding (Intermediate)

Most practical designs use field encoding, which is between horizontal and vertical. The control signals are divided into groups of mutually exclusive signals. Within each group, encoded (vertical) representation is used. Different groups can operate in parallel (horizontal). 

Field-encoded format:
+--------+--------+--------+--------+------+----------+
| Field1 | Field2 | Field3 | Field4 | Cond | Br Addr  |
| (ALU)  | (Bus)  | (Reg)  | (Mem)  |      |          |
+--------+--------+--------+--------+------+----------+
 3 bits    2 bits   3 bits   2 bits   3 bits  10 bits = 23 bits

Example: The ALU field (3 bits) can encode 8 operations (ADD, SUB, AND, OR, XOR, NOT, SHL, SHR). Since only one ALU operation happens at a time, encoding is efficient. But the ALU field and Memory field can be active simultaneously (different groups in parallel).

GATE Tip: Field encoding provides the best trade-off between microinstruction width and parallelism. Group mutually exclusive signals together and encode within each group. Different groups can be active simultaneously.

5. Microprogram Sequencing

The sequencer determines the address of the next microinstruction. The next address can come from several sources:

5.1 Next Address Generation Methods

Method Description
Sequential CAR ← CAR + 1. The next microinstruction is the next consecutive address.
Unconditional branch CAR ← branch address field of current microinstruction.
Conditional branch If condition is true, CAR ← branch address; else CAR ← CAR + 1.
Mapping (dispatch) CAR ← f(opcode). Maps machine instruction opcode to the starting address of its microprogram.
Subroutine call/return Save CAR on a micro-stack, then branch. On return, pop the stack.

5.2 Mapping Function

When a new machine instruction is fetched, the opcode must be converted to the starting address of its microprogram in control memory. This is called opcode mapping or dispatch.

Common mapping technique: insert the opcode bits into specific positions of the control memory address, with other bits set to 0.

Example mapping: 4-bit opcode mapped to 7-bit CM address
  Opcode:    b3 b2 b1 b0
  CM addr:   0  b3 b2 b1 b0 0 0

  This places each microprogram at intervals of 4 addresses,
  allowing up to 4 microinstructions per routine before
  needing a branch.

  Opcode 0000 -> CM address 0000000 (0)
  Opcode 0001 -> CM address 0000100 (4)
  Opcode 0010 -> CM address 0001000 (8)
  Opcode 1111 -> CM address 0111100 (60)

The MUX selects the next address from one of the following sources based on the condition field:

  1. CAR + 1 — for sequential execution
  2. Branch address — from the address field of the current microinstruction
  3. Mapping output — derived from the opcode for instruction dispatch
  4. Micro-stack top — for subroutine return

6. Wilkes Control Unit

Maurice Wilkes proposed the concept of microprogramming in 1951, well before it was practical to implement. His original design used a two-matrix scheme.

6.1 Wilkes' Two-Matrix Design

graph LR subgraph "Wilkes Control Unit" DEC2["Decoder
(selects row)"] --> M1["Matrix I
(Control Signals)"] DEC2 --> M2["Matrix II
(Next Address)"] M1 --> CS4["Control Signals
to datapath"] M2 --> COND2["Conditional
Logic"] COND2 --> DEC2 end
  • Matrix I (Diode Matrix): Generates control signals. Each row represents one micro-step. Diodes at intersections determine which signals are activated.
  • Matrix II (Diode Matrix): Generates the next address. Provides the address of the next row to be selected in the next clock cycle.
  • A decoder selects one row at a time (corresponding to the current micro-step).
  • Conditional logic can modify the next address based on status flags.

Wilkes' design is essentially a horizontal microprogrammed control unit where Matrix I is the control word and Matrix II provides next-address sequencing. This is the historical foundation of all microprogrammed control units.

Historical Note: Wilkes' idea was impractical in 1951 due to the cost of diode matrices. It became practical in the 1960s with the availability of inexpensive ROM chips. IBM's System/360 (1964) was the first major commercial machine to use microprogrammed control.

7. Nano-programming

Nano-programming adds an additional level of indirection to reduce the total size of control storage. It exploits the fact that many microinstructions share the same control signal pattern.

7.1 Two-Level Control Store

graph TD CAR3["CAR"] --> CMICRO["Control Memory
(Microprogram)
narrow words"] CMICRO --> NANO_ADDR["Nano-address
pointer"] NANO_ADDR --> CNANO["Nano-Memory
(wide words with
actual control signals)"] CNANO --> CS5["Control Signals"] CMICRO --> SEQ2["Sequencing
fields"] SEQ2 --> CAR3

Instead of storing the full (wide) control word in every microinstruction, each microinstruction contains a short pointer into a nano-memory that holds the distinct control signal patterns.

7.2 Size Analysis

Without nano-programming (conventional):
If there are M microinstructions, each W bits wide:
Total control store = M × W bits

With nano-programming:
If there are only D distinct control patterns (D << M):
Microprogram memory = M × (ceil(log2 D) + sequencing bits)
Nano-memory = D × W bits
Total = M × (log2 D + seq) + D × W

Worked Example: 

Conventional:
  M = 2048 microinstructions, W = 80 bits (horizontal)
  Total = 2048 x 80 = 163,840 bits

With nano-programming:
  D = 256 distinct patterns, seq = 12 bits
  Nano-pointer = log2(256) = 8 bits
  Micro-memory = 2048 x (8 + 12) = 2048 x 20 = 40,960 bits
  Nano-memory = 256 x 80 = 20,480 bits
  Total = 40,960 + 20,480 = 61,440 bits

Savings = 163,840 - 61,440 = 102,400 bits (62.5% reduction)
GATE Key Point: Nano-programming saves space when many microinstructions share the same control signal pattern. The trade-off is an additional memory access (two reads instead of one per micro-step), increasing the clock cycle time.

7.3 Real-World Example: Motorola 68000

The Motorola 68000 processor used a two-level micro/nano control store. The microprogram memory contained about 640 17-bit words, and the nano-memory had about 280 68-bit words. This was significantly smaller than a single-level store of 640 × 68 bits.

8. Designing a Simple Microprogrammed Control Unit

Let us design a microprogrammed control unit for a simple CPU with the following specifications:

8.1 CPU Specifications

  • Registers: AC (Accumulator), DR (Data Register), AR (Address Register), PC (Program Counter), IR (Instruction Register)
  • 16-bit data, 12-bit addresses
  • ALU operations: ADD, AND, complement, shift
  • Instructions: LDA, STA, ADD, AND, BUN, BSA, ISZ (7 basic instructions)

8.2 Micro-operations for Instruction Fetch

Every instruction begins with a common fetch cycle:

Step Micro-operation Description
T0 AR ← PC Place PC contents in address register
T1 DR ← M[AR], PC ← PC + 1 Read instruction from memory, increment PC
T2 IR ← DR(opcode), AR ← DR(address) Decode instruction, extract address
T3 Mapping: CAR ← f(opcode) Dispatch to appropriate microprogram

8.3 Microinstruction Format Design

We define a field-encoded microinstruction (20 bits total):

Bit positions:
+-----+-----+-----+------+-----+-----------+
| F1  | F2  | F3  | CD   | BR  | AD        |
| 3b  | 3b  | 3b  | 2b   | 2b  | 7b        |
+-----+-----+-----+------+-----+-----------+

F1 (3 bits): ALU / Register transfer group 1
  000: NOP       001: AC <- AC + DR    010: AC <- AC AND DR
  011: AC <- DR  100: AC <- AC'        101: AC <- shr AC
  110: AC <- shl AC   111: DR <- AC

F2 (3 bits): Register transfer group 2
  000: NOP       001: AC <- 0    010: DR <- M[AR]
  011: M[AR] <- DR   100: PC <- AR   101: AR <- PC
  110: DR <- AC      111: (reserved)

F3 (3 bits): Register transfer group 3
  000: NOP       001: PC <- PC + 1    010: AR <- DR(0-10)
  011: AR <- IR(0-10)  100: DR <- DR + 1  101: DR(15) <- 0
  110: (reserved)      111: (reserved)

CD (2 bits): Condition for branch
  00: Always (unconditional)   01: Branch if DR(15) = 1
  10: Branch if AC(15) = 1     11: Branch if AC = 0

BR (2 bits): Branch type
  00: JMP (jump to AD)    01: CALL (subroutine call to AD)
  10: RET (return)        11: MAP (dispatch using opcode)

8.4 Fetch Cycle Microprogram

CM Addr F1 F2 F3 CD BR AD Micro-operations
0 (000 0000) 000 101 000 00 00 000 0001 AR ← PC
1 (000 0001) 000 010 001 00 00 000 0010 DR ← M[AR], PC ← PC + 1
2 (000 0010) 000 000 010 00 11 000 0000 AR ← DR(addr), MAP to opcode

8.5 ADD Instruction Microprogram

Assume the MAP function maps ADD opcode (001) to CM address 0000100 (4):

CM Addr F1 F2 F3 CD BR AD Micro-operations
4 000 010 000 00 00 000 0101 DR ← M[AR] (read operand)
5 001 000 000 00 00 000 0000 AC ← AC + DR, then FETCH

9. Timing and Performance

9.1 Microprogrammed Control Timing

In a microprogrammed control unit, the clock cycle must be long enough to:

  1. Read a microinstruction from control memory (tCM)
  2. Propagate through the sequencer to determine next address (tseq)
  3. Execute the micro-operation in the datapath (texec)
Micro-cycle time = t_CM + t_seq + t_exec

For nano-programmed control:
Micro-cycle time = t_CM + t_nano + t_seq + t_exec
(additional nano-memory access time)

9.2 Performance Comparison

Worked Example:

Given:
  Hardwired control: clock cycle = 5 ns, ADD takes 4 cycles = 20 ns
  Microprogrammed: clock cycle = 8 ns (slower due to CM read)
    ADD microprogram: fetch (3 steps) + execute (2 steps) = 5 micro-cycles = 40 ns

Speed ratio = 40 / 20 = 2x slower for microprogrammed

But with nano-programming (extra 3 ns per cycle):
  clock cycle = 11 ns, ADD = 5 x 11 = 55 ns
  Speed ratio = 55 / 20 = 2.75x slower
Trade-off Summary: Microprogrammed control sacrifices speed for design flexibility, ease of modification, and the ability to support complex instruction sets. For GATE problems, remember that each micro-step requires one control memory access cycle.

9.3 Techniques to Speed Up Microprogrammed Control

Technique Description
Wide control word Use horizontal encoding to allow multiple micro-operations per cycle
Faster CM Use faster ROM/SRAM for control memory
Micro-pipelining Overlap fetch of next microinstruction with execution of current one
Prefetch buffer Buffer the next sequential microinstruction while branch is resolved
Writable Control Store (WCS) Use RAM instead of ROM — allows runtime microcode updates, but adds write circuitry

10. Advantages and Disadvantages Summary

10.1 Advantages of Microprogrammed Control

  • Systematic design: Regular structure makes design methodical and less error-prone
  • Flexibility: New instructions can be added by writing new microprograms
  • Bug fixes: Errors can be corrected by updating the control memory (firmware update)
  • Emulation: One machine can emulate another by changing the microprogram
  • Complex ISAs: Makes it feasible to implement instructions like string operations, decimal arithmetic
  • Family compatibility: Different hardware implementations can share the same ISA by using different microprograms (e.g., IBM 360 family)

10.2 Disadvantages of Microprogrammed Control

  • Slower: Additional control memory access in the critical path
  • Higher cost: Requires control memory (ROM/RAM) and sequencing hardware
  • Power consumption: Control memory and MIR consume power every cycle
  • Design overhead: Need to design the microinstruction format, microprogram, and sequencer
  • Not suitable for simple ISAs: Overhead is not justified for RISC-style processors

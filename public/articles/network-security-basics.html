Computer Networks

Network Security Fundamentals

Based on Kurose & Ross — Computer Networking: A Top-Down Approach, Chapter 8

Network security is one of the most important and frequently tested topics in the GATE Computer Science exam. This article provides a rigorous treatment of cryptographic primitives, protocols, and network security mechanisms that you need to master for the exam. Every concept is tied back to the layered network model so you can see where each mechanism operates.

1. Security Goals

Before diving into algorithms, we must understand what we are trying to achieve. Network security has four fundamental goals:

GoalMeaningThreat it CountersExample Mechanism
ConfidentialityOnly the intended receiver can read the messageEavesdropping / SniffingAES encryption
IntegrityThe message has not been altered in transitTampering / ModificationSHA-256 hash, HMAC
AuthenticationThe sender is who they claim to beImpersonation / SpoofingDigital signatures, certificates
Non-repudiationThe sender cannot deny having sent the messageRepudiationDigital signatures with PKI
GATE Tip: Authentication proves identity; non-repudiation prevents denial. A MAC provides authentication but not non-repudiation (because both parties share the key). Only digital signatures provide non-repudiation.

1.1 The Alice-and-Bob Model

Throughout cryptography, we use a standard cast of characters:

  • Alice (A) — wants to send a message securely
  • Bob (B) — the intended recipient
  • Trudy — the intruder who may eavesdrop, modify, inject, or delete messages

Trudy can be a passive attacker (only eavesdropping) or an active attacker (modifying, injecting, replaying messages). Good security must handle both.

2. Symmetric Key Cryptography

In symmetric key (also called shared key or secret key) cryptography, the same key K is used for both encryption and decryption:

Ciphertext = E(K, Plaintext)
Plaintext  = D(K, Ciphertext)

Both Alice and Bob must possess the key K before communication begins. This is the key distribution problem.

2.1 DES (Data Encryption Standard)

  • Block cipher: operates on 64-bit blocks
  • Key size: 56 bits (8 parity bits make the key 64 bits total, but effective security is 56 bits)
  • Uses a Feistel structure with 16 rounds
  • Each round uses a different 48-bit subkey derived from the 56-bit master key
  • No longer secure: 256 key space can be brute-forced in hours with modern hardware

2.2 Triple DES (3DES)

To extend the life of DES hardware, 3DES applies DES three times with two or three different keys:

C = E(K1, D(K2, E(K1, P)))    [2-key 3DES — 112 bits effective]
C = E(K3, D(K2, E(K1, P)))    [3-key 3DES — 168 bits effective]
Why Encrypt-Decrypt-Encrypt (EDE)? If K1 = K2 = K3, the EDE sequence reduces to single DES, providing backward compatibility.

2.3 AES (Advanced Encryption Standard)

  • Block size: 128 bits
  • Key sizes: 128, 192, or 256 bits
  • Based on the Rijndael algorithm (substitution-permutation network, not Feistel)
  • Number of rounds: 10 (128-bit key), 12 (192-bit), 14 (256-bit)
  • Each round: SubBytes → ShiftRows → MixColumns → AddRoundKey
  • Current standard; much faster and more secure than 3DES

2.5 Block Cipher Modes of Operation

A block cipher encrypts a fixed-size block. To encrypt messages longer than one block, we need a mode of operation.

ECB — Electronic Codebook

  • Each block is encrypted independently: Ci = E(K, Pi)
  • Weakness: identical plaintext blocks produce identical ciphertext blocks, leaking patterns
  • Never use ECB for data longer than one block

CBC — Cipher Block Chaining

  • Ci = E(K, Pi XOR Ci-1), where C0 = IV (Initialization Vector)
  • Identical plaintext blocks yield different ciphertext (because of chaining)
  • Requires padding if plaintext is not a multiple of block size
  • Encryption is sequential; decryption can be parallelized

CFB — Cipher Feedback

  • Turns a block cipher into a stream cipher
  • Ci = Pi XOR E(K, Ci-1)
  • No padding needed; can operate on units smaller than the block size

OFB — Output Feedback

  • Oi = E(K, Oi-1), Ci = Pi XOR Oi
  • Keystream can be pre-computed (does not depend on plaintext or ciphertext)
  • A bit error in ciphertext affects only the corresponding plaintext bit

CTR — Counter Mode

  • Ci = Pi XOR E(K, Nonce || Counteri)
  • Fully parallelizable for both encryption and decryption
  • Random access to any block
  • Preferred mode in modern systems (used in AES-GCM)

Mode Comparison Table

ModeParallel EncryptParallel DecryptPadding NeededError PropagationRandom Access
ECBYesYesYesSingle blockYes
CBCNoYesYesCurrent + next blockNo
CFBNoYesNoCurrent + next blockNo
OFBNoNoNoSingle bitNo
CTRYesYesNoSingle bitYes
GATE Favourite: "Which mode allows parallel encryption?" — Answer: ECB and CTR. CBC, CFB, OFB require sequential encryption because each block depends on the previous output.

2.6 The Key Distribution Problem

If Alice and Bob want to use AES, they must share a secret key before communicating. How do they establish this key securely?

  • Out-of-band exchange: Meet physically and exchange keys (impractical at scale)
  • Key Distribution Center (KDC): A trusted third party that shares a unique key with each user. The KDC generates a session key and sends it encrypted under each user's master key.
  • Diffie-Hellman key exchange: Allows two parties to agree on a shared secret over an insecure channel (see Section 3.2)
  • Public key cryptography: Alice encrypts the session key with Bob's public key (see Section 3)

3. Public Key Cryptography

Public key (asymmetric) cryptography uses a pair of keys:

  • Public key (Kpub): known to everyone
  • Private key (Kpriv): known only to the owner
Encryption:  C = E(K_pub, P)
Decryption:  P = D(K_priv, C)

The critical property: it is computationally infeasible to derive Kpriv from Kpub.

3.1 RSA Algorithm

RSA (Rivest-Shamir-Adleman) is the most important public key algorithm for GATE. Its security rests on the difficulty of integer factorization.

Step 1: Key Generation

  1. Choose two large primes p and q
  2. Compute n = p * q
  3. Compute Euler's totient: φ(n) = (p - 1)(q - 1)
  4. Choose e such that 1 < e < φ(n) and gcd(e, φ(n)) = 1
  5. Compute d such that e * d ≡ 1 (mod φ(n)), i.e., d = e-1 mod φ(n)
  6. Public key: (e, n)
  7. Private key: (d, n)

Step 2: Encryption and Decryption

Encryption:  C = M^e mod n
Decryption:  M = C^d mod n

This works because of Euler's theorem: Med mod n = M when ed ≡ 1 (mod φ(n)).

Worked Example: RSA with Small Numbers

Problem: Generate RSA keys with p = 7, q = 11. Encrypt the message M = 3.

Step 1: n = 7 × 11 = 77

Step 2: φ(n) = (7 - 1)(11 - 1) = 6 × 10 = 60

Step 3: Choose e such that gcd(e, 60) = 1. Let e = 13.
Check: gcd(13, 60) = 1. Valid.

Step 4: Find d such that 13 × d ≡ 1 (mod 60).
Using the extended Euclidean algorithm:
60 = 4 × 13 + 8
13 = 1 × 8 + 5
8 = 1 × 5 + 3
5 = 1 × 3 + 2
3 = 1 × 2 + 1
Back-substitute: 1 = 3 - 1×2 = 3 - 1×(5 - 3) = 2×3 - 5 = 2(8-5) - 5 = 2×8 - 3×5 = 2×8 - 3(13-8) = 5×8 - 3×13 = 5(60 - 4×13) - 3×13 = 5×60 - 23×13
So d ≡ -23 ≡ 37 (mod 60). Verify: 13 × 37 = 481 = 8 × 60 + 1. Correct!

Public key: (e, n) = (13, 77)
Private key: (d, n) = (37, 77)

Encryption: C = 313 mod 77
31 = 3
32 = 9
34 = 81 mod 77 = 4
38 = 42 = 16
313 = 38 × 34 × 31 = 16 × 4 × 3 = 192 mod 77 = 192 - 2×77 = 38

Decryption: M = 3837 mod 77
382 = 1444 mod 77 = 1444 - 18×77 = 1444 - 1386 = 58
384 = 582 = 3364 mod 77 = 3364 - 43×77 = 3364 - 3311 = 53
388 = 532 = 2809 mod 77 = 2809 - 36×77 = 2809 - 2772 = 37
3816 = 372 = 1369 mod 77 = 1369 - 17×77 = 1369 - 1309 = 60

We recover M = 3. RSA works!

3.2 Diffie-Hellman Key Exchange

Diffie-Hellman allows two parties to establish a shared secret over an insecure channel. Its security is based on the discrete logarithm problem.

Protocol Steps

  1. Alice and Bob publicly agree on a large prime p and a generator g (where g is a primitive root mod p)
  2. Alice picks a secret integer a, computes A = ga mod p, sends A to Bob
  3. Bob picks a secret integer b, computes B = gb mod p, sends B to Alice
  4. Alice computes the shared secret: S = Ba mod p = gba mod p
  5. Bob computes the shared secret: S = Ab mod p = gab mod p
  6. Both arrive at the same value S, which Trudy cannot compute without knowing a or b

Worked Example: Diffie-Hellman

Given: p = 23, g = 5

Alice chooses a = 6, computes A = 56 mod 23
52 = 25 mod 23 = 2
54 = 22 = 4
56 = 54 × 52 = 4 × 2 = 8
Alice sends A = 8 to Bob.

Bob chooses b = 15, computes B = 515 mod 23
51 = 5, 52 = 2, 54 = 4, 58 = 42 = 16
515 = 58 × 54 × 52 × 51 = 16 × 4 × 2 × 5 = 640 mod 23
640 / 23 = 27 remainder 19. So B = 19
Bob sends B = 19 to Alice.

Alice computes S = Ba mod p = 196 mod 23
192 = 361 mod 23 = 361 - 15×23 = 361 - 345 = 16
194 = 162 = 256 mod 23 = 256 - 11×23 = 256 - 253 = 3
196 = 194 × 192 = 3 × 16 = 48 mod 23 = 2

Bob computes S = Ab mod p = 815 mod 23
82 = 64 mod 23 = 64 - 2×23 = 18
84 = 182 = 324 mod 23 = 324 - 14×23 = 324 - 322 = 2
88 = 22 = 4
815 = 88 × 84 × 82 × 81 = 4 × 2 × 18 × 8 = 1152 mod 23
1152 / 23 = 50 remainder 2. So S = 2

Both Alice and Bob compute S = 2. They now share a secret!

Vulnerability: Diffie-Hellman is vulnerable to a man-in-the-middle (MITM) attack. Trudy can intercept A and B, establishing separate shared secrets with Alice and Bob. This is why DH must be combined with authentication (certificates, digital signatures).

Practical systems use both: Public key crypto to exchange a symmetric session key, then symmetric crypto for the actual data. This is exactly what TLS does.

4. Message Integrity and Digital Signatures

4.1 Cryptographic Hash Functions

A hash function H takes an arbitrary-length input and produces a fixed-length output (the digest or hash). A cryptographic hash function must satisfy:

  • Pre-image resistance: Given h, it is infeasible to find any m such that H(m) = h
  • Second pre-image resistance: Given m1, infeasible to find m2 ≠ m1 with H(m1) = H(m2)
  • Collision resistance: Infeasible to find any pair (m1, m2) with m1 ≠ m2 and H(m1) = H(m2)
Hash FunctionDigest SizeBlock SizeStatus
MD5128 bits512 bitsBroken (collisions found)
SHA-1160 bits512 bitsDeprecated (collisions found 2017)
SHA-256256 bits512 bitsSecure, widely used
SHA-384384 bits1024 bitsSecure
SHA-512512 bits1024 bitsSecure
Birthday Attack: Due to the birthday paradox, finding a collision in a hash function with n-bit output takes approximately 2n/2 operations, not 2n. For MD5 (128-bit): collision in ~264 operations. For SHA-256: ~2128 operations.

4.2 Message Authentication Code (MAC)

A MAC provides integrity + authentication using a shared secret key:

MAC = H(K || M)        [Simple but vulnerable to length-extension]
MAC = HMAC(K, M)       [Preferred — secure construction]

HMAC Construction

HMAC(K, M) = H( (K XOR opad) || H( (K XOR ipad) || M ) )
where:
  opad = 0x5c repeated
  ipad = 0x36 repeated

Properties of MAC:

  • Provides integrity: any modification to M changes the MAC
  • Provides authentication: only someone with K can produce a valid MAC
  • Does NOT provide non-repudiation: both Alice and Bob know K, so either could have generated the MAC

4.3 Digital Signatures

A digital signature provides integrity + authentication + non-repudiation. The signer uses their private key to create the signature; anyone can verify it using the signer's public key.

RSA Digital Signature

Signing:      S = H(M)^d mod n     [Sign the hash with private key]
Verification: H(M) == S^e mod n     [Verify with public key]

Why hash first? RSA is slow. Signing the entire message would be impractical. Instead, we hash the message (fast, fixed-size output) and sign only the hash.

Worked Example: RSA Signature

Using our earlier keys: public (e=13, n=77), private (d=37, n=77)

Message hash: Suppose H(M) = 5

Signing: S = 537 mod 77
52 = 25
54 = 625 mod 77 = 625 - 8×77 = 625 - 616 = 9
58 = 81 mod 77 = 4
516 = 16
532 = 256 mod 77 = 256 - 3×77 = 256 - 231 = 25
537 = 532 × 54 × 51 = 25 × 9 × 5 = 1125 mod 77 = 1125 - 14×77 = 1125 - 1078 = 47

Verification: 4713 mod 77
472 = 2209 mod 77 = 2209 - 28×77 = 2209 - 2156 = 53
474 = 532 = 2809 mod 77 = 2809 - 36×77 = 2809 - 2772 = 37
478 = 372 = 1369 mod 77 = 1369 - 17×77 = 1369 - 1309 = 60
4713 = 478 × 474 × 471 = 60 × 37 × 47 mod 77
60 × 37 = 2220 mod 77 = 2220 - 28×77 = 2220 - 2156 = 64
64 × 47 = 3008 mod 77 = 3008 - 39×77 = 3008 - 3003 = 5
We get 5, which matches H(M). Signature is valid!

5. Certificate Authorities and PKI

Public key cryptography solves the key distribution problem, but introduces a new one: how do you trust a public key? Trudy could give you her public key and claim it belongs to Bob.

5.1 Certificate Authority (CA)

A CA is a trusted third party that binds a public key to an identity by issuing a digital certificate.

A certificate contains:

  • Subject's identity (name, domain, organization)
  • Subject's public key
  • CA's identity
  • Validity period
  • CA's digital signature over all the above

5.2 PKI (Public Key Infrastructure)

PKI is the entire ecosystem of CAs, certificates, and protocols:

  • Root CA: Self-signed; pre-installed in browsers/OS
  • Intermediate CA: Certified by Root CA; issues end-entity certificates
  • Certificate chain: End-entity cert → Intermediate CA cert → Root CA cert
  • Certificate Revocation List (CRL): List of revoked certificates
  • OCSP: Online Certificate Status Protocol — real-time revocation check
X.509 is the standard format for public key certificates. It uses ASN.1 encoding and is the basis for TLS/SSL certificates.

6. SSL/TLS Protocol

SSL (Secure Sockets Layer) and its successor TLS (Transport Layer Security) operate between the application layer and transport layer, providing secure communication.

6.1 TLS Handshake (Simplified)

sequenceDiagram participant C as Client participant S as Server C->>S: ClientHello (supported ciphers, random_C) S->>C: ServerHello (chosen cipher, random_S) S->>C: Server Certificate (public key) S->>C: ServerHelloDone C->>S: ClientKeyExchange (Pre-Master Secret encrypted with server's public key) Note over C,S: Both compute Master Secret = PRF(PMS, random_C, random_S) Note over C,S: Derive session keys from Master Secret C->>S: ChangeCipherSpec + Finished (encrypted) S->>C: ChangeCipherSpec + Finished (encrypted) Note over C,S: Application data (encrypted with session keys)

6.2 Key Derivation

From the Master Secret, four session keys are derived:

KeyPurpose
Client write encryption key (Kc)Encrypts data from client to server
Server write encryption key (Ks)Encrypts data from server to client
Client write MAC key (Mc)MACs data from client to server
Server write MAC key (Ms)MACs data from server to client
Why separate keys? Using different keys for each direction prevents reflection attacks where an attacker sends a party's own encrypted data back to them.

6.3 TLS Record Protocol

After the handshake, each TLS record is processed as follows:

  1. Fragment the application data
  2. Compute MAC over fragment + sequence number + header
  3. Append MAC to fragment
  4. Encrypt (fragment + MAC) with the session encryption key
  5. Add TLS record header
GATE Note: TLS provides confidentiality (encryption), integrity (MAC), and authentication (certificates). The sequence number in the MAC prevents replay attacks.

7. Firewalls

A firewall isolates an organization's internal network from the public Internet, allowing some traffic to pass and blocking the rest.

7.1 Types of Firewalls

Packet Filtering Firewall (Stateless)

  • Operates at Network and Transport layers (examines IP and TCP/UDP headers)
  • Filters based on: source/destination IP, source/destination port, protocol, interface
  • Each packet is examined independently — no connection state
  • Implemented using an Access Control List (ACL)

Example ACL Rules:

RuleSrc IPDst IPProtocolDst PortAction
1Any222.22.0.0/16TCP80Allow
2222.22.0.0/16AnyTCP>1023Allow
3AnyAnyAnyAnyDeny

Rule 1 allows incoming HTTP requests; Rule 2 allows responses from the web server; Rule 3 is the default deny.

Stateful Inspection Firewall

  • Maintains a connection table tracking the state of each TCP connection
  • Allows return packets only for established connections
  • Blocks packets that do not belong to an existing connection
  • More secure than stateless: prevents many spoofing attacks

Application Gateway (Proxy Firewall)

  • Operates at the Application layer
  • Understands application protocols (HTTP, FTP, DNS)
  • Can inspect packet content, block specific URLs, scan for malware
  • Acts as an intermediary: client connects to proxy, proxy connects to server
  • Slower but much more thorough inspection

8. IPSec

IPSec (IP Security) provides security at the network layer (Layer 3). It can protect all traffic between two hosts or networks without any changes to applications.

8.1 IPSec Protocols

AH — Authentication Header

  • Provides: authentication + integrity
  • Does NOT provide: confidentiality (no encryption)
  • Covers the entire IP packet including (most) header fields
  • IP Protocol number: 51

ESP — Encapsulating Security Payload

  • Provides: confidentiality + authentication + integrity
  • Encrypts and authenticates the payload
  • Does NOT authenticate the outer IP header
  • IP Protocol number: 50

8.2 IPSec Modes

FeatureTransport ModeTunnel Mode
ProtectionOnly the IP payloadEntire original IP packet
IP headerOriginal header preservedNew outer header added
Use caseHost-to-host communicationGateway-to-gateway (VPN)
Packet structure (ESP)[IP Hdr][ESP Hdr][Payload][ESP Trailer][ESP Auth][New IP Hdr][ESP Hdr][Original IP + Payload][ESP Trailer][ESP Auth]
IKE (Internet Key Exchange) is the protocol used to set up Security Associations (SAs) in IPSec. IKE uses Diffie-Hellman for key exchange and can use certificates or pre-shared keys for authentication.

8.3 Security Association (SA)

An SA is a one-way logical connection that defines the security parameters:

  • SPI (Security Parameter Index): 32-bit identifier
  • Protocol: AH or ESP
  • Encryption algorithm and key
  • Authentication algorithm and key
  • Mode: Transport or Tunnel

SAs are stored in the Security Association Database (SAD). The Security Policy Database (SPD) determines which packets should be protected and how.

GATE Favourite: IPSec SAs are unidirectional. For bidirectional communication, you need two SAs (one in each direction).

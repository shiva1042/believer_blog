1. Integrity Constraints

Integrity constraints are rules that ensure the accuracy and consistency of data in a relational database.

ConstraintDescriptionExample

2. Functional Dependencies

A functional dependency (FD) X → Y means that for any two tuples, if they agree on X, they must agree on Y. X is called the determinant and Y is the dependent.

Example: In Student(StudentID, Name, Dept, DeptHead):
  • StudentID → Name, Dept (student determines name and department)
  • Dept → DeptHead (department determines department head)
  • StudentID → DeptHead (transitively, student determines department head)

2.1 Types of Functional Dependencies

TypeDefinitionExample
Trivial FDX → Y where Y ⊆ X{A, B} → A
Non-trivial FDX → Y where Y ⊄ XStudentID → Name
Full FDX → Y and no proper subset of X determines Y{StudentID, CourseID} → Grade
Partial FDX → Y and some proper subset of X determines Y{StudentID, CourseID} → Name (StudentID alone determines Name)
Transitive FDX → Z via X → Y and Y → Z, where Y is not a superkeyStudentID → Dept → DeptHead

2.2 Armstrong's Axioms

Armstrong's axioms are sound and complete inference rules for functional dependencies:

AxiomRuleDescription
ReflexivityIf Y ⊆ X, then X → YA set of attributes determines any subset of itself
AugmentationIf X → Y, then XZ → YZAdding the same attributes to both sides preserves the FD
TransitivityIf X → Y and Y → Z, then X → ZFDs can be chained

Derived rules:

RuleStatement
UnionIf X → Y and X → Z, then X → YZ
DecompositionIf X → YZ, then X → Y and X → Z
PseudotransitivityIf X → Y and WY → Z, then WX → Z

2.3 Closure and Candidate Keys

The attribute closure X+ is the set of all attributes functionally determined by X. If X+ contains all attributes of the relation, X is a superkey. A candidate key is a minimal superkey.

Example: R(A, B, C, D), FDs: A → B, B → C, A → D.
  • A+ = {A, B, C, D} — A is a superkey
  • No proper subset of {A} determines everything, so A is a candidate key

3. Normal Forms

3.1 Normalization Process Flowchart

3.2 First Normal Form (1NF)

A relation is in 1NF if all attributes contain only atomic (indivisible) values. No repeating groups or multivalued attributes.

Not in 1NF: Student(ID, Name, PhoneNumbers) where PhoneNumbers can hold multiple values.
In 1NF: Create a separate row for each phone number, or a separate table StudentPhone(ID, Phone).

3.3 Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and every non-prime attribute is fully functionally dependent on the entire candidate key (no partial dependencies).

Not in 2NF: Enrollment(StudentID, CourseID, Grade, StudentName)
StudentID → StudentName is a partial dependency (StudentName depends only on part of the key).
Fix: Decompose into Student(StudentID, StudentName) and Enrollment(StudentID, CourseID, Grade).
Note: If a relation has a single-attribute candidate key, it is automatically in 2NF (no partial dependency is possible).

3.4 Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and no non-prime attribute is transitively dependent on the candidate key.

Formally: For every FD X → A, at least one of the following holds:

  • X → A is trivial (A ∈ X)
  • X is a superkey
  • A is a prime attribute (part of some candidate key)
Not in 3NF: Student(StudentID, Name, Dept, DeptHead)
StudentID → Dept → DeptHead is a transitive dependency.
Fix: Student(StudentID, Name, Dept) and Department(Dept, DeptHead).

3.5 Boyce-Codd Normal Form (BCNF)

A relation is in BCNF if for every non-trivial FD X → Y, X is a superkey. BCNF is stricter than 3NF (the third condition about prime attributes is removed).

Normal FormCondition
1NFAtomic values only
2NF1NF + no partial dependencies
3NF2NF + no transitive dependencies (or every determinant is a superkey, or dependent is prime)
BCNFFor every X → Y, X is a superkey (strictly stronger than 3NF)
3NF vs BCNF: Every BCNF relation is in 3NF, but not vice versa. The difference arises when a non-trivial FD has a determinant that is not a superkey but the dependent is a prime attribute. 3NF allows this; BCNF does not.

4. Decomposition Properties

4.1 Lossless Join Decomposition

A decomposition of R into R1 and R2 is lossless (non-additive) if joining R1 and R2 produces exactly R (no spurious tuples). The condition is:

  • R1 ∩ R2 → R1, or
  • R1 ∩ R2 → R2

That is, the common attributes must be a superkey of at least one of the decomposed relations.

4.2 Dependency Preservation

A decomposition is dependency-preserving if the union of FDs that can be checked in individual decomposed relations is equivalent to the original set of FDs. This means no FD requires a join to verify.

4.3 Key Differences

Property3NF DecompositionBCNF Decomposition
Lossless JoinAlways achievableAlways achievable
Dependency PreservationAlways achievableNot always achievable
RedundancySome redundancy may remainNo redundancy from FDs
GATE Tip: It is always possible to decompose into 3NF with both lossless join and dependency preservation. For BCNF, lossless join is always possible but dependency preservation may be sacrificed.

4.4 Synthesis Algorithm for 3NF

  1. Find a minimal cover (canonical cover) of the FDs.
  2. For each FD X → A in the minimal cover, create a relation R(X, A).
  3. If no relation contains a candidate key, add a relation containing a candidate key.
  4. Remove redundant relations (those whose attributes are a subset of another).

4.5 Decomposition Algorithm for BCNF

  1. Find a non-trivial FD X → Y that violates BCNF (X is not a superkey).
  2. Decompose R into R1(X, Y) and R2(R − Y).
  3. Repeat until all relations are in BCNF.

5. Summary and GATE Key Points

Key Points for GATE:
  • Entity integrity: PK cannot be NULL. Referential integrity: FK must reference existing PK or be NULL.
  • Armstrong's axioms (reflexivity, augmentation, transitivity) are sound and complete.
  • To find candidate keys: compute closure of every possible attribute combination. Attributes that appear only on the left side of FDs must be in every candidate key.
  • A relation with all prime attributes is always in 3NF.
  • If a relation has a single-attribute key, it is automatically in 2NF.
  • Every 2-attribute relation is in BCNF.
  • BCNF eliminates all redundancy due to FDs; 3NF may retain some.
  • Lossless join test: R1 ∩ R2 must be a key of R1 or R2.
  • Number of candidate keys can be exponential in the number of attributes (worst case).
  • Minimal cover: no extraneous attributes on the left, no redundant FDs, right side is a single attribute.

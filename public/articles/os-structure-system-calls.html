Operating System

OS Structure, System Calls, and Kernel Design

Reference: Silberschatz, Galvin, Gagne — Operating System Concepts (Chapters 1–2)

Understanding how an operating system is structured and how user programs communicate with the kernel through system calls is fundamental to cracking GATE CS questions on operating systems. This article provides a comprehensive treatment of OS services, the system call interface, parameter-passing mechanisms, and every major kernel architecture — all illustrated with diagrams, comparison tables, and worked examples.

1. Operating System Services

An operating system provides a set of services that are helpful to the user and to the system itself. These can be grouped into two broad categories.

1.1 Services for the User

  • User Interface (UI): Command-line interface (CLI), Graphical User Interface (GUI), or touch-screen interface.
  • Program Execution: The OS loads a program into memory, runs it, and terminates it (normally or abnormally).
  • I/O Operations: Users cannot directly access I/O devices; the OS provides controlled access.
  • File-System Manipulation: Create, delete, read, write, search files/directories; manage permissions.
  • Communications: Inter-process communication via shared memory or message passing.
  • Error Detection: The OS must detect and handle errors in CPU, memory, I/O, and user programs.

1.2 Services for Efficient System Operation

  • Resource Allocation: CPU scheduling cycles, memory blocks, I/O devices — all shared among concurrent processes.
  • Logging / Accounting: Track resource usage per user or process (important for billing and tuning).
  • Protection and Security: Ensure controlled access to resources; authenticate users; defend against external attacks.

2. System Calls

A system call is the programmatic interface through which a user-level process requests a service from the operating system kernel. System calls form the boundary between user mode and kernel mode.

2.1 System Call Interface and API

Most programmers never make raw system calls. Instead, they use a higher-level Application Programming Interface (API) such as:

  • Win32 API — Windows
  • POSIX API — UNIX, Linux, macOS
  • Java API — Java Virtual Machine

The API function internally invokes the correct system call. The system call interface maintains an indexed table of system call handlers. The caller passes a system call number (index) and the OS dispatches to the corresponding kernel routine.

sequenceDiagram participant App as User Application participant API as API Layer (e.g. libc) participant SCI as System Call Interface participant Kernel as OS Kernel App->>API: open("file.txt", O_RDONLY) API->>SCI: syscall #5 (sys_open) SCI->>Kernel: Dispatch to sys_open handler Kernel-->>SCI: Return file descriptor (fd=3) SCI-->>API: Return fd API-->>App: Return fd=3
GATE Tip: A system call causes a trap (software interrupt) that switches the CPU from user mode to kernel mode. After servicing, the CPU returns to user mode.

2.2 Types of System Calls

Galvin classifies system calls into five categories:

Category Examples (UNIX / Linux) Purpose
Process Control fork(), exec(), exit(), wait(), kill() Create, terminate, load, execute processes; wait for events
File Management open(), close(), read(), write(), lseek() Create, delete, open, close, read, write files
Device Management ioctl(), read(), write() Request/release device, read/write device, get/set attributes
Information Maintenance getpid(), alarm(), time(), uname() Get/set time, date, system data, process attributes
Communication pipe(), shmget(), mmap(), socket(), send(), recv() Create connection, send/receive messages, shared memory

2.3 System Call Parameter Passing Methods

When a system call requires parameters (e.g., filename, buffer address, size), the OS must receive them from the user process. Three general methods exist:

Method Mechanism Advantage Limitation
Registers Parameters are placed in CPU registers before the trap instruction Fastest; no memory access needed Number of registers is limited; fails for large parameter sets
Block / Table Parameters stored in a block (table) in memory; address of block passed in a register No limit on number of parameters Extra memory access to read the block
Stack Parameters pushed onto the program stack by the user; popped by the OS No limit on number or length of parameters Slightly slower due to push/pop overhead
Key Point: Linux on x86-64 uses registers for system call parameters (up to 6 parameters in rdi, rsi, rdx, r10, r8, r9), with the system call number in rax. If more than 6 parameters are needed, a pointer to a memory block is passed.

2.4 The fork() and exec() System Calls — Worked Example

The fork() system call creates a child process that is a copy of the parent. The exec() family replaces the process image with a new program.

#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();   // System call: process control

    if (pid < 0) {
        // fork failed
        perror("fork");
        return 1;
    } else if (pid == 0) {
        // Child process
        printf("Child PID = %d\n", getpid());   // System call: info maintenance
        execlp("/bin/ls", "ls", "-l", NULL);     // System call: process control
    } else {
        // Parent process
        wait(NULL);  // System call: process control
        printf("Child finished. Parent PID = %d\n", getpid());
    }
    return 0;
}

Execution trace:

  1. fork() creates child → both processes continue from after fork().
  2. Child (pid == 0): calls execlp to replace itself with ls -l.
  3. Parent (pid > 0): waits for child to finish via wait().
  4. After child exits, parent resumes and prints its message.
GATE Tip: fork() returns 0 to the child and the child's PID to the parent. After fork(), both parent and child execute concurrently (the order depends on the scheduler).

2.5 Counting fork() Calls — Classic GATE Problem

Problem: How many processes are created by the following code?

fork();
fork();
fork();

Solution: Each fork() doubles the number of processes.

  • After 1st fork(): 2 processes
  • After 2nd fork(): 4 processes
  • After 3rd fork(): 8 processes

Formula: With n consecutive fork() calls, total processes = 2n.
New child processes created = 2n − 1 = 8 − 1 = 7.

3. User Mode vs Kernel Mode

Modern CPUs support at least two modes of operation indicated by a mode bit in the processor status register:

Aspect User Mode (Mode bit = 1) Kernel Mode (Mode bit = 0)
Privilege Restricted; cannot execute privileged instructions Full access to all instructions and hardware
Memory Access Only own address space Entire physical memory
Transition Switches to kernel mode via system call (trap) Returns to user mode after servicing
Examples of code Application programs, shell commands Interrupt handlers, device drivers, scheduler
Failure impact Only the process crashes Entire system may crash (kernel panic)

(mode bit = 1)"] KM["Kernel Mode
(mode bit = 0)"] UM -->|"System Call / Interrupt / Trap"| KM KM -->|"Return from kernel
(iret / sysret)"| UM
Privileged Instructions (can only execute in kernel mode):
  • I/O instructions (in, out)
  • Halt instruction (hlt)
  • Interrupt enable/disable (cli, sti)
  • Modify page table base register
  • Switch from kernel to user mode (setting mode bit)
  • Access I/O ports
Note: A trap to OS (system call) is not a privileged instruction — it is executed from user mode. Switching from user mode to kernel mode is not privileged (the hardware does it on trap). But switching from kernel to user mode (clearing the mode bit) IS privileged.

4. Operating System Structures

The internal design of an OS can vary dramatically. The structure determines modularity, performance, reliability, and ease of maintenance. We cover six major designs.

4.1 Simple / Monolithic Structure

The earliest operating systems had no well-defined structure. All OS functionality was packed into a single program running in kernel mode.

Example: MS-DOS

  • No separation between user and kernel space (no dual mode).
  • Applications could directly access hardware and I/O ports.
  • No memory protection — a buggy application could crash the entire system.

Example: Early UNIX (Traditional Monolithic)

  • Two separable parts: kernel and system programs.
  • The kernel itself was one massive program in kernel mode.
  • Everything below the system call interface and above the hardware constitutes the kernel.
  • File system, CPU scheduling, memory management, device drivers — all in one binary.

Advantages Disadvantages
High performance (no inter-module IPC overhead) Difficult to maintain and extend
Direct function calls within the kernel A bug in any component can crash the whole OS
Simple to implement for small systems Poor isolation — no modularity

4.2 Layered Approach

The OS is divided into N layers (numbered 0 to N−1). Layer 0 is hardware; layer N−1 is the user interface. Each layer uses functions and services only of lower layers.

graph TB L5["Layer N-1: User Interface"] L4["Layer N-2: I/O Management"] L3["Layer N-3: Device Drivers"] L2["Layer 2: Memory Management"] L1["Layer 1: CPU Scheduling"] L0["Layer 0: Hardware"] L5 --> L4 L4 --> L3 L3 --> L2 L2 --> L1 L1 --> L0
Advantages Disadvantages
Simplicity of construction and debugging Difficult to define layers (circular dependencies)
Each layer verified independently Performance overhead: each layer adds function call overhead
Good abstraction and information hiding Inflexible: changing one layer may affect others
GATE Tip: The THE operating system (by Dijkstra, 1968) was the first layered OS. Layer ordering is critical: if the file system needs virtual memory, it must be in a higher layer than the memory manager.

4.3 Microkernel

A microkernel moves as much functionality as possible out of the kernel into user-space servers. The kernel retains only the absolute minimum:

  • Inter-process communication (IPC)
  • Basic memory management
  • CPU scheduling (minimal)

Services like file systems, device drivers, and networking run as user-space processes and communicate via message passing.

Examples: Mach, QNX, L4, MINIX 3, seL4

Advantages Disadvantages
High reliability — failure in a server doesn't crash the kernel Performance penalty due to frequent user-kernel mode switches and IPC
Easy to extend — add new services without modifying the kernel Increased system call overhead (message passing vs direct function call)
More secure — services have limited privileges More complex design for achieving good performance
Portable — only the small kernel needs to be ported —
GATE Favorite: In a microkernel, a simple file read requires: Application → message to kernel → kernel forwards to file server → file server processes → message back through kernel → Application. This involves at least 2 context switches and 2 message copies more than a monolithic kernel.

4.4 Modular Approach (Loadable Kernel Modules)

The modern standard. The kernel has a core set of components and can dynamically load additional modules at boot time or runtime (e.g., Linux kernel modules via insmod/modprobe).

  • Each module has a defined interface and can communicate with any other module.
  • Similar to a layered system but more flexible — any module can call any other module (not restricted to adjacent layers).
  • Similar to microkernel but more efficient — modules run in kernel space (no IPC overhead).

Example: Linux, Solaris

4.5 Hybrid Systems

Most modern operating systems are not purely one type. They combine multiple approaches for performance, security, and modularity.

macOS / iOS (XNU kernel)

  • Combines Mach microkernel + BSD monolithic components in a single kernel address space.
  • Mach provides IPC, memory management, thread scheduling.
  • BSD provides POSIX API, file system, networking.
  • I/O Kit (modular framework) for device drivers.

Windows NT

  • Microkernel-inspired design but most services (file system, networking, graphics) run in kernel mode for performance.
  • Executive layer provides object management, I/O, security, etc.
  • HAL (Hardware Abstraction Layer) at the bottom.

Android

  • Based on Linux kernel (monolithic with loadable modules).
  • Adds Android Runtime (ART), middleware, and application framework on top.

4.6 Exokernel

An exokernel provides minimal abstraction. It allocates physical resources (CPU time, memory pages, disk blocks) directly to application-level library operating systems (libOSes). Each application can implement its own OS abstractions.

  • The exokernel only ensures protection (no application can access another's resources).
  • Applications have maximum flexibility to optimize resource usage.
  • Developed at MIT (Aegis / ExOS).

5. System Boot Process

The process of starting an OS involves several stages:

  1. Power On / Reset: CPU begins executing at a fixed memory address (reset vector).
  2. Firmware (BIOS/UEFI): Initializes hardware, performs POST (Power-On Self Test), locates bootable device.
  3. Boot Loader: A small program (e.g., GRUB for Linux, Windows Boot Manager) loaded from the boot sector. It knows just enough to load the OS kernel into memory.
  4. Kernel Initialization: The kernel initializes its data structures, starts drivers, mounts root file system.
  5. Init Process: The first user-space process (PID 1) is started. On modern Linux, this is systemd. It spawns all other services and login processes.

Bootstrap Program / Bootstrap Loader: The firmware / boot loader is sometimes collectively called the bootstrap program. It is stored in ROM or EEPROM (non-volatile). On modern UEFI systems, the boot manager is more sophisticated and can understand file systems.

6. Virtual Machines

A virtual machine (VM) provides an interface identical to the underlying bare hardware. The operating system creates the illusion that a process has its own processor and memory (via virtual CPU and virtual memory).

6.1 Types of Virtualization

Type Description Example
Type 1 Hypervisor (Bare-metal) Runs directly on hardware; manages guest OSes VMware ESXi, Xen, Hyper-V
Type 2 Hypervisor (Hosted) Runs on a host OS as an application VirtualBox, VMware Workstation
Paravirtualization Guest OS is modified to work with hypervisor for better performance Xen (paravirt mode)
Containers OS-level virtualization; share host kernel Docker, LXC
graph TB subgraph "Type 1 Hypervisor" G1["Guest OS 1"] G2["Guest OS 2"] G3["Guest OS 3"] HYP1["Hypervisor (bare-metal)"] HW4["Hardware"] G1 --> HYP1 G2 --> HYP1 G3 --> HYP1 HYP1 --> HW4 end

6.2 Benefits of Virtual Machines

  • Isolation: Each VM is completely isolated from others.
  • Consolidation: Multiple OS instances on a single physical machine.
  • Testing / Development: Run different OS versions safely.
  • Security: If one VM is compromised, others are unaffected.
  • Live Migration: Move running VMs between physical machines.
GATE Concept — Trap-and-Emulate: In Type 1 hypervisors, when a guest OS tries to execute a privileged instruction, it traps to the hypervisor. The hypervisor emulates the instruction on behalf of the guest. This requires that all sensitive instructions are a subset of privileged instructions (true for most architectures, but was historically problematic on x86 before hardware virtualization support like Intel VT-x).

7. Worked Examples for GATE

Example 1: Identifying System Calls

Problem: In the following C program, identify all the system calls:

#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>

int main() {
    int fd = open("data.txt", O_RDONLY);   // (1) open - file management
    char buf[100];
    int n = read(fd, buf, 100);            // (2) read - file management
    write(STDOUT_FILENO, buf, n);          // (3) write - file management (or device)
    close(fd);                             // (4) close - file management
    return 0;                              // exit() called implicitly - process control
}

Answer: Five system calls: open(), read(), write(), close(), and the implicit exit() when main() returns.

Note: printf() is a library function, not a system call. Internally, printf() eventually calls the write() system call. Similarly, malloc() is a library function that internally may invoke brk() or mmap() system calls.

Example 2: fork() Output Analysis

Problem: What is the output of the following program?

#include <stdio.h>
#include <unistd.h>

int main() {
    int x = 10;
    pid_t pid = fork();
    if (pid == 0) {
        x = x + 5;
    } else {
        x = x - 5;
    }
    printf("x = %d\n", x);
    return 0;
}

Solution:

  • After fork(), both parent and child have their own copy of x = 10.
  • Child (pid == 0): x = 10 + 5 = 15 → prints x = 15
  • Parent (pid > 0): x = 10 - 5 = 5 → prints x = 5

Output (order may vary):

x = 15
x = 5

Example 3: Number of printf Executions

Problem: How many times is "hello" printed?

fork();
fork();
printf("hello\n");
fork();
printf("hello\n");

Solution:

  1. After 2 fork() calls: 22 = 4 processes.
  2. All 4 processes print "hello" → 4 prints from 1st printf.
  3. Then each of the 4 processes calls fork() → 8 processes total.
  4. All 8 processes print "hello" → 8 prints from 2nd printf.
  5. Total: 4 + 8 = 12 times.

Example 4: Microkernel vs Monolithic for File Read

Problem: Compare the number of mode switches for reading a file in a monolithic kernel vs. a microkernel.

Monolithic Kernel:

  1. Application (user mode) → system call read() → trap to kernel mode (1 switch)
  2. Kernel reads file, returns data
  3. Return to user mode (1 switch)
  4. Total: 2 mode switches

Microkernel:

  1. Application (user mode) → send message to kernel (1 switch to kernel)
  2. Kernel routes message to file server → switch to user mode (1 switch)
  3. File server processes request → sends message back to kernel (1 switch)
  4. Kernel routes response to application → switch to user mode (1 switch)
  5. Total: 4 mode switches (minimum)
Key Insight: Microkernels trade performance (extra mode switches and message copies) for reliability and modularity.

Example 5: System Call Parameter Passing

Problem: A system call requires 8 parameters. Which parameter passing method(s) can be used?

Answer:

  • Register method: Not feasible if fewer than 8 general-purpose registers are available for parameters. On x86-64 Linux, only 6 registers are used for syscall parameters, so this method alone is insufficient.
  • Block/Table method: Store all 8 parameters in a memory block; pass the block address in a register. This works regardless of parameter count.
  • Stack method: Push all 8 parameters onto the stack. Also works regardless of count.
  • Hybrid: Pass some parameters in registers and use a block pointer for the rest (used in practice for Linux system calls with many parameters, like clone()).

8. Additional GATE-Relevant Concepts

8.1 System Programs vs Application Programs

System programs (also called system utilities) provide a convenient environment for program development and execution. They are distinct from the kernel. Examples include: file management tools (cp, mv), compilers, assemblers, loaders, text editors, shell.

8.2 Linker and Loader

  • Compiler: Source code → object file (.o)
  • Linker: Combines object files + libraries into a single executable
  • Loader: Loads the executable into memory for execution

For dynamically linked libraries (DLLs / shared objects), the linker records references, and the dynamic linker/loader resolves them at runtime.

8.3 Why Applications Are OS-Specific

Applications compiled for one OS generally do not run on another because:

  • Different system call interfaces and APIs
  • Different executable file formats (ELF for Linux, PE for Windows, Mach-O for macOS)
  • Different ABIs (Application Binary Interfaces)
  • Different CPU instruction sets (though ISA compatibility is a separate issue)

8.4 Policy vs Mechanism

A key design principle in OS design is the separation of policy (what will be done) from mechanism (how to do it). For example:

  • Mechanism: Timer interrupt for CPU protection.
  • Policy: How long the timer should be set (time quantum for round-robin).

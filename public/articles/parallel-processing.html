Parallel Processing and Multiprocessor Systems

Computer Organization

Parallel processing is the simultaneous use of multiple processing elements to solve a problem faster than possible with a single processor. This article covers the key concepts from Stallings Chapters 17-18, including Flynn's taxonomy, pipelining, multiprocessor architectures, cache coherence, interconnection networks, and the fundamental performance laws (Amdahl's Law and Gustafson's Law).

1. Flynn's Taxonomy

Michael Flynn (1966) classified computer architectures based on the number of concurrent instruction streams and data streams. This remains the most widely used classification scheme.

1.1 The Four Categories

Category Instruction Streams Data Streams Description
SISD Single Single Conventional uniprocessor. One instruction operates on one data item at a time.
SIMD Single Multiple One instruction simultaneously operates on multiple data items. Array/vector processors.
MISD Multiple Single Multiple instructions operate on the same data stream. Rarely implemented; systolic arrays are sometimes classified here.
MIMD Multiple Multiple Multiple processors execute different instructions on different data. Most general parallel architecture.

1.2 Examples for Each Category

Category Examples
SISD Traditional von Neumann machines, early microprocessors (Intel 8086), single-core CPUs without pipelining
SIMD GPU shader cores, Intel SSE/AVX extensions, Cray-1 vector processor, array processors (ILLIAC IV), DSP processors
MISD Rare in practice. Sometimes cited: systolic arrays, redundant fault-tolerant systems (space shuttle flight computers), pipelining (debated)
MIMD Multi-core processors (Intel i7, AMD Ryzen), clusters, distributed systems, SMP servers, supercomputers
GATE Note: Pipelining is sometimes classified as MISD (multiple instructions on same data in different stages) but this is debated. The standard GATE answer treats pipelining as a technique within SISD, not a separate Flynn category. GPUs are classified as SIMD.

2. Pipelining Review

Pipelining is a technique where multiple instructions are overlapped in execution. Each stage of the pipeline performs part of the instruction, and multiple instructions are in different stages simultaneously.

2.1 Instruction Pipeline

A classic 5-stage instruction pipeline:

Stage Abbreviation Function
1IFInstruction Fetch
2IDInstruction Decode / Register Read
3EXExecute / ALU Operation
4MEMMemory Access
5WBWrite Back to Register

2.2 Pipeline Timing Diagram

Clock Cycle:    1    2    3    4    5    6    7    8    9
Instruction 1: IF   ID   EX  MEM   WB
Instruction 2:      IF   ID   EX  MEM   WB
Instruction 3:           IF   ID   EX  MEM   WB
Instruction 4:                IF   ID   EX  MEM   WB
Instruction 5:                     IF   ID   EX  MEM   WB

Without pipeline: 5 instructions x 5 cycles = 25 cycles
With pipeline: 5 + (5-1) = 9 cycles
Speedup = 25/9 = 2.78x

2.3 Pipeline Performance Formulas

Formula Description
T_pipeline = k + (n - 1) cycles k = number of stages, n = number of instructions
T_non-pipeline = n x k cycles Sequential execution time
Speedup = (n x k) / (k + n - 1) Ratio of sequential to pipelined time
Max Speedup = k (as n -> infinity) Ideal speedup equals the number of stages
Throughput = n / (k + n - 1) instructions/cycle Approaches 1 instruction/cycle for large n
Efficiency = Speedup / k = n / (k + n - 1) Utilization of pipeline stages
GATE Formula (with stalls): If there are s stall cycles total, then:
T_pipeline = k + (n - 1) + s cycles
Speedup = (n x k) / (k + n - 1 + s)
CPI for ideal pipeline = 1. With stalls: CPI = 1 + (stall cycles per instruction).

2.4 Pipeline Hazards

Hazard Type Cause Solutions
Structural Hardware resource conflict (e.g., single memory port for both IF and MEM) Duplicate resources, separate instruction/data memory (Harvard architecture), stalling
Data Instruction depends on result of a previous instruction still in the pipeline Forwarding (bypassing), stalling (pipeline bubble), reordering (compiler)
Control Branch instruction changes the flow — next instruction is unknown until branch resolves Branch prediction, delayed branching, branch target buffer, speculative execution

2.5 Data Hazard Types

Type Pattern Description
RAW (Read After Write) i: Write R1
j: Read R1		 True dependency. j needs the value written by i. Most common and problematic.
WAR (Write After Read) i: Read R1
j: Write R1		 Anti-dependency. Can occur in out-of-order execution. Solved by register renaming.
WAW (Write After Write) i: Write R1
j: Write R1		 Output dependency. Can occur in out-of-order execution. Solved by register renaming.
GATE Tip: In a standard in-order pipeline, only RAW hazards are possible. WAR and WAW hazards only arise in out-of-order (superscalar) pipelines.

2.6 Arithmetic Pipeline

Arithmetic operations like floating-point addition can be pipelined by breaking them into stages:

Stage Operation (FP Addition)
1Compare exponents
2Align mantissas (shift smaller mantissa right)
3Add/subtract mantissas
4Normalize result and handle overflow/underflow

With a 4-stage arithmetic pipeline, a new FP addition can begin every clock cycle, even though each addition takes 4 cycles to complete. Throughput = 1 result/cycle (after pipeline is full).

3. Vector Processing

A vector processor operates on entire arrays (vectors) of data with a single instruction. Instead of a loop that processes one element at a time, a vector instruction processes all elements simultaneously (or in a pipelined fashion).

3.1 Scalar vs Vector Processing

Scalar (SISD) code to add two arrays:
  for i = 0 to N-1:
    C[i] = A[i] + B[i]     // N iterations, N add instructions

Vector (SIMD) code:
  VADD C, A, B             // Single instruction adds entire vectors

3.2 Vector Processor Characteristics

  • Vector registers: Hold vectors of 64-128 elements
  • Pipelined functional units: Arithmetic units are deeply pipelined
  • Vector load/store: Streaming memory access for sequential data
  • Chaining: Output of one vector operation feeds directly to the next (like forwarding)
  • Masking: Conditional execution on individual vector elements

Vector length register (VLR): Specifies how many elements to process (for arrays shorter than the vector register length).

Vector stride: The distance between successive elements in memory. Stride = 1 means consecutive elements (best for cache). Non-unit strides occur when accessing columns of a matrix stored in row-major order.

Example: The Cray-1 (1976) had 8 vector registers, each holding 64 elements of 64 bits. Modern CPUs include vector extensions: Intel AVX-512 operates on 512-bit (8 double) vectors.

4. Multiprocessor Architectures

Multiprocessor systems have multiple CPUs that communicate and cooperate to execute programs. They are classified by how memory is organized and shared.

4.1 Shared Memory Multiprocessors

All processors share a single global address space. Any processor can access any memory location.

4.1.1 UMA (Uniform Memory Access)

  • All processors have equal access time to all memory locations
  • Also called SMP (Symmetric Multiprocessor)
  • Typically connected via a shared bus or crossbar switch
  • Scalability limited by bus/interconnect bandwidth
  • Examples: Intel multi-core desktop processors, small servers

4.1.2 NUMA (Non-Uniform Memory Access)

  • Each processor has local memory (fast access) and can access remote memory (slower)
  • Access time depends on whether the address is local or remote
  • Better scalability than UMA — each processor adds both compute and memory bandwidth
  • More complex programming — data placement matters for performance
  • Examples: AMD EPYC multi-socket servers, SGI Origin, large-scale servers

  • Each processor has its own private memory — no shared address space

  • Highly scalable — adding nodes adds both compute and memory

  • Examples: Clusters, Beowulf systems, most supercomputers (Top500)

5. Cache Coherence

In a shared-memory multiprocessor, each processor typically has a private cache. When multiple caches hold copies of the same memory location and one processor modifies it, the other caches may hold stale (outdated) data. This is the cache coherence problem.

5.1 The Problem Illustrated

Initial state: Memory location X = 5
  Cache of CPU 1: X = 5
  Cache of CPU 2: X = 5

CPU 1 writes: X = 10
  Cache of CPU 1: X = 10
  Cache of CPU 2: X = 5  (STALE!)
  Main Memory: X = 5 or 10 (depends on write policy)

If CPU 2 reads X, it gets stale value 5 instead of 10.
This violates cache coherence.

5.2 Coherence Requirements

A memory system is coherent if:

  1. Program order: A read by processor P to location X that follows a write by P to X always returns the value written by P (if no other processor writes X in between).
  2. Coherent view: A read by any processor to X eventually returns the value of the last write to X by any processor.
  3. Write serialization: All writes to the same location are seen in the same order by all processors.

5.3 Snooping Protocols

Snooping (bus-based) protocols require that all caches monitor (snoop on) the shared bus. When one cache performs a write, all other caches observe the bus transaction and take appropriate action.

Write-Invalidate Protocol

  • When a processor writes to a shared block, it broadcasts an invalidation signal on the bus
  • All other caches that hold a copy of that block invalidate their copy
  • Subsequent reads by other processors will miss and fetch the updated value
  • Most common approach (used in MESI protocol)

Write-Update (Write-Broadcast) Protocol

  • When a processor writes to a shared block, the new value is broadcast to all caches
  • All copies are updated simultaneously
  • Higher bus traffic (broadcasts every write) but lower read latency (copies always up to date)
  • Less common in practice — Dragon protocol is an example

5.4 MESI Protocol

The MESI protocol is the most widely used snooping-based cache coherence protocol. Each cache line is in one of four states:

State Letter Meaning
Modified (M) M Only copy, and it has been modified. Different from main memory. Must write back before eviction.
Exclusive (E) E Only copy, but it is clean (same as main memory). Can be written without bus transaction.
Shared (S) S Multiple copies may exist in different caches. All are clean. Must invalidate others before writing.
Invalid (I) I Cache line is not valid. Equivalent to not being in the cache.

GATE Key Point: The Exclusive state is an optimization over the basic MSI protocol. It allows a processor to write to a cache line without generating bus traffic if it is the only owner. Without the E state, even a private write to a clean line would require a bus invalidation.

5.6 Directory-Based Coherence

For systems with many processors (where bus snooping does not scale), directory-based protocols are used. A central directory maintains the sharing status of each memory block.

Aspect Snooping Directory-Based
Communication Broadcast on shared bus Point-to-point messages
Scalability Poor (bus bandwidth limit) Good (no broadcast)
Complexity Simpler More complex (directory storage)
Latency Low (single bus transaction) Higher (directory lookup + messages)
Processors Typically ≤ 16-32 Hundreds to thousands
Used in SMP, multi-core desktop/server NUMA, large-scale multiprocessors

The directory stores for each block: which caches have a copy (bit vector or limited pointer set) and the state (uncached, shared, exclusive/modified).

6.3 Multistage Interconnection Networks (MIN)

MINs provide a middle ground between bus (cheap, low bandwidth) and crossbar (expensive, high bandwidth). They use multiple stages of smaller switches.

Omega Network (Example)

An N-input Omega network uses log2 N stages of N/2 switches each (2×2 crossbar switches). Routing is based on the destination address bits. 

8-input Omega Network:
  Stages: log2(8) = 3 stages
  Switches per stage: 8/2 = 4 switches (2x2 each)
  Total switches: 3 x 4 = 12
  Compare: crossbar needs 8 x 8 = 64 crosspoints

Routing algorithm: For destination address dn-1 dn-2 ... d0, at stage i, if bit di = 0, take the upper output; if di = 1, take the lower output.

Number of stages = log2 N
Switches per stage = N/2
Total 2×2 switches = (N/2) × log2 N
Total crosspoints = 4 × (N/2) × log2 N = 2N log2 N

7. Amdahl's Law

Amdahl's Law (Gene Amdahl, 1967) gives the theoretical maximum speedup when parallelizing a program, given that a fraction of the program must remain sequential.

7.1 The Formula

Speedup(N) =          1
              ─────────────────
              (1 - f) + f / N

Where:
  f = fraction of the program that can be parallelized (0 ≤ f ≤ 1)
  (1 - f) = fraction that must remain sequential
  N = number of processors

7.2 Maximum Speedup (N -> infinity)

As N → ∞:  f/N → 0

Speedup_max =    1
              ───────
              (1 - f)

Example: If 95% of a program is parallelizable (f = 0.95):
  Speedup_max = 1 / (1 - 0.95) = 1 / 0.05 = 20x

No matter how many processors you add, you cannot exceed 20x speedup!

7.3 Worked Examples

Example 1: A program has 80% parallelizable code. What is the speedup with 4 processors?

Speedup(4) = 1 / ((1 - 0.8) + 0.8/4)
           = 1 / (0.2 + 0.2)
           = 1 / 0.4
           = 2.5x

Example 2: Same program with 16 processors:

Speedup(16) = 1 / ((1 - 0.8) + 0.8/16)
            = 1 / (0.2 + 0.05)
            = 1 / 0.25
            = 4x

Example 3: Maximum speedup:

Speedup_max = 1 / (1 - 0.8) = 1 / 0.2 = 5x

Even with infinite processors, 80% parallelizable code gives at most 5x speedup.

GATE Insight: Amdahl's Law shows diminishing returns. Doubling processors does NOT double speedup. The sequential fraction (1 - f) becomes the dominant bottleneck as N increases. This is why optimizing the sequential portion is critical.

8. Gustafson's Law

Gustafson's Law (John Gustafson, 1988) provides a more optimistic view of parallel speedup by changing the assumption: instead of fixing the problem size, we fix the execution time and scale the problem size with the number of processors.

8.1 The Formula

Scaled Speedup(N) = N - (1 - f) x (N - 1)
                  = N x f + (1 - f)
                  = 1 + (N - 1) x f

Where:
  f = fraction of parallel execution time (on N processors)
  N = number of processors

8.2 Comparison: Amdahl vs Gustafson

Aspect Amdahl's Law Gustafson's Law
Assumption Fixed problem size Fixed execution time, scaled problem size
Perspective Pessimistic (diminishing returns) Optimistic (scalable speedup)
Speedup limit 1 / (1 - f) — bounded Grows linearly with N
Use case Real-time constraints (fixed deadline) Scientific computing (larger problems)
Formula 1 / ((1-f) + f/N) 1 + (N-1) x f

8.3 Worked Example

Given: f = 0.9 (90% parallel), N = 64 processors

Amdahl's Law:
  Speedup = 1 / (0.1 + 0.9/64) = 1 / (0.1 + 0.014) = 1 / 0.114 = 8.77x

Gustafson's Law:
  Speedup = 1 + (64 - 1) x 0.9 = 1 + 56.7 = 57.7x

Gustafson is much more optimistic because it assumes the problem
size has been scaled up to keep all 64 processors busy.
Key Insight: Gustafson's Law argues that in practice, when we have more processors, we solve bigger problems (not the same problem faster). The parallel portion scales with the problem size, so the sequential fraction becomes proportionally smaller. This is commonly observed in scientific simulations, weather modeling, and big data processing.

9. Symmetric Multiprocessors (SMP)

An SMP (Symmetric Multiprocessor) is a shared-memory multiprocessor where all processors are identical and have equal access to all resources (memory, I/O).

9.1 SMP Characteristics

  • Two or more identical processors with similar capabilities
  • Processors share main memory and I/O facilities through a common bus or interconnect
  • All processors can perform the same functions (symmetry)
  • Single copy of the OS manages all processors
  • Processors share memory with uniform access time (UMA model)

9.3 SMP Advantages and Challenges

Advantages Challenges
Performance: multiple processors share the workload Bus contention: bandwidth bottleneck with many processors
Availability: if one processor fails, others continue Cache coherence: must ensure data consistency across caches
Incremental growth: add processors to scale OS complexity: scheduler must balance load across processors
Scaling: well-understood programming model (shared memory, threads) Synchronization overhead: locks, barriers, etc. limit parallelism

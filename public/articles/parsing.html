1. Role of the Parser

The parser receives a stream of tokens from the lexical analyzer and verifies that the token sequence can be generated by the grammar. It produces a parse tree (or abstract syntax tree) as output.

  • Reports syntax errors with meaningful messages.
  • Recovers from common errors to continue parsing.
  • Guides semantic actions (syntax-directed translation).

3. Top-Down Parsing

Top-down parsers build the parse tree from the root (start symbol) to the leaves (terminals). They attempt to find a leftmost derivation for the input.

3.1 Recursive Descent Parsing

A set of mutually recursive procedures, one for each non-terminal. Each procedure attempts to match the input against the productions for its non-terminal.

Grammar: E -> T E'
         E' -> + T E' | epsilon
         T -> F T'
         T' -> * F T' | epsilon
         F -> ( E ) | id

Procedure for E':
  void E_prime() {
    if (lookahead == '+') {
      match('+');
      T();
      E_prime();
    }
    // else: epsilon production (do nothing)
  }

3.2 LL(1) Parsing

A table-driven top-down parser. "LL(1)" means: scan input Left to right, produce Leftmost derivation, with 1 lookahead symbol.

FIRST and FOLLOW Sets

SetDefinition
FIRST(alpha)Set of terminals that can begin strings derived from alpha. If alpha can derive epsilon, then epsilon is in FIRST(alpha).
FOLLOW(A)Set of terminals that can appear immediately after A in some derivation. $ (end marker) is in FOLLOW(S) for start symbol S.

Computing FIRST

Rules for FIRST(X):
1. If X is a terminal: FIRST(X) = {X}
2. If X -> epsilon: add epsilon to FIRST(X)
3. If X -> Y1 Y2 ... Yk:
   Add FIRST(Y1) - {epsilon} to FIRST(X)
   If epsilon in FIRST(Y1), add FIRST(Y2) - {epsilon}
   Continue until Yi where epsilon not in FIRST(Yi)
   If epsilon in FIRST of all Yi, add epsilon to FIRST(X)

Computing FOLLOW

Rules for FOLLOW(A):
1. Add $ to FOLLOW(S) where S is start symbol
2. If B -> alpha A beta:
   Add FIRST(beta) - {epsilon} to FOLLOW(A)
3. If B -> alpha A, or B -> alpha A beta where epsilon in FIRST(beta):
   Add FOLLOW(B) to FOLLOW(A)

LL(1) Parse Table Construction

For each production A -> alpha:

  • For each terminal a in FIRST(alpha): add A -> alpha to M[A, a].
  • If epsilon in FIRST(alpha): for each terminal b in FOLLOW(A), add A -> alpha to M[A, b].
  • If epsilon in FIRST(alpha) and $ in FOLLOW(A): add A -> alpha to M[A, $].
LL(1) Conflicts: If any cell in the parse table has more than one entry, the grammar is not LL(1). Common causes: left recursion, left factoring needed, or inherent ambiguity.

Eliminating Left Recursion

Left-recursive: A -> A alpha | beta
Convert to:     A -> beta A'
                A' -> alpha A' | epsilon

Left Factoring

Before: A -> alpha beta1 | alpha beta2
After:  A -> alpha A'
        A' -> beta1 | beta2

4. Bottom-Up Parsing

Bottom-up parsers build the parse tree from the leaves to the root. They find a rightmost derivation in reverse. The key operation is identifying a handle (the substring to reduce).

4.1 Shift-Reduce Parsing

Uses a stack and four operations:

OperationDescription
ShiftPush next input symbol onto stack
ReduceReplace a handle on top of stack with a non-terminal
AcceptParsing complete, input accepted
ErrorSyntax error detected

4.2 LR Parsing Family

"LR(k)" means: scan input Left to right, produce Rightmost derivation in reverse, with k lookahead symbols.

LR(0) Items

An LR(0) item is a production with a dot indicating how much has been seen so far.

Production: A -> X Y Z
LR(0) items:
  A -> . X Y Z    (nothing seen yet)
  A -> X . Y Z    (X has been seen)
  A -> X Y . Z    (X and Y seen)
  A -> X Y Z .    (complete item, ready to reduce)

LR(0) Parser

Constructs a DFA of item sets (canonical LR(0) collection). Conflicts arise when a state has both shift and reduce actions, or multiple reduce actions.

SLR(1) Parser

Uses LR(0) items but resolves conflicts using FOLLOW sets. Reduce A -> alpha only when lookahead is in FOLLOW(A). More powerful than LR(0).

CLR(1) / Canonical LR(1) Parser

Uses LR(1) items: [A -> alpha . beta, a] where a is a lookahead terminal. Most powerful LR parser but may have very large parse tables.

LALR(1) Parser

Merges LR(1) states with the same core (LR(0) items) but different lookaheads. Fewer states than CLR(1) with almost the same power. Used by YACC/Bison.

6. Error Recovery Strategies

StrategyDescription
Panic ModeDiscard input tokens until a synchronizing token (e.g., semicolon) is found. Simple and never enters infinite loop.
Phrase-LevelReplace prefix of remaining input with a string that allows parsing to continue. E.g., insert missing semicolon.
Error ProductionsAdd grammar productions that capture common errors. Parser handles errors as valid constructs and reports them.
Global CorrectionFind the minimum edit distance to a valid program. Theoretically elegant but too costly in practice.

7. Key GATE Takeaways

  • Top-down parsers (LL) build tree top-to-bottom; bottom-up parsers (LR) build bottom-to-top.
  • LL(1) requires: no left recursion, left-factored grammar, and disjoint FIRST/FOLLOW.
  • FIRST and FOLLOW set computation is fundamental and appears frequently in GATE.
  • SLR(1) uses FOLLOW sets to resolve conflicts; CLR(1) uses exact lookaheads.
  • LALR(1) = CLR(1) states merged by core. Same language power as CLR(1) but may introduce reduce-reduce conflicts.
  • Parser power hierarchy: LL(1) is a subset of SLR(1) is a subset of LALR(1) is a subset of CLR(1).
  • Ambiguous grammars cannot be LR (any variant) or LL.

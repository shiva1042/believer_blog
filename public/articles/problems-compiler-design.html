10 GATE-level compiler design problems covering FIRST/FOLLOW sets, LL(1) parsing tables, LR(0) items, operator precedence, three-address code, live variable analysis, common subexpression elimination, and syntax-directed translation.

Problem 1: FIRST and FOLLOW Sets

Problem: Compute FIRST and FOLLOW sets for the grammar:
E → TE'
E' → +TE' | ε
T → FT'
T' → *FT' | ε
F → (E) | id
Approach: FIRST: for each non-terminal, find all terminals that can begin a string derived from it. FOLLOW: find terminals that can appear immediately after a non-terminal.
Step-by-step Solution:
FIRST sets:
FIRST(F) = FIRST((E)) ∪ FIRST(id) = { (, id }
FIRST(T') = { *, ε }
FIRST(T) = FIRST(F) = { (, id }
FIRST(E') = { +, ε }
FIRST(E) = FIRST(T) = { (, id }

Non-terminalFIRST
E{ (, id }
E'{ +, ε }
T{ (, id }
T'{ *, ε }
F{ (, id }
FOLLOW sets:
FOLLOW(E): E is start symbol, so $ ∈ FOLLOW(E). Also E appears in (E), so ) ∈ FOLLOW(E).
FOLLOW(E) = { ), $ }

FOLLOW(E'): E' is at the end of E → TE', so FOLLOW(E') = FOLLOW(E) = { ), $ }.
FOLLOW(E') = { ), $ }

FOLLOW(T): T is followed by E' in E → TE'. FIRST(E') = {+, ε}. So + ∈ FOLLOW(T). Since ε ∈ FIRST(E'), FOLLOW(E) ⊂ FOLLOW(T). Also T appears in E' → +TE', same analysis.
FOLLOW(T) = { +, ), $ }

FOLLOW(T'): T' is at end of T → FT'. FOLLOW(T') = FOLLOW(T).
FOLLOW(T') = { +, ), $ }

FOLLOW(F): F is followed by T' in T → FT'. FIRST(T') = {*, ε}. So * ∈ FOLLOW(F). Since ε ∈ FIRST(T'), FOLLOW(T) ⊂ FOLLOW(F).
FOLLOW(F) = { *, +, ), $ }
GATE Tip: FIRST rules: (1) terminal a: FIRST(a)={a}. (2) A→ε: ε ∈ FIRST(A). (3) A→Bα: FIRST(B)-{ε} ⊂ FIRST(A); if ε ∈ FIRST(B), continue with α. FOLLOW rules: (1) Start symbol has $. (2) A→αBβ: FIRST(β)-{ε} ⊂ FOLLOW(B). (3) If ε ∈ FIRST(β) or β is empty: FOLLOW(A) ⊂ FOLLOW(B).

Problem 2: LL(1) Parsing Table

Problem: Using the FIRST and FOLLOW sets from Problem 1, construct the LL(1) parsing table for the grammar. Verify that the grammar is LL(1).
Approach: For each production A → α: for each terminal a in FIRST(α), add A → α to M[A, a]. If ε ∈ FIRST(α), add A → α to M[A, b] for each b in FOLLOW(A).
Step-by-step Solution:
id+*()$
EE→TE'E→TE'
E'E'→+TE'E'→εE'→ε
TT→FT'T→FT'
T'T'→εT'→*FT'T'→εT'→ε
FF→idF→(E)
No cell has multiple entries. The grammar IS LL(1).

Example parse of "id + id * id":

$E'Tid+id*id$T→FT'
$E'T'Fid+id*id$F→id
$E'T'idid+id*id$match id
$E'T'+id*id$T'→ε
$E'+id*id$E'→+TE'
$E'T++id*id$match +
$E'Tid*id$T→FT'
$E'T'Fid*id$F→id
$E'T'idid*id$match id
$E'T'*id$T'→*FT'
$E'T'F**id$match *
$E'T'Fid$F→id
$E'T'idid$match id
$E'T'$T'→ε
$E'$E'→ε
$$Accept
Key Takeaway: A grammar is LL(1) iff the parsing table has at most one entry per cell. Left recursion and common prefixes make a grammar non-LL(1). Eliminate left recursion and left-factor before building the table. This expression grammar is the standard LL(1) example.

Problem 3: LR(0) Items and DFA

Problem: Construct LR(0) item sets for the grammar: S → AA, A → aA | b. Determine if the grammar is SLR(1).
Approach: Add augmented production S' → S. Compute closure and goto for each item set. Check for shift-reduce or reduce-reduce conflicts in SLR.
Step-by-step Solution:
Augmented grammar: S' → S, S → AA, A → aA, A → b.

I0 = closure({S' → .S}):
S' → .S
S → .AA
A → .aA
A → .b

I1 = goto(I0, S): S' → S.
I2 = goto(I0, A): S → A.A, A → .aA, A → .b
I3 = goto(I0, a): A → a.A, A → .aA, A → .b
I4 = goto(I0, b): A → b.
I5 = goto(I2, A): S → AA.
I6 = goto(I2, a): same as I3 (A → a.A, A → .aA, A → .b)
I7 = goto(I2, b): same as I4 (A → b.)
I8 = goto(I3, A): A → aA.
I9 = goto(I3, a): same as I3
I10 = goto(I3, b): same as I4

So the distinct states are I0-I5, I8 (with I6=I3, I7=I4, I9=I3, I10=I4).
States: I0, I1, I2, I3, I4, I5, I8. Total = 7 states.

SLR check:
FOLLOW(S') = {$}, FOLLOW(S) = {$}, FOLLOW(A) = {a, b, $}
I1: S' → S. Reduce on $. No shift. No conflict.
I4: A → b. Reduce on FOLLOW(A) = {a, b, $}. No shift. No conflict.
I5: S → AA. Reduce on FOLLOW(S) = {$}. No shift. No conflict.
I8: A → aA. Reduce on FOLLOW(A) = {a, b, $}. No shift. No conflict.

No conflicts. The grammar IS SLR(1).
GATE Tip: SLR uses FOLLOW sets for reduce actions. CLR uses exact lookahead (more precise). LALR merges CLR states with same core. Power: LR(0) ⊂ SLR(1) ⊂ LALR(1) ⊂ CLR(1). GATE most often asks for SLR parsing table construction.

Problem 4: Operator Precedence Parsing

Problem: Given operators with precedence (* > + > id) and the operator precedence table below, parse the expression "id + id * id" using shift-reduce with operator precedence relations.
id+*$
id>>>
+<><>
*<>>>
$<<<
Approach: Compare top stack terminal with input. <: shift. >: reduce (find handle between last < and current >). =: shift (for matching brackets).
Step-by-step Solution:
StackRelationInputAction
$<id+id*id$Shift id
$id>+id*id$Reduce id to E
$E<+id*id$Shift +
$E+<id*id$Shift id
$E+id>*id$Reduce id to E
$E+E<*id$Shift * (+ < *)
$E+E*<id$Shift id
$E+E*id>$Reduce id to E
$E+E*E>$Reduce E*E to E
$E+E>$Reduce E+E to E
$E$Accept
Parse successful. * binds tighter than + as expected.
Key Takeaway: Operator precedence parsing works only for operator grammars (no two adjacent non-terminals, no ε productions). The precedence relations encode associativity and precedence: higher precedence operator gets < before and > after. Left-associative: same level gets >.

Problem 5: Three-Address Code Generation

Problem: Generate three-address code for: a = b + c * d - e / f + g. Also show the quadruple and triple representations.
Approach: Respect operator precedence: * and / first, then + and - left to right. Generate temporaries for each sub-expression.
Step-by-step Solution:
Expression tree (respecting precedence):
a = ((b + (c * d)) - (e / f)) + g

Three-address code: 
t1 = c * d
t2 = b + t1
t3 = e / f
t4 = t2 - t3
t5 = t4 + g
a = t5
Quadruple representation:
#OpArg1Arg2Result
0*cdt1
1+bt1t2
2/eft3
3-t2t3t4
4+t4gt5
5=t5a
Triple representation:
#OpArg1Arg2
0*cd
1+b(0)
2/ef
3-(1)(2)
4+(3)g
5=a(4)
(Numbers in parentheses refer to the result of that triple.)
GATE Tip: Quadruples use explicit temporary names (easy to reorder). Triples use position references (no extra names but reordering requires updating references). Indirect triples add an indirection table to allow reordering while keeping triple format.

Problem 6: Live Variable Analysis

Problem: For the following basic block, compute the live variables at each program point: 
1: a = b + c
2: d = a * e
3: f = d + b
4: a = f - c
5: g = a + d
Assume g is live at exit.
Approach: Work backwards. A variable is live if it is used before being redefined. At each point: LIVE_before = (LIVE_after - DEF) ∪ USE.
Step-by-step Solution:
Work from bottom to top.

After line 5: LIVE = {g} (g is live at exit)
Line 5: g = a + d DEF={g}, USE={a,d}.
Before 5: ({g} - {g}) ∪ {a,d} = {a, d}

Line 4: a = f - c DEF={a}, USE={f,c}.
Before 4: ({a,d} - {a}) ∪ {f,c} = {d, f, c}

Line 3: f = d + b DEF={f}, USE={d,b}.
Before 3: ({d,f,c} - {f}) ∪ {d,b} = {d, c, b}

Line 2: d = a * e DEF={d}, USE={a,e}.
Before 2: ({d,c,b} - {d}) ∪ {a,e} = {c, b, a, e}

Line 1: a = b + c DEF={a}, USE={b,c}.
Before 1: ({c,b,a,e} - {a}) ∪ {b,c} = {b, c, e}

PointLive Variables
Entry{b, c, e}
After line 1{a, b, c, e}
After line 2{b, c, d}
After line 3{c, d, f}
After line 4{a, d}
After line 5 (exit){g}
Key Takeaway: Live variable analysis is a backward data flow analysis. It tells us which variables need to be kept in registers. A variable that is defined but not live afterward is a dead assignment and can be eliminated. This is crucial for register allocation and dead code elimination.

Problem 7: Common Subexpression Elimination

Problem: Optimize the following code using common subexpression elimination (CSE): 
t1 = a + b
t2 = c * d
t3 = a + b
t4 = t2 + t3
t5 = c * d
t6 = t1 + t5
Approach: Identify expressions computed more than once with the same operand values. Replace duplicates with the previously computed result.
Step-by-step Solution:
Identify common subexpressions:
- t3 = a + b: same as t1 (a and b not modified between lines 1 and 3)
- t5 = c * d: same as t2 (c and d not modified between lines 2 and 5)

Optimized code: 
t1 = a + b
t2 = c * d
t3 = t1         (reuse t1 instead of recomputing a+b)
t4 = t2 + t3
t5 = t2         (reuse t2 instead of recomputing c*d)
t6 = t1 + t5
Further simplification (copy propagation): 
t1 = a + b
t2 = c * d
t4 = t2 + t1    (substitute t3=t1)
t6 = t1 + t2    (substitute t5=t2)
Eliminated 2 additions/multiplications. If t3 and t5 are dead, their assignments are also removed.
GATE Tip: CSE requires that operands are not modified between the original and duplicate computation. Use a DAG (Directed Acyclic Graph) to systematically detect common subexpressions. In GATE, DAG-based optimization questions are common.

Problem 8: Syntax-Directed Translation

Problem: Given the SDT for a desk calculator:
E → E1 + T { E.val = E1.val + T.val }
E → T { E.val = T.val }
T → T1 * F { T.val = T1.val * F.val }
T → F { T.val = F.val }
F → (E) { F.val = E.val }
F → num { F.val = num.lexval }
Evaluate "3 + 4 * 2" using this SDT.
Approach: Build parse tree bottom-up, evaluate attributes (synthesized) at each node.
Step-by-step Solution:
Parse tree for "3 + 4 * 2":
E → E + T → T + T → F + T → 3 + T
T → T * F → F * F → 4 * F → 4 * 2

Bottom-up evaluation:
1. F.val = 3 (from num.lexval)
2. T.val = F.val = 3 (T → F)
3. E1.val = T.val = 3 (E → T)
4. F.val = 4 (from num.lexval)
5. T1.val = F.val = 4 (T → F)
6. F.val = 2 (from num.lexval)
7. T.val = T1.val * F.val = 4 * 2 = 8 (T → T*F)
8. E.val = E1.val + T.val = 3 + 8 = 11 (E → E+T)

Result: E.val = 11
GATE Tip: S-attributed SDTs (only synthesized attributes) can be evaluated bottom-up during LR parsing. L-attributed SDTs (inherited attributes flow left-to-right) can be evaluated during LL parsing. GATE often asks you to evaluate an SDT for a given input string.

Problem 9: SLR Parsing Table and Parse

Problem: Construct the SLR(1) parsing table for: S → CC, C → cC | d. Then parse the input "cdd".
Approach: Build LR(0) item sets (from Problem 3 pattern), compute FOLLOW for reduce actions, fill the ACTION and GOTO table.
Step-by-step Solution:
Using the item sets from Problem 3 (same grammar pattern with a→c, b→d):
I0: S'→.S, S→.CC, C→.cC, C→.d
I1: S'→S.
I2: S→C.C, C→.cC, C→.d
I3: C→c.C, C→.cC, C→.d
I4: C→d.
I5: S→CC.
I8: C→cC.

FOLLOW(S) = {$}, FOLLOW(C) = {c, d, $}

SLR Parsing Table:

3s3s48

(r1: S→CC, r2: C→cC, r3: C→d)

Parse "cdd":

Key Takeaway: In SLR parsing: shift puts state on stack, reduce pops |RHS| states, then push goto[new_top, LHS]. Reduce actions go in FOLLOW columns. This is the most commonly tested parsing technique in GATE.

Problem 10: DAG for Basic Block Optimization

Problem: Construct a DAG for the following basic block and identify the optimized code: 
a = b + c
d = a - e
f = b + c
g = d * f
Approach: Build DAG node by node. Before creating a new node, check if an identical expression exists. Reuse existing nodes.
Step-by-step Solution:
Line 1: a = b + c
Create leaf nodes for b and c. Create + node. Label it 'a'.

Line 2: d = a - e
Node 'a' exists (the + node). Create leaf for e. Create - node. Label it 'd'.

Line 3: f = b + c
Check: b + c already exists (line 1's node). Common subexpression!
Add label 'f' to the existing + node. f and a point to the same node.

Line 4: g = d * f
d and f both exist as nodes. Create * node. Label it 'g'.

DAG has 6 nodes: b, c, e, (b+c)[a,f], (a-e)[d], (d*f)[g]

Optimized code from DAG: 
a = b + c
d = a - e
f = a          (copy, or simply use 'a' wherever 'f' was used)
g = d * a      (since f = a)
Or with copy propagation: 
a = b + c
d = a - e
g = d * a
Saved one addition operation.
GATE Tip: DAG construction: (1) check if node for variable exists, (2) check if identical expression node exists. Labels on nodes indicate all variables holding that value. DAGs identify common subexpressions, dead code, and enable algebraic simplifications. GATE may ask you to draw the DAG and count the nodes.

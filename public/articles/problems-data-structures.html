Problem 1: Array Rotation

Problem: Given an array A = [1,2,3,4,5,6,7], rotate it left by 3 positions to get [4,5,6,7,1,2,3] using the reversal algorithm. Show each step. What is the time complexity?
Approach: Reversal algorithm: (1) Reverse first d elements, (2) Reverse remaining n-d elements, (3) Reverse entire array. O(n) time, O(1) space.
Step-by-step Solution:
Original: [1, 2, 3, 4, 5, 6, 7], d = 3

Step 1: Reverse A[0..2]: [3, 2, 1, 4, 5, 6, 7]
Step 2: Reverse A[3..6]: [3, 2, 1, 7, 6, 5, 4]
Step 3: Reverse A[0..6]: [4, 5, 6, 7, 1, 2, 3]

Time: O(n), Space: O(1)
Key Takeaway: The reversal algorithm is the optimal in-place rotation method. An alternative is the juggling algorithm using GCD(n,d) sets. Both are O(n) time and O(1) space.

Problem 2: Stack-Based Expression Evaluation

Problem: Evaluate the postfix expression: 6 3 2 * + 5 - 4 /. Show the stack state after each token.
Approach: Scan left to right. Push operands. On operator, pop two operands, apply operator, push result.
Step-by-step Solution:

GATE Tip: For subtraction and division in postfix, the first popped element is the second operand: (second popped) OP (first popped). The equivalent infix is ((6+(3*2))-5)/4 = 7/4.

Problem 4: Queue Simulation — Circular Queue

Problem: A circular queue of size 5 (indices 0-4) has front=0, rear=-1 (empty). Perform: Enqueue(10), Enqueue(20), Enqueue(30), Dequeue, Dequeue, Enqueue(40), Enqueue(50), Enqueue(60), Enqueue(70). Show state after each operation.
Approach: rear = (rear+1) % SIZE for enqueue. front = (front+1) % SIZE for dequeue. Full condition: (rear+1)%SIZE == front.
Step-by-step Solution:

GATE Tip: A circular queue of size n can hold at most n-1 elements (one slot wasted) unless a separate count variable is maintained. Always check: is the implementation using a count or a wasted slot?

Problem 5: Linked List — Reverse a Singly Linked List

Problem: Given the linked list 1 → 2 → 3 → 4 → NULL, reverse it iteratively. Show pointer states at each step.
Approach: Use three pointers: prev, curr, next. Traverse and reverse links one by one.
Step-by-step Solution:
Initial: prev=NULL, curr=1, next=NULL

Stepprevcurrnext (saved)Action
1NULL121.next = NULL
21232.next = 1
32343.next = 2
434NULL4.next = 3
curr becomes NULL, loop ends. New head = prev = 4.
Result: 4 → 3 → 2 → 1 → NULL

Node* reverse(Node* head) {
    Node *prev = NULL, *curr = head, *next;
    while (curr != NULL) {
        next = curr->next;
        curr->next = prev;
        prev = curr;
        curr = next;
    }
    return prev;
}
Key Takeaway: Iterative reversal is O(n) time, O(1) space. Recursive reversal uses O(n) stack space. In GATE, questions often ask about pointer manipulation after k steps of reversal.

Problem 8: BST Insertion and Deletion

Problem: Insert the sequence [50, 30, 70, 20, 40, 60, 80] into an empty BST. Then delete 50 (root). Show the tree after each major operation.
Approach: Insert by BST property. Delete node with two children: replace with inorder successor (smallest in right subtree) or inorder predecessor.
Step-by-step Solution:
After all insertions:
graph TD 50((50)) --> 30((30)) 50 --> 70((70)) 30 --> 20((20)) 30 --> 40((40)) 70 --> 60((60)) 70 --> 80((80))
Delete 50 (has two children):
Inorder successor of 50 = leftmost node in right subtree = 60.
Replace 50 with 60, then delete original 60 (leaf).
After deletion:
graph TD 60((60)) --> 30((30)) 60 --> 70((70)) 30 --> 20((20)) 30 --> 40((40)) 70 --> 80((80))
GATE Tip: BST deletion cases: (1) Leaf: just remove. (2) One child: replace with child. (3) Two children: replace with inorder successor or predecessor. The tree structure differs depending on which you choose. Inorder successor is more common in practice.

Problem 9: BST — Count Nodes in Range

Problem: Given a BST with nodes [5, 10, 15, 20, 25, 30, 35], how many nodes fall in the range [12, 28]? What is the optimal time complexity?
Approach: Traverse BST, pruning branches that cannot contain values in range. If node < low, go right only. If node > high, go left only. Otherwise, count and recurse both.
Step-by-step Solution:
BST (balanced):
graph TD 20((20)) --> 10((10)) 20 --> 30((30)) 10 --> 5((5)) 10 --> 15((15)) 30 --> 25((25)) 30 --> 35((35))
Range [12, 28]:
- Visit 20: 12 ≤ 20 ≤ 28. Count it. Recurse both.
- Visit 10: 10 < 12. Go right only.
- Visit 15: 12 ≤ 15 ≤ 28. Count it. Left child=NULL, right=NULL. Done.
- Visit 30: 30 > 28. Go left only.
- Visit 25: 12 ≤ 25 ≤ 28. Count it. Left=NULL, right=NULL. Done.

Nodes in range: {15, 20, 25}. Count = 3
Nodes visited: 5 (out of 7). Pruned subtrees rooted at 5 and 35.
Time: O(h + k) where h = height, k = number of nodes in range.
Key Takeaway: Range queries on BST are efficient because of pruning. For a balanced BST, this gives O(log n + k) where k is the output size. This is optimal since you must visit all k nodes.

Problem 10: Min-Heap Operations

Problem: Build a min-heap from the array [9, 6, 5, 2, 8, 3, 1]. Then perform two extract-min operations. Show the heap after each step.
Approach: Build heap using bottom-up heapify (O(n)). Extract-min: replace root with last element, heapify down.
Step-by-step Solution:
Build heap (bottom-up):
Initial array: [9, 6, 5, 2, 8, 3, 1]

Heapify index 2 (5): children are 3, 1. Min child = 1. Swap 5 and 1.
Array: [9, 6, 1, 2, 8, 3, 5]

Heapify index 1 (6): children are 2, 8. Min child = 2. Swap 6 and 2.
Array: [9, 2, 1, 6, 8, 3, 5]

Heapify index 0 (9): children are 2, 1. Min child = 1. Swap 9 and 1.
Array: [1, 2, 9, 6, 8, 3, 5]
Heapify index 2 (9): children are 3, 5. Min child = 3. Swap 9 and 3.
Array: [1, 2, 3, 6, 8, 9, 5]
Wait -- recheck: index 2 children are at 5 and 6, values 9 and 5. Min is 5. Swap 9 and 5.
Array: [1, 2, 5, 6, 8, 3, 9]... Let me redo carefully.

Actually, let me redo the build with correct child indices.
For index i: left=2i+1, right=2i+2.
Array (0-indexed): [9, 6, 5, 2, 8, 3, 1]

Last non-leaf = (7/2)-1 = 2.
i=2 (val=5): left=5(val=3), right=6(val=1). Min=1 at index 6. Swap. [9,6,1,2,8,3,5]
i=1 (val=6): left=3(val=2), right=4(val=8). Min=2 at index 3. Swap. [9,2,1,6,8,3,5]
i=0 (val=9): left=1(val=2), right=2(val=1). Min=1 at index 2. Swap. [1,2,9,6,8,3,5]
Heapify 9 at index 2: left=5(val=3), right=6(val=5). Min=3 at index 5. Swap. [1,2,3,6,8,9,5]

Final heap: [1, 2, 3, 6, 8, 9, 5]

Extract-Min #1: Remove 1. Replace root with last (5). Array: [5, 2, 3, 6, 8, 9]
Heapify: 5 vs children 2,3. Min=2. Swap. [2, 5, 3, 6, 8, 9]
Heapify 5 at index 1: children 6,8. 5 < both. Stop.

GATE Tip: Build heap is O(n) not O(n log n). Extract-min is O(log n). A common GATE question: "After building heap from array X and performing k extract-min, what is the resulting array?" Always show the array representation, not just the tree.

Problem 11: AVL Tree Rotations

Problem: Insert keys 30, 20, 10, 25, 28 into an empty AVL tree. Show each rotation performed with the type (LL, RR, LR, RL).
Approach: Insert as in BST, then check balance factors. If |BF| > 1, perform appropriate rotation.
Step-by-step Solution:
Insert 30: Tree = [30]. Balanced.
Insert 20: 20 < 30, goes left. BF(30)=1. Balanced.
Insert 10: 10 < 20, goes left of 20. BF(30)=2 (left-heavy). BF(20)=1 (left-heavy).
This is an LL case. Right rotation on 30.
graph TD subgraph After LL Rotation B20((20)) --> B10((10)) B20 --> B30((30)) end
Insert 25: 25 > 20, right to 30, left of 30. Now 25 is left child of 30.
BF(20) = -1, all balanced. Tree: 20(10, 30(25, _)).

Insert 28: Goes: 20→right→30→left→25→right→28.
BF(25)=-1, BF(30)=2 (left-heavy, since left subtree height=2, right=0). BF(25)=-1 (right-heavy).
This is an LR case at node 30. Left rotation on 25, then right rotation on 30.

Step 1: Left rotate at 25: 28 becomes parent of 25.
Step 2: Right rotate at 30: 28 becomes parent of 30.
graph TD subgraph Final AVL Tree C20((20)) --> C10((10)) C20 --> C28((28)) C28 --> C25((25)) C28 --> C30((30)) end
Key Takeaway: AVL rotation rules: LL→Right rotate, RR→Left rotate, LR→Left then Right, RL→Right then Left. The rotation type is determined by the path from the unbalanced node to the newly inserted node.

Approach: h(k,i) = (k mod 11 + i) mod 11. On collision, probe next slot linearly.

Problem 14: Priority Queue using Heap

Problem: A min-priority queue contains: [3, 7, 5, 12, 9, 8, 6]. Perform: decrease-key(index 3, new value 1), then extract-min. Show the heap after each operation.
Approach: Decrease-key: update value, then bubble up. Extract-min: swap root with last, remove last, heapify down.
Step-by-step Solution:
Heap array: [3, 7, 5, 12, 9, 8, 6]

Decrease-key(index 3, value 1):
Set A[3] = 1. Array: [3, 7, 5, 1, 9, 8, 6]
Bubble up: parent of 3 is 1 (value 7). 1 < 7, swap. [3, 1, 5, 7, 9, 8, 6]
Parent of 1 is 0 (value 3). 1 < 3, swap. [1, 3, 5, 7, 9, 8, 6]
Index 0 is root. Stop.
After decrease-key: [1, 3, 5, 7, 9, 8, 6]

Extract-min:
Remove 1. Replace with last (6). Array: [6, 3, 5, 7, 9, 8]
Heapify down: 6 vs children 3, 5. Min = 3. Swap. [3, 6, 5, 7, 9, 8]
6 at index 1 vs children 7, 9. 6 < both. Stop.
After extract-min: [3, 6, 5, 7, 9, 8]
GATE Tip: Decrease-key is O(log n) and is crucial for Dijkstra's algorithm efficiency. With a binary heap, Dijkstra runs in O((V+E) log V). With a Fibonacci heap, decrease-key is O(1) amortized.

Problem 15: Trie Construction

Problem: Insert the words ["the", "there", "their", "them", "then"] into a trie. How many nodes does the trie have? How many leaf nodes?
Approach: Build trie character by character. Shared prefixes share nodes.
Step-by-step Solution:
graph TD R((root)) --> T((t)) T --> H((h)) H --> E1((e)) E1 --> M((m*)) E1 --> N((n*)) E1 --> R2((r)) E1 --> I((i)) R2 --> E2((e*)) I --> R3((r*)) E1 -.->|"the*"| E1
Building step by step:
1. "the": root → t → h → e* (3 new nodes + root)
2. "there": root → t → h → e → r → e* (2 new nodes: r, e)
3. "their": root → t → h → e → i → r* (2 new nodes: i, r)
4. "them": root → t → h → e → m* (1 new node: m)
5. "then": root → t → h → e → n* (1 new node: n)

Total nodes (excluding root): 9 (t, h, e, r, e, i, r, m, n)
Including root: 10 nodes
Leaf nodes: 4 (the second e in "there", r in "their", m in "them", n in "then")
Words ending mid-trie: 1 ("the" ends at 'e' which is not a leaf)
Key Takeaway: Tries provide O(L) lookup where L is the key length, independent of the number of stored keys. Space can be optimized with compressed tries (Patricia/radix trees). In GATE, trie questions usually involve counting nodes or analyzing complexity.

🧩 Problem Solving

12 GATE-level database problems covering relational algebra, SQL, functional dependencies, normalization, B+ trees, transaction serializability, and concurrency control.

Problem 1: Relational Algebra Queries

Problem: Given relations:
Student(RollNo, Name, Dept), Enrollment(RollNo, CourseID, Grade), Course(CourseID, CourseName, Credits).
Write relational algebra for: (a) Names of students enrolled in "DBMS". (b) Students enrolled in all 3-credit courses.
Approach: (a) Join + Select + Project. (b) Division operation.
Step-by-step Solution:
(a) Names of students enrolled in "DBMS":
1. Select DBMS course: σCourseName='DBMS'(Course)
2. Join with Enrollment: Enrollment ⨝ σCourseName='DBMS'(Course)
3. Join with Student and project Name:
πName(Student ⨝ (Enrollment ⨝ σCourseName='DBMS'(Course)))

(b) Students enrolled in ALL 3-credit courses (division):
1. Get all 3-credit CourseIDs: T1 = πCourseID(σCredits=3(Course))
2. Get (RollNo, CourseID) from Enrollment: T2 = πRollNo, CourseID(Enrollment)
3. Division: T3 = T2 ÷ T1
4. Get names: πName(Student ⨝ (T2 ÷ T1))
GATE Tip: Division (R ÷ S) finds tuples in R that are associated with ALL tuples in S. It's equivalent to the "for all" quantifier. Division can be expressed as: πA(R) - πA((πA(R) × S) - R).

Problem 2: SQL Queries

Problem: Using the same schema. Write SQL for: (a) Find departments where average grade (numeric) > 8.0. (b) Find students who have enrolled in more than 3 courses. (c) Find courses with no enrollments.
Approach: (a) GROUP BY + HAVING. (b) GROUP BY + HAVING COUNT. (c) LEFT JOIN or NOT IN/NOT EXISTS.
Step-by-step Solution:
(a) 
SELECT S.Dept, AVG(E.Grade) as AvgGrade
FROM Student S JOIN Enrollment E ON S.RollNo = E.RollNo
GROUP BY S.Dept
HAVING AVG(E.Grade) > 8.0;
(b) 
SELECT S.RollNo, S.Name
FROM Student S JOIN Enrollment E ON S.RollNo = E.RollNo
GROUP BY S.RollNo, S.Name
HAVING COUNT(E.CourseID) > 3;
(c) Three equivalent methods: 
-- Method 1: LEFT JOIN
SELECT C.CourseID, C.CourseName
FROM Course C LEFT JOIN Enrollment E ON C.CourseID = E.CourseID
WHERE E.CourseID IS NULL;

-- Method 2: NOT IN
SELECT CourseID, CourseName FROM Course
WHERE CourseID NOT IN (SELECT CourseID FROM Enrollment);

-- Method 3: NOT EXISTS
SELECT C.CourseID, C.CourseName FROM Course C
WHERE NOT EXISTS (SELECT 1 FROM Enrollment E
                  WHERE E.CourseID = C.CourseID);
Key Takeaway: HAVING filters groups, WHERE filters rows before grouping. NOT EXISTS is generally preferred over NOT IN because NOT IN has unexpected behavior with NULLs (if subquery returns NULL, NOT IN returns no rows).

Problem 3: Finding Candidate Keys from FDs

Problem: Relation R(A, B, C, D, E) with FDs: A → B, BC → D, D → E, E → A. Find all candidate keys.
Approach: Find attributes not on RHS (must be in every key). Compute closures starting from those. Check if they determine all attributes.
Step-by-step Solution:
Attributes on RHS: B, D, E, A → all except C.
C is not on any RHS, so C must be in every candidate key.

Try C alone: C+ = {C}. Not a superkey.

Try CA: (CA)+ = {C, A, B} (A→B) → {C, A, B, D} (BC→D) → {C, A, B, D, E} (D→E) = ABCDE. CA is a candidate key.

Try CB: (CB)+ = {C, B, D} (BC→D) → {C, B, D, E} (D→E) → {C, B, D, E, A} (E→A) = ABCDE. CB is a candidate key.

Try CD: (CD)+ = {C, D, E} (D→E) → {C, D, E, A} (E→A) → {C, D, E, A, B} (A→B) = ABCDE. CD is a candidate key.

Try CE: (CE)+ = {C, E, A} (E→A) → {C, E, A, B} (A→B) → {C, E, A, B, D} (BC→D) = ABCDE. CE is a candidate key.

All candidate keys: {AC, BC, CD, CE}
GATE Tip: Attributes that appear only on LHS (never on RHS) must be in every key. Attributes that appear only on RHS are never in any key. Attributes on both sides may or may not be in keys -- compute closures to check. There can be multiple candidate keys.

Problem 4: Normalization to 3NF

Problem: R(A, B, C, D, E) with FDs: A → BC, C → D, D → E. Candidate key is A. Current form? Decompose into 3NF.
Approach: Check: 2NF (no partial dependency on key), 3NF (no transitive dependency). Decompose using the synthesis algorithm.
Step-by-step Solution:
Key = {A}. All attributes are fully dependent on A (no partial dependency possible with single-attribute key).
R is in 2NF.

But: A → C → D (transitive). Also A → D → E (transitive).
R is NOT in 3NF. (Non-prime attributes D and E are transitively dependent on key A.)

3NF Decomposition (Synthesis Algorithm):
1. Minimal cover: {A → BC, C → D, D → E} (already minimal)
2. Create a relation for each FD:
  R1(A, B, C) -- from A → BC
  R2(C, D) -- from C → D
  R3(D, E) -- from D → E
3. Check if any relation contains a candidate key of R. R1 contains A. Yes.

3NF decomposition: R1(A, B, C), R2(C, D), R3(D, E)
This is lossless-join and dependency-preserving.
Key Takeaway: 3NF synthesis algorithm always gives lossless-join, dependency-preserving decomposition. BCNF decomposition may lose some FDs. If the question says "decompose into 3NF," use synthesis. If "BCNF," use the decomposition algorithm (may not preserve all FDs).

Problem 5: BCNF Decomposition

Problem: R(A, B, C, D) with FDs: AB → C, C → A, BC → D. Candidate keys? Is R in BCNF? If not, decompose.
Approach: Find candidate keys. BCNF: every non-trivial FD X→Y, X must be a superkey. If not, decompose on the violating FD.
Step-by-step Solution:
Finding keys:
B is not on any RHS → B must be in every key.
(AB)+ = {A,B,C,D} (AB→C, BC→D). AB is a key.
(BC)+ = {B,C,A,D} (C→A, BC→D). BC is a key.

BCNF check:
- AB → C: AB is a superkey. OK.
- C → A: C is not a superkey (C+ = {C, A} ≠ ABCD). BCNF violation!
- BC → D: BC is a superkey. OK.

Decompose on C → A:
R1(C, A) -- from the violating FD
R2(B, C, D) -- remaining attributes with C

Check R1(C, A): C → A, C is key. In BCNF.
Check R2(B, C, D): FD BC → D holds. BC is key. In BCNF.

BCNF decomposition: R1(C, A), R2(B, C, D)

Dependency preservation? AB → C cannot be checked within R1 or R2 alone (A and B are in different relations). Not dependency-preserving!
GATE Tip: BCNF decomposition is always lossless but may not preserve all FDs. 3NF is the highest normal form that guarantees both. In GATE, if asked "is the decomposition dependency-preserving?" check if the closure of FDs projected onto each sub-relation equals the original FD set.

Problem 6: B+ Tree Insertion

Problem: Insert keys 5, 8, 1, 7, 3, 12, 9, 6 into a B+ tree of order 3 (max 2 keys per node, min 1). Show the tree after all insertions.
Approach: Order 3: each node holds max 2 keys. When full, split: middle key goes up. Leaf nodes linked for range queries.
Step-by-step Solution:
Insert 5: [5]
Insert 8: [5, 8]
Insert 1: [1, 5, 8] -- overflow! Split at middle (5).
graph TD R5((5)) --> L1["[1]"] R5 --> L2["[5, 8]"]
Insert 7: Goes to right leaf [5, 7, 8] -- overflow! Split at 7.
graph TD R57["[5, 7]"] --> L1["[1]"] R57 --> L2["[5]"] R57 --> L3["[7, 8]"]
Wait -- let me be more careful. In a B+ tree of order 3, internal nodes have max 2 keys and 3 children. Leaf nodes have max 2 keys.

Insert 7: Search: 7 > 5, go to [5, 8]. Insert: [5, 7, 8] overflow.
Split leaf: left = [5], right = [7, 8], push 7 up.
Root becomes [5, 7] with children [1], [5], [7, 8].

Insert 3: 3 < 5, go to [1]. Insert: [1, 3]. Fits.
Insert 12: 12 > 7, go to [7, 8]. Insert: [7, 8, 12] overflow.
Split: left = [7], right = [8, 12], push 8 up.
Root = [5, 7, 8] -- overflow in root!
Split root at 7: new root [7], left child [5], right child [8].
graph TD R7((7)) --> I5["[5]"] R7 --> I8["[8]"] I5 --> L1["[1,3]"] I5 --> L2["[5]"] I8 --> L3["[7]"] I8 --> L4["[8,12]"]
Insert 9: 9 > 7, then > 8, go to [8, 12]. Insert: [8, 9, 12] overflow.
Split: [8], [9, 12], push 9 up. Internal [8, 9]. Fits.

Insert 6: 6 < 7, then > 5, go to [5]. Insert: [5, 6]. Fits.

Final tree:
graph TD R7((7)) --> I5["[5]"] R7 --> I89["[8, 9]"] I5 --> L1["[1,3]"] I5 --> L2["[5,6]"] I89 --> L3["[7]"] I89 --> L4["[8]"] I89 --> L5["[9,12]"]
GATE Tip: B+ tree of order p: internal nodes have max p-1 keys and p children, min ⌈p/2⌉ children. Leaf nodes have max p-1 keys, min ⌈(p-1)/2⌉ keys. All leaf nodes are at the same level. GATE often asks for the number of levels given n keys and order p.

Problem 7: Transaction Serializability

Problem: Given schedule S: R1(A), R2(A), W1(A), R1(B), W2(A), W1(B), R2(B), W2(B). Is S conflict-serializable? If yes, find the equivalent serial order.
Approach: Build a precedence graph. Find conflicting operations (same data item, at least one write, different transactions). Check for cycles.
Step-by-step Solution:
Conflicting pairs:
1. R2(A) before W1(A): T2 → T1 (R-W conflict on A)
2. W1(A) before W2(A): T1 → T2 (W-W conflict on A)
3. R1(A) before W2(A): T1 → T2 (R-W conflict on A)
4. W1(B) before R2(B): T1 → T2 (W-R conflict on B)
5. W1(B) before W2(B): T1 → T2 (W-W conflict on B)

Precedence graph:
Edges: T2 → T1 (from conflict 1), T1 → T2 (from conflicts 2,3,4,5)

graph LR T1((T1)) -->|"W1(A) before W2(A)"| T2((T2)) T2 -->|"R2(A) before W1(A)"| T1
Cycle exists: T1 → T2 → T1

Schedule S is NOT conflict-serializable.
Key Takeaway: A schedule is conflict-serializable iff the precedence graph is acyclic. The topological sort of the graph gives the equivalent serial order. GATE often gives a schedule and asks you to draw the precedence graph and check for cycles.

Problem 8: Two-Phase Locking Analysis

Problem: Given operations of T1 and T2 with lock requests: T1: Lock-S(A), Read(A), Lock-X(B), Write(B), Unlock(A), Unlock(B). T2: Lock-S(B), Read(B), Lock-X(A), Write(A), Unlock(B), Unlock(A). Can these transactions deadlock? Does each follow 2PL?
Approach: 2PL: locks only acquired before any release (growing phase before shrinking). Check if concurrent execution can lead to circular wait.
Step-by-step Solution:
2PL check for T1:
Locks acquired: Lock-S(A), Lock-X(B). Then unlocks: Unlock(A), Unlock(B).
All locks before all unlocks. T1 follows 2PL.

2PL check for T2:
Locks acquired: Lock-S(B), Lock-X(A). Then unlocks: Unlock(B), Unlock(A).
All locks before all unlocks. T2 follows 2PL.

Deadlock analysis:
If interleaved: T1: Lock-S(A) [granted], T2: Lock-S(B) [granted]
T1: Lock-X(B) [blocked -- T2 holds S-lock on B]
T2: Lock-X(A) [blocked -- T1 holds S-lock on A]

Circular wait: T1 waits for T2 (on B), T2 waits for T1 (on A). DEADLOCK!

2PL guarantees serializability but does NOT prevent deadlocks.
GATE Tip: 2PL prevents non-serializable schedules but allows deadlocks. Strict 2PL (hold all locks until commit) prevents cascading rollbacks. Rigorous 2PL additionally prevents deadlocks in some implementations. To prevent deadlock: use wait-die or wound-wait schemes, or timeout.

Problem 9: SQL Aggregation and Subqueries

Problem: Table: Employee(EmpID, Name, Dept, Salary, MgrID). Write SQL to find: (a) Second highest salary, (b) Employees earning more than their department average, (c) Departments where all employees earn > 50000.
Approach: (a) Subquery with MAX, or DENSE_RANK. (b) Correlated subquery. (c) NOT EXISTS with negation, or HAVING MIN.
Step-by-step Solution:
(a) Second highest salary: 
-- Method 1: Subquery
SELECT MAX(Salary) FROM Employee
WHERE Salary < (SELECT MAX(Salary) FROM Employee);

-- Method 2: DENSE_RANK
SELECT Salary FROM (
    SELECT Salary, DENSE_RANK() OVER (ORDER BY Salary DESC) as rnk
    FROM Employee
) t WHERE rnk = 2;
(b) Employees earning more than department average: 
SELECT E.Name, E.Salary, E.Dept
FROM Employee E
WHERE E.Salary > (
    SELECT AVG(E2.Salary)
    FROM Employee E2
    WHERE E2.Dept = E.Dept
);
(c) Departments where ALL employees earn > 50000: 
SELECT Dept FROM Employee
GROUP BY Dept
HAVING MIN(Salary) > 50000;
GATE Tip: "For all" conditions can be written as: HAVING MIN/MAX, NOT EXISTS with negation, or using COUNT matching. DENSE_RANK handles ties properly for "nth highest" questions. MAX-based approach fails when there are ties for the highest value.

Problem 10: Functional Dependency — Canonical Cover

Problem: Find the canonical (minimal) cover of: F = {A → BC, B → C, AB → D, D → B}.
Approach: (1) Split RHS to singletons. (2) Remove extraneous attributes from LHS. (3) Remove redundant FDs.
Step-by-step Solution:
Step 1: Split RHS:
A → B, A → C, B → C, AB → D, D → B

Step 2: Remove extraneous LHS attributes:
AB → D: Is B extraneous? Check if A → D under remaining FDs.
A+ = {A, B, C} (using A→B, A→C). Does not include D.
But wait: A+ = {A} → {A,B} (A→B) → {A,B,C} (A→C, B→C) -- still no D.
B is NOT extraneous.

Is A extraneous in AB→D? Check B+ = {B, C}. No D. A is NOT extraneous.
AB → D stays.

Step 3: Remove redundant FDs:
Check A → C: Without it, can we derive A → C? A+ = {A, B} (A→B) → {A, B, C} (B→C). Yes! A → C is redundant via A → B → C.
Remove A → C.

Remaining: A → B, B → C, AB → D, D → B

Check AB → D: Without it, A+ (with A→B) = {A,B,C}. No D. Not redundant.

Check A → B: Without it, A+ = {A}. Cannot get B. Not redundant.

Canonical cover: {A → B, B → C, AB → D, D → B}
Key Takeaway: Canonical cover has: (1) Single attribute on RHS, (2) No extraneous LHS attributes, (3) No redundant FDs. The canonical cover is not unique if you remove redundant FDs in different orders. It's used in 3NF synthesis.

Problem 11: Recoverable and Cascadeless Schedules

Problem: Schedule S: W1(A), R2(A), W2(B), Commit2, Commit1. Is S (a) recoverable? (b) cascadeless? (c) strict?
Approach: Recoverable: if Tj reads from Ti, Ti commits before Tj. Cascadeless: Tj reads A only after Ti (that wrote A) commits. Strict: no read or write of A until Ti commits.
Step-by-step Solution:
T2 reads A after T1 writes A (R2(A) after W1(A)), so T2 reads from T1.

(a) Recoverable?
T2 reads from T1. T2 commits (Commit2) BEFORE T1 commits (Commit1).
If T1 aborts after T2 has committed, T2 used an invalid value. Cannot undo T2.
NOT recoverable.

(b) Cascadeless?
T2 reads A before T1 commits. NOT cascadeless. (Cascadeless requires T1 to commit before T2 reads A.)

(c) Strict?
T2 reads A before T1 commits. NOT strict.

Hierarchy: Strict ⊂ Cascadeless ⊂ Recoverable ⊂ Serializable (in general).
This schedule is NONE of the above.
GATE Tip: The schedule hierarchy: Serial ⊂ Strict ⊂ Cascadeless ⊂ Recoverable. A non-recoverable schedule should never be allowed by a DBMS. Strict 2PL ensures strict schedules. In GATE, carefully track which transaction's write was read by which transaction.

Problem 12: Relational Algebra — Equivalence

Problem: Show that σp∧q(R) = σp(σq(R)). Also, is πA(σp(B)(R)) = σp(B)(πA(R)) always valid? When can selection be pushed below projection?
Approach: Selection is commutative and can be cascaded. Projection and selection can be swapped only if the selection condition involves only projected attributes.
Step-by-step Solution:
Part 1: σp∧q(R) = σp(σq(R))
A tuple t is in σp∧q(R) iff t satisfies both p and q.
σq(R) keeps tuples satisfying q. Then σp keeps those also satisfying p.
Both sides select tuples satisfying p AND q. They are equivalent.

Also: σp(σq(R)) = σq(σp(R)) -- selection is commutative.

Part 2: πA(σp(B)(R)) vs σp(B)(πA(R))
If B ∉ A, then πA(R) does not contain attribute B. The condition p(B) cannot be evaluated.
NOT always valid.

Valid only when the selection attributes are a subset of the projected attributes (B ⊆ A).

Example: R(X, Y, Z). πX(σY>5(R)) ≠ σY>5(πX(R)) because Y is not available after πX.
GATE Tip: Query optimization rules: (1) Push selections down (reduces intermediate result size). (2) Push projections down (reduces tuple width). (3) Selection before join is almost always better. These equivalence rules are the basis of query optimization and frequently appear in GATE.

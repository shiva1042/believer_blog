Problem 1: FCFS Scheduling

Problem: Processes with (Arrival Time, Burst Time): P1(0,6), P2(1,8), P3(2,7), P4(3,3). Compute average waiting time and turnaround time for FCFS scheduling.
Approach: FCFS: execute in arrival order. Waiting time = Start time - Arrival time. Turnaround = Completion - Arrival.
Step-by-step Solution:

Key Takeaway: FCFS is non-preemptive and suffers from the convoy effect (short processes wait behind long ones). It is the simplest but often has the worst average waiting time.

Problem 2: SJF (Non-Preemptive) Scheduling

Problem: Same processes: P1(0,6), P2(1,8), P3(2,7), P4(3,3). Compute average WT for non-preemptive SJF.
Approach: At each decision point (when CPU becomes free), choose the arrived process with shortest burst time.
Step-by-step Solution:
t=0: Only P1 available. Run P1 (BT=6). Completes at t=6.
t=6: P2(8), P3(7), P4(3) available. Shortest = P4(3). Run P4. Completes at t=9.
t=9: P2(8), P3(7) available. Shortest = P3(7). Run P3. Completes at t=16.
t=16: P2(8) left. Run P2. Completes at t=24.

Problem 8: Disk Scheduling — SCAN

Problem: Disk has 200 cylinders (0-199). Head starts at cylinder 53, moving toward 0. Request queue: 98, 183, 37, 122, 14, 124, 65, 67. Calculate total seek time for SCAN and C-SCAN.
Approach: SCAN: move in current direction servicing requests, reverse at end. C-SCAN: move in one direction, jump to other end.
Step-by-step Solution:
SCAN (moving toward 0):
Order: 53 → 37 → 14 → 0 (end) → 65 → 67 → 98 → 122 → 124 → 183

Seek distances: |53-37| + |37-14| + |14-0| + |0-65| + |65-67| + |67-98| + |98-122| + |122-124| + |124-183|
= 16 + 23 + 14 + 65 + 2 + 31 + 24 + 2 + 59 = 236

C-SCAN (moving toward 0, circular):
Order: 53 → 37 → 14 → 0 (end) → 199 (jump to other end) → 183 → 124 → 122 → 98 → 67 → 65

Seek = 53 + 199 + (199-65) = 53 + 199 + 134 = 386
(53 to 0, then 0 to 199 is counted as 199, then 199 down to 65 is 134)
Key Takeaway: SCAN provides uniform wait time for middle cylinders but not edges. C-SCAN provides more uniform wait times. LOOK and C-LOOK are practical variants that don't go to the absolute ends if there are no requests there.

Problem 9: Virtual Memory — Page Table Calculation

Problem: Virtual address space = 32 bits, page size = 4 KB, physical memory = 256 MB, page table entry = 4 bytes. Calculate: (a) Number of pages, (b) Number of frames, (c) Page table size, (d) Number of page tables if using 2-level paging with 10-bit first level.
Approach: Pages = VA space / page size. Frames = physical memory / page size. Page table size = pages * PTE size.
Step-by-step Solution:
(a) Page size = 4 KB = 212 bytes. Offset bits = 12.
Number of pages = 232 / 212 = 220 = 1,048,576 pages (1M)

(b) Physical memory = 256 MB = 228 bytes.
Number of frames = 228 / 212 = 216 = 65,536 frames

(c) Page table size = 220 × 4 bytes = 4 MB
Single-level page table = 4 MB (too large to keep in contiguous memory!)

(d) 2-level paging: 20 page number bits split as 10 + 10.
First level: 210 entries × 4 bytes = 4 KB (1 page -- fits nicely)
Each second level table: 210 entries × 4 bytes = 4 KB
Max second level tables: 210 = 1024
But only the needed ones are allocated.
Total if all allocated: 1 + 1024 = 1025 page tables, total size = 1025 × 4KB ≈ 4 MB + 4 KB
GATE Tip: Two-level paging reduces memory usage because not all second-level tables need to be in memory. Inverted page table: one entry per frame (216 entries here = 256 KB), but requires hashing for lookup. TLB hit dramatically reduces access time.

Problem 10: Effective Memory Access Time with TLB

Problem: TLB access time = 5 ns, Memory access time = 100 ns, TLB hit ratio = 90%, page fault rate = 2% (of TLB misses), page fault service time = 5 ms. Calculate effective memory access time.
Approach: EMAT = TLB_hit_time + TLB_miss_time, accounting for page faults on misses.
Step-by-step Solution:
TLB Hit (90%): TLB access + Memory access = 5 + 100 = 105 ns

TLB Miss, No Page Fault (10% × 98% = 9.8%):
TLB access + Page table access (memory) + Memory access = 5 + 100 + 100 = 205 ns

TLB Miss, Page Fault (10% × 2% = 0.2%):
TLB access + Page table access + Page fault service + Memory access = 5 + 100 + 5,000,000 + 100 = 5,000,205 ns

EMAT = 0.90(105) + 0.098(205) + 0.002(5,000,205)
= 94.5 + 20.09 + 10,000.41
= 10,115 ns ≈ 10.1 microseconds
GATE Tip: Even a tiny page fault rate dominates EMAT because disk access is ~107 times slower than memory. For acceptable performance (< 10% slowdown), page fault rate must be < 1 in 10 million. This is why locality of reference is critical.

Problem 11: Semaphore — Producer-Consumer

Problem: In the bounded-buffer problem with buffer size 5, if initially 3 items are in the buffer, what are the initial values of semaphores? If 2 producers and 1 consumer operate, trace the semaphore values after: Producer produces, Producer produces, Consumer consumes, Consumer consumes.
Approach: Three semaphores: mutex=1, empty=N-current, full=current. Producer: wait(empty), wait(mutex), produce, signal(mutex), signal(full). Consumer: wait(full), wait(mutex), consume, signal(mutex), signal(empty).
Step-by-step Solution:
N = 5, initially 3 items in buffer.
Initial: mutex=1, empty=2, full=3
OperationmutexemptyfullBuffer items
Initial1233

Producer produces1055
Consumer consumes1144

Note: mutex returns to 1 after each complete operation (wait+signal pair within each operation).
If another producer tries when empty=0, it blocks on wait(empty) until a consumer runs.
Key Takeaway: In producer-consumer: empty + full = N always. If you swap the order of wait(empty) and wait(mutex) in the producer, deadlock can occur (holding mutex while blocked on empty). Order matters!

Problem 12: Deadlock Detection

Problem: Given: Available = [0, 0, 0].
ProcessAllocationRequest
P00,1,00,0,0
P12,0,02,0,2
P23,0,30,0,0
P32,1,11,0,0
P40,0,20,0,2
Is the system deadlocked? If so, which processes are deadlocked?
Approach: Similar to Banker's safety algorithm but using Request instead of Need. Find processes whose Request ≤ Available.
Step-by-step Solution:
Available = [0, 0, 0]

Step 1: Check P0: Request [0,0,0] ≤ [0,0,0]. Yes! Run P0, release [0,1,0]. Available = [0,1,0].
Step 2: Check P2: Request [0,0,0] ≤ [0,1,0]. Yes! Run P2, release [3,0,3]. Available = [3,1,3].
Step 3: Check P1: Request [2,0,2] ≤ [3,1,3]. Yes! Run P1, release [2,0,0]. Available = [5,1,3].
Step 4: Check P3: Request [1,0,0] ≤ [5,1,3]. Yes! Run P3, release [2,1,1]. Available = [7,2,4].
Step 5: Check P4: Request [0,0,2] ≤ [7,2,4]. Yes! Run P4.

All processes can complete. The system is NOT deadlocked.
GATE Tip: Deadlock detection is like the safety algorithm but uses actual pending requests, not maximum future needs. If some processes cannot be completed, those processes are deadlocked. The four necessary conditions for deadlock: mutual exclusion, hold and wait, no preemption, circular wait.

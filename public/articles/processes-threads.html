Processes and Threads

⚙️ Section 8: Operating System

1. Process Concept

A process is a program in execution. It includes the program code (text section), current activity represented by the program counter and CPU registers, a stack for temporary data, a data section for global variables, and possibly a heap for dynamically allocated memory.

1.1 Process States

A process transitions through several states during its lifetime:

StateDescription
NewThe process is being created.
ReadyThe process is waiting to be assigned to a processor.
RunningInstructions are being executed on the CPU.
Waiting (Blocked)The process is waiting for an I/O event or signal.
TerminatedThe process has finished execution.

1.3 Process Control Block (PCB)

The PCB is a data structure maintained by the OS for every process. It contains:

FieldPurpose
Process ID (PID)Unique identifier for the process
Process StateCurrent state (new, ready, running, waiting, terminated)
Program CounterAddress of the next instruction to execute
CPU RegistersContents of all process-centric registers
Memory Management InfoPage tables, segment tables, base/limit registers
I/O Status InfoList of open files, allocated I/O devices
Scheduling InfoPriority, scheduling queue pointers, CPU time used
Accounting InfoCPU time used, time limits, job or process numbers

2. Process Creation and Termination

2.1 Process Creation

Processes are created through system calls. In UNIX-like systems, the primary mechanism is the fork() system call, which creates a child process as a copy of the parent.

Example: fork() in C 
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();
    if (pid == 0) {
        // Child process
        printf("Child PID: %d\n", getpid());
        execlp("/bin/ls", "ls", NULL);
    } else if (pid > 0) {
        // Parent process
        wait(NULL);   // wait for child to finish
        printf("Child completed\n");
    }
    return 0;
}

2.2 Key System Calls

System CallDescription
fork()Creates a new child process (copy of parent). Returns 0 in child, child PID in parent.
exec()Replaces the process memory with a new program. Variants: execlp, execvp, etc.
wait()Parent blocks until a child process terminates.
exit()Terminates the calling process and returns a status code to the parent.

2.3 Process Termination

A process terminates when it finishes its last statement and calls exit(). The parent can also terminate a child using kill(). A zombie process occurs when a child terminates but the parent has not yet called wait(). An orphan process results when the parent terminates before the child; the child is then adopted by the init process (PID 1).

3. Threads

A thread is the smallest unit of CPU execution within a process. Threads within the same process share the code section, data section, and OS resources such as open files, but each thread has its own thread ID, program counter, register set, and stack.

3.1 User-Level vs. Kernel-Level Threads

FeatureUser-Level Threads (ULT)Kernel-Level Threads (KLT)
ManagementThread library in user spaceOS kernel
Context SwitchFast (no kernel involvement)Slower (requires kernel mode switch)
SchedulingNot visible to OS schedulerScheduled by OS directly
BlockingOne thread blocks entire processOnly the blocked thread is suspended
Multiprocessor SupportCannot run on multiple CPUsCan run on multiple CPUs in parallel
ExamplesPOSIX Pthreads (user), GNU Portable ThreadsWindows threads, Linux native threads

3.2 Multithreading Models

ModelMappingDescription
Many-to-OneN user threads → 1 kernel threadEfficient but one blocking call blocks all. No true parallelism.
One-to-One1 user thread → 1 kernel threadMore concurrency, but creating a thread requires kernel thread. Used in Linux and Windows.
Many-to-ManyN user threads → M kernel threads (N ≥ M)Best flexibility. OS creates sufficient kernel threads. Solaris prior to version 9 used this model.

4. Inter-Process Communication (IPC)

Processes need to communicate and synchronize. The two fundamental IPC models are shared memory and message passing.

4.1 Shared Memory

Two or more processes share a region of memory. The OS sets up the shared region; after that, communication is handled entirely in user space without further kernel intervention. This is fast but requires explicit synchronization (e.g., semaphores) to avoid race conditions.

POSIX Shared Memory (conceptual): 
// Producer creates shared memory
int shm_fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0666);
ftruncate(shm_fd, 4096);
void *ptr = mmap(0, 4096, PROT_WRITE, MAP_SHARED, shm_fd, 0);
sprintf(ptr, "Hello from producer");

4.2 Message Passing

Processes communicate by sending and receiving messages through a communication link managed by the kernel. This is simpler to program but slower than shared memory because every message requires a system call.

AspectShared MemoryMessage Passing
SpeedFaster (after setup)Slower (kernel involved per message)
SynchronizationMust be explicitly managedImplicitly handled by send/receive
Ease of UseMore complexSimpler API
Best forLarge data, same machineSmall messages, distributed systems

4.3 Pipes

Ordinary (unnamed) pipes provide one-way communication between a parent and child process. Data is written at one end and read at the other. They are created with the pipe() system call and exist only while processes are running.

Named pipes (FIFOs) exist as special files in the file system and allow unrelated processes to communicate. They are created with mkfifo() and persist beyond process lifetime.

Pipe Example: 
int fd[2];
pipe(fd);       // fd[0] = read end, fd[1] = write end
if (fork() == 0) {
    close(fd[1]);
    char buf[100];
    read(fd[0], buf, sizeof(buf));
    printf("Child received: %s\n", buf);
} else {
    close(fd[0]);
    write(fd[1], "Hello child", 12);
    wait(NULL);
}

4.4 Signals

Signals are software interrupts sent to a process to notify it of an event. Common signals include:

SignalNumberDescription
SIGINT2Interrupt from keyboard (Ctrl+C)
SIGKILL9Kill signal (cannot be caught or ignored)
SIGSEGV11Invalid memory reference (segmentation fault)
SIGTERM15Termination signal (can be caught)
SIGCHLD17Child process stopped or terminated
SIGSTOP19Stop process (cannot be caught)

A process can handle signals using signal() or sigaction() system calls. Default actions include termination, core dump, ignore, or stop.

5. Summary and GATE Key Points

Key Points for GATE:
  • fork() returns 0 to the child and the child's PID to the parent. A sequence of n fork() calls creates 2n − 1 new processes.
  • Threads share code, data, and files but have separate stacks and registers.
  • User-level threads are invisible to the OS; a blocking system call blocks the entire process.
  • Shared memory is faster but needs explicit synchronization; message passing is simpler but slower.
  • A zombie process has terminated but its entry remains in the process table because the parent has not called wait().
  • Context switching between processes is more expensive than between threads of the same process.

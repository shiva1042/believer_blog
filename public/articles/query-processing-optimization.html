Databases

Query Processing and Optimization

Query processing refers to the range of activities involved in extracting data from a database. It includes translating queries expressed in high-level database languages (such as SQL) into expressions that can be used at the physical level of the file system, a variety of query-optimizing transformations, and actual evaluation of queries. This article covers Chapters 15 and 16 of Korth/Silberschatz's Database System Concepts and is essential for GATE CSE preparation.

1. Overview of Query Processing Steps

The processing of a query in a relational database system goes through three major phases:

  1. Parsing and Translation: The SQL query is parsed, validated against the schema, and translated into an equivalent relational algebra expression (or an internal query tree).
  2. Optimization: The query optimizer generates multiple equivalent evaluation plans and chooses the one with the lowest estimated cost.
  3. Evaluation: The chosen query-evaluation plan is executed by the query-evaluation engine, and the result is returned to the user.
flowchart LR A["SQL Query"] --> B["Parser & Translator"] B --> C["Relational Algebra Expression"] C --> D["Query Optimizer"] D --> E["Execution Plan"] E --> F["Evaluation Engine"] F --> G["Query Result"] D -- "Statistics from catalog" --> D
Key Insight: The query optimizer does NOT find the absolute best plan. It finds a reasonably good plan while avoiding the worst plans. An exhaustive search of all plans is often too expensive, so heuristics and cost-based pruning are used.

2. Measures of Query Cost

To compare different evaluation strategies, we need a way to estimate the cost of each. The major cost components are:

Cost Component Description Typical Weight
Disk I/O Number of block transfers (reads/writes) between disk and memory Dominant factor
Seek Time Number of disk seeks required Significant (5-10ms per seek)
CPU Cost Time for in-memory processing (comparisons, hashing, etc.) Usually secondary
Network Cost Cost of transferring data over network (distributed databases) Relevant in distributed settings

The standard cost model uses two parameters:

  • tT = time for one block transfer (typically 0.1 ms)
  • tS = time for one disk seek (typically 4-10 ms)

Total Cost = (number of block transfers) × tT + (number of seeks) × tS

GATE Convention: In most GATE problems, the cost is measured purely in terms of the number of disk block transfers (ignoring seek time and CPU cost), unless stated otherwise.

2.1 Notation Used

SymbolMeaning
brNumber of disk blocks containing records of relation r
nrNumber of tuples in relation r
frBlocking factor of r (tuples per block)
V(A, r)Number of distinct values of attribute A in relation r
hiHeight of index i (for B+ tree indices)
MNumber of memory buffer blocks available

Recall: br = ceil(nr / fr)

3. Selection Operation

The selection operation σθ(r) retrieves tuples from relation r that satisfy the predicate θ. Several algorithms exist with different costs.

3.1 Linear Search (Full Table Scan)

Scan every block of the relation and test each tuple against the selection condition.

  • Cost (general predicate): br block transfers + 1 seek
  • Cost (key attribute equality): Average case = br/2 block transfers + 1 seek (stop after finding the match)

Linear search is always applicable regardless of selection condition, ordering of the file, or availability of indices. It is the baseline algorithm.

3.2 Binary Search

Applicable only when the file is sorted on the selection attribute and the condition is an equality comparison on that attribute.

  • Cost: ceil(log2(br)) block transfers for the search + seeks
  • If multiple records match (non-key), add the cost of reading contiguous overflow blocks.

Formula for non-key equality: ceil(log2(br)) + ceil(nr / V(A,r) / fr) - 1

3.3 Index Scan

Selection using an index. The type of index determines the cost.

3.3.1 Primary Index (Clustering Index), Equality on Key

  • Traverse the B+ tree from root to leaf, then read one data block.
  • Cost: (hi + 1) block transfers + (hi + 1) seeks

3.3.2 Primary Index, Equality on Non-Key

  • Multiple records may match; they are stored contiguously.
  • Cost: hi + ceil(nr / V(A,r) / fr) block transfers

3.3.3 Secondary Index (Non-clustering), Equality on Key

  • Cost: (hi + 1) block transfers (same as primary key case)

3.3.4 Secondary Index, Equality on Non-Key

  • Each matching record may be on a different block (worst case).
  • Cost: hi + nr / V(A,r) block transfers (up to one seek per record)
GATE Trap: Secondary index on a non-key attribute can be more expensive than a linear scan when the number of matching tuples is large, because each record fetch may require a separate disk block read.

3.4 Cost Summary Table for Selection

Algorithm Condition Block Transfers
Linear search (key) Equality on key br/2 (avg)
Linear search (general) Any br
Binary search (key) Equality on sort key ceil(log2(br))
Primary index, key Equality on key hi + 1
Primary index, non-key Equality on non-key hi + ceil(sc(A,r)/fr)
Secondary index, key Equality on key hi + 1
Secondary index, non-key Equality on non-key hi + sc(A,r)

Where sc(A,r) = nr / V(A,r) is the selection cardinality.

4. Join Operations

The join operation is one of the most expensive and frequently used operations in query processing. Several algorithms have been developed. We use r and s to denote the two relations being joined, with br, bs blocks and nr, ns tuples respectively.

4.1 Nested-Loop Join (Simple / Tuple-Based)

for each tuple t_r in r:
    for each tuple t_s in s:
        if (t_r, t_s) satisfies join condition:
            output (t_r, t_s)

Cost Analysis:

  • For each tuple in r (the outer relation), we scan all of s (the inner relation).
  • Block transfers: nr × bs + br
  • Seeks: nr + br
Tip: The relation with fewer tuples should be the outer relation to minimize cost.

4.2 Block Nested-Loop Join

Instead of iterating tuple-by-tuple in the outer loop, we iterate block-by-block. For each block of the outer relation, we scan all blocks of the inner relation. 

for each block B_r of r:
    for each block B_s of s:
        for each tuple t_r in B_r:
            for each tuple t_s in B_s:
                if (t_r, t_s) satisfies join condition:
                    output (t_r, t_s)

Cost Analysis:

  • Block transfers: br × bs + br
  • Seeks: 2 × br

Improvement with M Memory Blocks

If we have M buffer blocks, use (M-2) blocks for the outer relation, 1 block for the inner, and 1 block for output.

  • Block transfers: ceil(br/(M-2)) × bs + br
  • Seeks: 2 × ceil(br/(M-2))
GATE Favorite: Block nested-loop join cost formula with M buffers is a frequently tested topic. Always make the smaller relation the outer to minimize the number of scans of the inner.

4.3 Sort-Merge Join

Both relations are first sorted on the join attribute, then merged in a manner similar to the merge step of merge sort. 

1. Sort r on join attribute A  (if not already sorted)
2. Sort s on join attribute A  (if not already sorted)
3. Merge:
   pr = first tuple of r
   ps = first tuple of s
   while (pr != null and ps != null):
       if pr.A == ps.A:
           output all matching pairs
       else if pr.A < ps.A:
           advance pr
       else:
           advance ps

Cost Analysis:

  • Sorting cost for r: 2 × br × ceil(logM-1(ceil(br/M))) (for external sort, explained below)
  • Sorting cost for s: 2 × bs × ceil(logM-1(ceil(bs/M)))
  • Merge cost: br + bs (assuming no duplicates spanning many blocks)
  • Total (if not pre-sorted): Sort cost for r + Sort cost for s + br + bs
When is sort-merge efficient? When one or both relations are already sorted on the join attribute, or when the result needs to be sorted anyway. It scales well for large relations.

4.4 Hash Join

Uses a hash function on the join attribute to partition both relations into corresponding buckets, then matches tuples within the same bucket. 

Phase 1 (Partition):
  Hash r into n partitions: r_0, r_1, ..., r_(n-1)
  Hash s into n partitions: s_0, s_1, ..., s_(n-1)

Phase 2 (Build & Probe):
  for each partition i:
      Build hash table on r_i (smaller partition)
      Probe with each tuple of s_i
      Output matching pairs

Cost Analysis (no overflow):

  • Partitioning phase: 2 × (br + bs) block transfers (read + write each block once)
  • Matching phase: br + bs block transfers
  • Total: 3 × (br + bs) block transfers

Memory requirement: The smaller relation (say r) must satisfy br ≤ M × (M-1) or equivalently each partition of the build input must fit in memory. Typically requires M ≥ ceil(sqrt(bs)) + 1 where s is the larger relation (but the build is done on the smaller).

4.5 Join Cost Comparison Table

Algorithm Block Transfers Memory Needed Best When
Simple nested-loop nr × bs + br 3 blocks Tiny outer relation
Block nested-loop ceil(br/(M-2)) × bs + br M blocks Small relations, limited memory
Sort-merge join Sort costs + br + bs M blocks Pre-sorted data or sort needed later
Hash join 3(br + bs) sqrt(bsmaller) Equi-joins, adequate memory
Index nested-loop br + nr × c Minimal + index Index exists on inner join attr

Where c is the cost of a single index lookup and retrieval on the inner relation.

5. External Sorting (Multi-Way Merge Sort)

When the data to be sorted does not fit in main memory, we use external sorting. The standard algorithm is the multi-way merge sort.

5.1 Algorithm

  1. Create initial sorted runs: Read M blocks into memory, sort them in-memory, write them out as a sorted run. Repeat until all data is processed. This creates ceil(br/M) initial runs.
  2. Merge passes: In each pass, merge up to (M-1) runs at a time (using 1 block per run for input and 1 block for output). Repeat until only one run remains.

5.2 Cost Formula

Let br be the number of blocks in the relation and M be the number of available memory blocks.

  • Number of initial runs: N0 = ceil(br/M)
  • Number of merge passes: ceil(logM-1(N0))
  • Total number of passes: 1 + ceil(logM-1(N0)) (including the initial run creation pass)
  • Total block transfers: 2 × br × (1 + ceil(logM-1(ceil(br/M))))

The factor of 2 arises because each pass reads and writes all blocks. The final pass write may not be counted if the output is pipelined.

5.3 Worked Example

Example: Sort a relation with br = 1000 blocks using M = 10 buffer blocks.
  • Initial runs: ceil(1000/10) = 100 runs, each of 10 blocks
  • Pass 1: Merge 9 runs at a time → ceil(100/9) = 12 runs
  • Pass 2: Merge 9 at a time → ceil(12/9) = 2 runs
  • Pass 3: Merge 2 runs → 1 final sorted run
  • Number of merge passes: ceil(log9(100)) = ceil(2.096) = 3
  • Total passes: 1 + 3 = 4
  • Total block transfers: 2 × 1000 × 4 = 8000

6. Evaluation of Expressions

A query typically involves multiple operations (selections, joins, projections). There are two broad approaches to evaluating the full expression.

6.1 Materialization

Each intermediate operation is computed in full and its result is materialized (stored temporarily on disk) before being used as input to the next operation.

  • Advantage: Simple to implement; each operation can use any algorithm.
  • Disadvantage: High I/O cost due to writing and reading intermediate results.

6.2 Pipelining

Tuples are passed directly from one operation to the next as they are generated, without being written to disk. Operations run concurrently.

  • Advantage: Eliminates the cost of writing/reading intermediate results. Reduced memory usage.
  • Disadvantage: Not always possible (e.g., sorting requires all input before producing output). More complex to implement.

Types of Pipelining

Type Description Mechanism
Demand-driven (lazy/pull) Each operator requests tuples from its child when needed Iterator model with open(), next(), close()
Producer-driven (eager/push) Each operator pushes tuples to its parent as soon as available Uses buffers between operators
Blocking vs Non-blocking operators: A blocking operator (e.g., sort, hash-join build phase) must consume all its input before producing any output. Non-blocking operators (e.g., select, project) can produce output tuple-by-tuple. Pipelining is most effective with non-blocking operators.

7. Query Optimization

7.1 Equivalence Rules for Relational Algebra

The query optimizer uses algebraic equivalence rules to transform a query expression into different but equivalent forms. The key equivalence rules are:

#RuleDescription
1 σθ1 AND θ2(E) = σθ1(σθ2(E)) Cascade of selections
2 σθ1(σθ2(E)) = σθ2(σθ1(E)) Commutativity of selection
3 ΠL1(ΠL2(...(ΠLn(E))...)) = ΠL1(E) Cascade of projections (if L1 ⊆ L2 ⊆ ... ⊆ Ln)
4 E1 ⋈ E2 = E2 ⋈ E1 Commutativity of join
5 (E1 ⋈ E2) ⋈ E3 = E1 ⋈ (E2 ⋈ E3) Associativity of join
6 σθ(E1 ⋈ E2) = σθ(E1) ⋈ E2 Selection pushdown (if θ involves only E1's attributes)
7 ΠL(E1 ⋈ E2) = ΠL1(E1) ⋈ ΠL2(E2) Projection pushdown (with appropriate attribute sets)
8 E1 ∪ E2 = E2 ∪ E1 Commutativity of union
9 E1 ∩ E2 = E2 ∩ E1 Commutativity of intersection
10 σθ(E1 - E2) = σθ(E1) - E2 Selection distributes over set difference

7.2 Heuristic Optimization

Heuristic (rule-based) optimization applies a set of transformation rules that are almost always beneficial, without estimating actual costs:

  1. Push selections down: Perform selections as early as possible to reduce the size of intermediate results. This is the single most important heuristic.
  2. Push projections down: Eliminate unnecessary attributes early to reduce tuple width.
  3. Perform most restrictive selections first: Among multiple selections, apply the one that eliminates the most tuples first.
  4. Avoid Cartesian products: Replace Cartesian products followed by selections with join operations.
  5. Choose optimal join ordering: For multi-way joins, the order of joins can dramatically affect performance.

7.3 Cost-Based Optimization

Cost-based optimization uses statistical information from the system catalog to estimate the cost of different evaluation plans and selects the cheapest.

Statistics Used

  • nr: number of tuples in relation r
  • br: number of blocks
  • V(A, r): number of distinct values of attribute A
  • Histogram information (distribution of values)
  • Index information (type, height, etc.)

Size Estimation

The optimizer estimates the size of intermediate results:

OperationEstimated Size (tuples)
σA=v(r)nr / V(A,r)
σA≤v(r)nr × (v - min(A,r)) / (max(A,r) - min(A,r)) (uniform assumption)
r ⋈ s (on common attribute)nr × ns / max(V(A,r), V(A,s))
r × snr × ns
ΠA(r)V(A,r)
GATE Important: The join size estimation formula nr × ns / max(V(A,r), V(A,s)) is frequently tested. Know it cold.

Join Ordering

For n relations, there are (2(n-1))! / (n-1)! possible join orderings (Catalan number growth). For n=4, that is already 120 possible orderings. Dynamic programming is used to find the optimal plan in O(3n) time by building up optimal plans for subsets of relations.

8. Query Execution Plans

A query execution plan (or evaluation plan) specifies not just the order of operations but also the algorithm to use for each operation. It includes:

  • The order in which operations are performed
  • The algorithm for each operation (e.g., hash join vs sort-merge join)
  • How results are passed between operations (materialization vs pipelining)
  • Which indices to use

9. Worked Examples with Cost Calculations

Example 1: Choosing a Selection Algorithm

Given: Relation student with nr = 10,000 tuples, fr = 20 tuples per block, so br = 500 blocks. There is a B+ tree index on dept_name with height hi = 3. V(dept_name, student) = 25.

Query: σdept_name='CS'(student)

Solution:
Option A - Linear scan: Cost = 500 block transfers
Option B - B+ tree index (secondary): Cost = hi + sc(dept_name, student) = 3 + 10000/25 = 3 + 400 = 403 block transfers

Index scan (403) is better than linear scan (500) in this case.

But if V(dept_name, student) = 5, then index cost = 3 + 10000/5 = 2003, which is much worse than linear scan. This illustrates why the optimizer must consider selectivity.

Example 2: Choosing a Join Algorithm

Given:
  • Relation r: br = 100 blocks, nr = 5000 tuples
  • Relation s: bs = 400 blocks, ns = 10000 tuples
  • Available memory: M = 52 blocks
Compare join costs:

Block nested-loop join (r as outer):
Cost = ceil(100/50) × 400 + 100 = 2 × 400 + 100 = 900 block transfers

Block nested-loop join (s as outer):
Cost = ceil(400/50) × 100 + 400 = 8 × 100 + 400 = 1200 block transfers

Hash join:
Cost = 3 × (100 + 400) = 1500 block transfers
Memory check: sqrt(100) + 1 = 11 ≤ 52, so hash join is feasible.

Sort-merge join:
Sort r: 2 × 100 × (1 + ceil(log51(2))) = 2 × 100 × 2 = 400
Sort s: 2 × 400 × (1 + ceil(log51(8))) = 2 × 400 × 2 = 1600
Merge: 100 + 400 = 500
Total: 400 + 1600 + 500 = 2500 block transfers

Winner: Block nested-loop join with r as outer = 900 block transfers.

Example 3: Heuristic Optimization

Query: 
SELECT name
FROM student, takes, course
WHERE student.ID = takes.ID
  AND takes.course_id = course.course_id
  AND course.dept_name = 'CS'
  AND student.tot_cred > 100
Initial relational algebra:
Πname(σcourse.dept='CS' AND student.tot_cred>100(student × takes × course))

After heuristic optimization:
1. Push selections down:
Πname((σtot_cred>100(student)) ⋈ID takes ⋈course_id (σdept='CS'(course)))

2. Push projections down:
Πname((ΠID,name(σtot_cred>100(student))) ⋈ID (ΠID,course_id(takes)) ⋈course_id (Πcourse_id(σdept='CS'(course))))

This dramatically reduces the size of intermediate results.

10. Summary of Key Points for GATE

  • Know all selection algorithm costs by heart.
  • Block nested-loop join with M buffers: ceil(br/(M-2)) × bs + br
  • Hash join cost: 3(br + bs) (no overflow).
  • External sort: total passes = 1 + ceil(logM-1(ceil(br/M))).
  • Join size estimation: nr × ns / max(V(A,r), V(A,s)).
  • Pushing selections and projections down is the primary heuristic optimization.
  • Pipelining avoids materialization costs but is not always possible (blocking operators).

Operating System

Real-Time Scheduling and Advanced CPU Scheduling

Reference: Silberschatz, Galvin, Gagne — Operating System Concepts (Chapters 5–6)

This article covers advanced CPU scheduling topics that are heavily tested in GATE CS: real-time scheduling algorithms (Rate Monotonic and Earliest Deadline First), multiprocessor scheduling strategies, thread scheduling, the Linux CFS scheduler, and the critical problem of priority inversion. Each concept is explained with rigorous detail, formulas, diagrams, and worked numerical problems.

1. Real-Time Systems: Overview

A real-time system is one in which the correctness of a computation depends not only on its logical result but also on the time at which the result is produced. Missing a deadline can range from inconvenient to catastrophic.

1.1 Hard Real-Time vs Soft Real-Time

Property Hard Real-Time Soft Real-Time
Deadline Must ALWAYS be met; missing a deadline = system failure Deadlines are desirable but not critical; occasional misses are tolerable
Consequence of miss Catastrophic (e.g., aircraft control, pacemaker) Degraded quality (e.g., video buffering, audio glitch)
Examples Anti-lock braking (ABS), nuclear reactor control, airbag deployment Multimedia streaming, VoIP, online gaming
OS Support Specialized RTOS (VxWorks, QNX, FreeRTOS) General-purpose OS with priority scheduling (Linux with PREEMPT_RT)
Scheduling guarantee Must provide schedulability guarantee before task admission Best-effort scheduling with priority over non-real-time tasks

1.2 Real-Time Task Model

A periodic real-time task Ti is characterized by:

  • Period (pi): The task repeats every pi time units.
  • Computation time / Execution time (Ci): Worst-case execution time (WCET) per invocation.
  • Deadline (di): The task must complete within di time units of its release. Often di = pi (implicit deadline).
  • Utilization (Ui): Ui = Ci / pi — the fraction of CPU time consumed by task Ti.

The total CPU utilization is:

U = sum(C_i / p_i) for all tasks i = 1 to n
Necessary Condition for Schedulability: For any scheduling algorithm, U ≤ 1 is necessary (you cannot use more than 100% of the CPU). If U > 1, no scheduling algorithm can meet all deadlines.

2. Rate Monotonic Scheduling (RMS)

Rate Monotonic Scheduling is a static-priority, preemptive scheduling algorithm for periodic real-time tasks, proposed by Liu and Layland (1973).

2.1 Priority Assignment Rule

RMS Rule: Tasks with shorter periods get higher priority. Priority is inversely proportional to the period.

If p1 < p2 < p3, then Priority(T1) > Priority(T2) > Priority(T3).

Priorities are fixed (static) and assigned at design time. They do not change during execution.

2.2 Schedulability Test (Liu-Layland Bound)

RMS can guarantee that all tasks meet their deadlines if:

U = sum(C_i / p_i) ≤ n * (2^(1/n) - 1)

where n is the number of tasks. This is a sufficient condition (not necessary). If the condition holds, the task set is definitely schedulable. If it does not hold, the task set may or may not be schedulable (further analysis needed).

GATE Tip: As n → ∞, the utilization bound converges to ln(2) ≈ 0.693 (about 69.3%). This means RMS can always guarantee schedulability if total utilization is at most 69.3%, regardless of the number of tasks.

Given:

  • Task T1: C1 = 1, p1 = 4

2.4 Worked Example 2: Three Tasks under RMS (Fails Bound but Still Schedulable)

Given:

  • T1: C1 = 1, p1 = 3
  • T2: C2 = 1, p2 = 5
  • T3: C3 = 1, p3 = 7

Step 1: Compute utilization

U = 1/3 + 1/5 + 1/7 = 0.333 + 0.200 + 0.143 = 0.676

Step 2: Check Liu-Layland bound for n = 3

Bound = 3 * (2^(1/3) - 1) = 3 * 0.2599 = 0.780

Since 0.676 ≤ 0.780, the task set passes the utilization bound test and is guaranteed schedulable under RMS.

2.5 Worked Example 3: RMS Failure Case

Given:

  • T1: C1 = 1, p1 = 2
  • T2: C2 = 2, p2 = 5

Step 1: Compute utilization

U = 1/2 + 2/5 = 0.5 + 0.4 = 0.9

Step 2: Check Liu-Layland bound for n = 2

Bound = 2 * (2^(1/2) - 1) = 0.828

Since 0.9 > 0.828, the bound test fails. But remember, the bound is only a sufficient condition. We need to check the actual schedule.

Step 3: Construct the schedule (LCM = 10)

  • t=0: T1 released, T2 released. T1 runs (higher priority, shorter period).
  • t=1: T1 completes. T2 starts.
  • t=2: T1 released again. T2 has run 1 unit (needs 1 more). T1 preempts T2.
  • t=3: T1 completes. T2 resumes (1 unit remaining).
  • t=4: T2 completes. (Deadline was t=5. Met!)
  • t=4-5: CPU idle.
  • t=5: T2 released again. T2 starts.
  • t=6: T1 released. T2 has run 1 unit. T1 preempts.
  • t=7: T1 completes. T2 resumes (1 unit remaining).
  • t=8: T1 released. T2 completes at t=8. T1 starts.
  • t=9: T1 completes. Idle.
  • t=10: T2 deadline at t=10. T2 completed at t=8. Met!
Result: Despite failing the utilization bound test, this task set IS schedulable under RMS. This illustrates that the Liu-Layland bound is sufficient but not necessary. To definitively determine schedulability when the bound fails, construct the actual schedule over the hyperperiod (LCM of all periods).

3. Earliest Deadline First (EDF)

EDF is a dynamic-priority, preemptive scheduling algorithm. It assigns priorities based on absolute deadlines: the task with the nearest (earliest) deadline gets the highest priority.

3.1 Key Properties

  • Dynamic priorities: A task's priority changes over time as new instances arrive with new deadlines.
  • Optimal: EDF is optimal among all preemptive scheduling algorithms for uniprocessor systems. If any algorithm can schedule a task set, EDF can too.
  • Preemptive: When a new task arrives with an earlier deadline than the currently running task, it preempts immediately.

3.2 Schedulability Condition

EDF Schedulability (Necessary and Sufficient for Implicit-Deadline Tasks): 
U = sum(C_i / p_i) ≤ 1
A task set with implicit deadlines (di = pi) is schedulable under EDF if and only if the total utilization U ≤ 1.

This is a much better bound than RMS (which converges to 69.3%). EDF can fully utilize the CPU (100%).

3.3 Worked Example 1: Two Tasks under EDF

Given:

  • T1: C1 = 1, p1 = 2
  • T2: C2 = 3, p2 = 5

Utilization:

U = 1/2 + 3/5 = 0.5 + 0.6 = 1.1 > 1

Since U > 1, this task set is NOT schedulable under any algorithm, including EDF.

3.5 Worked Example 3: Detailed EDF vs RMS Comparison

Given:

  • T1: C1 = 1, p1 = 3
  • T2: C2 = 3, p2 = 8

Utilization: U = 1/3 + 3/8 = 0.333 + 0.375 = 0.708

RMS check (n=2): 0.708 ≤ 0.828 ⇒ Schedulable under RMS.

EDF check: 0.708 ≤ 1.0 ⇒ Schedulable under EDF.

Both algorithms work here. Under RMS, T1 always has higher priority (period 3 < 8). Under EDF, priorities shift dynamically based on nearest deadline.

4. RMS vs EDF: Comprehensive Comparison

Property RMS (Rate Monotonic) EDF (Earliest Deadline First)
Priority Type Static (fixed at design time) Dynamic (changes at runtime)
Priority Rule Shorter period = higher priority Earlier absolute deadline = higher priority
Optimality Optimal among static-priority algorithms Optimal among all preemptive algorithms (uniprocessor)
Schedulability Bound n(21/n - 1); converges to ~69.3% 100% (U ≤ 1 is both necessary and sufficient)
CPU Utilization Leaves up to ~30.7% CPU unused in worst case Can fully utilize the CPU
Implementation Complexity Simple — priorities never change More complex — must track absolute deadlines, recompute priorities
Overload Behavior Low-priority tasks miss deadlines first (predictable) Unpredictable domino effect — multiple tasks can miss deadlines
Context Switches Fewer (static priorities, less preemption) More (priorities change, more preemptions)
Practical Usage Widely used in RTOS (safety-critical systems) Used when high utilization is needed
GATE Tip: If the question asks "which algorithm can schedule this task set?" — check EDF first (U ≤ 1). If yes, EDF can schedule it. For RMS, you need the tighter bound. RMS is optimal among static-priority algorithms, meaning no other fixed-priority scheme can do better.

5. Proportional Share Scheduling

In proportional share (or fair share) scheduling, each process receives a guaranteed share of the CPU proportional to assigned shares or tickets.

Example: If process A has 50 shares and process B has 150 shares out of a total of 200 shares:

  • A gets 50/200 = 25% of CPU time
  • B gets 150/200 = 75% of CPU time

Lottery Scheduling is a probabilistic implementation: each process holds lottery tickets, and at each scheduling decision, a ticket is drawn randomly. Over time, the proportion converges to the share ratio.

Admission Control: A new process requesting t shares is admitted only if the total shares including t do not exceed the CPU capacity.

6. Multiprocessor Scheduling

When a system has multiple CPUs (or cores), scheduling becomes more complex. We must decide how to assign processes to processors and how to balance load.

6.2 Processor Affinity

When a process runs on a particular processor, the processor's cache becomes populated with the process's data (warm cache). Migrating the process to another processor means starting with a cold cache, which hurts performance. Processor affinity refers to the tendency to keep a process on the same processor.

Type Description Implementation
Soft Affinity OS tries to keep process on same CPU but may migrate if needed for load balancing Default behavior in most SMP OSes
Hard Affinity Process is bound to a specific CPU (or set of CPUs) and will NOT be migrated Linux: sched_setaffinity() or taskset command

6.3 Load Balancing

In SMP systems with per-processor queues, load can become uneven. Load balancing redistributes processes across CPUs.

Strategy Mechanism Trigger
Push Migration A dedicated task periodically checks load on each CPU. If imbalanced, it pushes processes from overloaded CPUs to underloaded ones. Periodic (timer-based)
Pull Migration An idle CPU pulls a waiting process from a busy CPU's queue. Event-driven (when CPU becomes idle)
GATE Tip: Push and pull migration are not mutually exclusive. Linux uses both. Load balancing often conflicts with processor affinity — the OS must balance these competing goals.

6.4 NUMA-Aware Scheduling

In Non-Uniform Memory Access (NUMA) systems, each CPU has its own local memory, and accessing remote memory (attached to another CPU) is slower.

graph LR subgraph "NUMA Node 0" CPU0["CPU 0"] MEM0["Memory 0"] CPU0 <-->|"fast"| MEM0 end subgraph "NUMA Node 1" CPU1["CPU 1"] MEM1["Memory 1"] CPU1 <-->|"fast"| MEM1 end CPU0 <-->|"slow (interconnect)"| MEM1 CPU1 <-->|"slow (interconnect)"| MEM0

NUMA-aware scheduling means:

  • Schedule a process on a CPU that is in the same NUMA node as its memory allocation.
  • Allocate memory on the NUMA node where the process is running.
  • Avoid migrating processes across NUMA nodes (as it separates process from its memory).
Key Insight: In NUMA systems, processor affinity becomes even more important because migrating a process to a different NUMA node means all its memory accesses become remote (slower). The scheduler must be NUMA-aware to achieve good performance.

7. Thread Scheduling: PCS vs SCS

Thread scheduling depends on the threading model. The OS scheduler schedules kernel threads (not user threads directly).

Property Process-Contention Scope (PCS) System-Contention Scope (SCS)
Scope Threads within the SAME process compete for CPU Threads across ALL processes compete for CPU
Used With Many-to-one and many-to-many models One-to-one model
Scheduler User-level thread library schedules user threads onto available LWPs (kernel threads) OS kernel directly schedules each thread onto a CPU
Priority Priority set by programmer; competition within process Priority used by kernel; competition system-wide
Examples Solaris (M:M model) Linux, Windows (1:1 model)
Linux Threading: Linux uses the 1:1 model (each user thread maps to one kernel thread), so all thread scheduling is SCS. The kernel treats each thread as an independently schedulable entity via the clone() system call.

8. Linux CFS (Completely Fair Scheduler)

The Completely Fair Scheduler (CFS) has been the default Linux CPU scheduler since kernel 2.6.23 (2007). It aims to allocate CPU time fairly among all runnable tasks.

8.1 Key Concepts

  • Virtual Runtime (vruntime): Each task has a vruntime counter that tracks how much CPU time it has received, weighted by its priority (nice value). A task with lower priority accumulates vruntime faster.
  • Scheduling Decision: CFS always picks the task with the smallest vruntime to run next.
  • Red-Black Tree: Runnable tasks are stored in a self-balancing red-black tree, ordered by vruntime. The leftmost node (smallest vruntime) is selected in O(1) time (cached). Insertion and deletion are O(log n).
  • Time Slice: CFS does not use fixed time quanta. Instead, each task's "time slice" is proportional to its weight (priority) relative to all runnable tasks. The targeted latency (sched_latency) defines the period in which every runnable task should run at least once.
  • Nice Values: Range from -20 (highest priority) to +19 (lowest priority). Default is 0. Each nice level corresponds to roughly a 10% change in CPU share (actually a factor of 1.25).

8.2 CFS Weight Calculation

The weight of a task with nice value n is approximately:

weight(n) = 1024 / (1.25^n)

Examples:
  nice 0  -> weight = 1024
  nice 1  -> weight = 1024 / 1.25 = 820 (approximately)
  nice -1 -> weight = 1024 * 1.25 = 1280 (approximately)

The proportion of CPU time a task receives:

CPU_share(i) = weight(i) / sum(weight(j)) for all runnable tasks j

Example: Two tasks, A (nice 0, weight 1024) and B (nice 5, weight ~335):

CPU_share(A) = 1024 / (1024 + 335) = 75.3%
CPU_share(B) = 335 / (1024 + 335) = 24.7%

8.3 CFS Virtual Runtime Update

When a task runs for delta actual time:

vruntime += delta * (weight_of_nice_0 / weight_of_this_task)

For nice 0 task:  vruntime += delta * (1024/1024) = delta
For nice 5 task:  vruntime += delta * (1024/335)  = delta * 3.06

Higher-nice (lower priority) tasks accumulate vruntime faster, causing them to be selected less often.

Key Properties of CFS:
  • No fixed time quantum — adapts to system load
  • O(log n) scheduling decisions (red-black tree)
  • Fair — all tasks of equal priority get equal CPU time
  • Starvation-free — every runnable task will eventually have the smallest vruntime
  • Does not distinguish between I/O-bound and CPU-bound tasks explicitly

9. Priority Inversion and Priority Inheritance Protocol

Priority inversion is a scheduling anomaly where a higher-priority task is indirectly blocked by a lower-priority task, because a medium-priority task preempts the lower-priority task that holds a resource needed by the higher-priority task.

9.1 The Classic Priority Inversion Scenario

Consider three tasks: TH (high priority), TM (medium priority), TL (low priority), and a shared resource R protected by a lock.

  1. TL starts running and acquires lock on R.
  2. TH arrives, preempts TL, and tries to acquire R → blocked (TL holds it).
  3. TM arrives. Since TH is blocked, TM runs (higher priority than TL).
  4. TM runs to completion before TL can resume and release R.
  5. Only after TM finishes can TL resume, release R, and finally TH runs.

Result: TH (highest priority) is effectively blocked by TM (medium priority), which doesn't even use the shared resource. This is unbounded priority inversion — the blocking time depends on medium-priority tasks, which can be arbitrarily long.

gantt title Priority Inversion Example dateFormat X axisFormat %s section T_H (High) Blocked waiting for R :crit, h1, 3, 9 Running :h2, 9, 11 section T_M (Medium) Running :m1, 3, 8 section T_L (Low) Running + holds R :l1, 0, 3 Preempted :done, l2, 3, 8 Resumes + releases R :l3, 8, 9
Mars Pathfinder Bug (1997): The Mars Pathfinder rover experienced repeated system resets due to priority inversion. A low-priority meteorological task held a shared bus lock, was preempted by medium-priority tasks, and this blocked the high-priority bus management task. The watchdog timer detected the delay and triggered a system reset. NASA fixed it by enabling priority inheritance remotely.

9.2 Priority Inheritance Protocol (PIP)

Solution: When a high-priority task is blocked by a lower-priority task holding a resource, temporarily raise the priority of the lower-priority task to that of the highest-priority task waiting for the resource.

With priority inheritance applied to the scenario above:

  1. TL starts, acquires lock on R.
  2. TH arrives, tries to acquire R, is blocked.
  3. TL's priority is raised to TH's priority level.
  4. TM arrives but cannot preempt TL (because TL now has TH's high priority).
  5. TL finishes its critical section, releases R, and its priority drops back to normal.
  6. TH acquires R and runs.
  7. After TH finishes, TM runs.

9.3 Priority Ceiling Protocol (PCP)

An alternative to PIP that also prevents deadlocks:

  • Each resource is assigned a priority ceiling = the highest priority of any task that may access it.
  • A task can lock a resource only if its priority is strictly higher than the ceiling of every resource currently locked by other tasks.
  • This prevents circular waiting and thus prevents deadlocks.
Property Priority Inheritance (PIP) Priority Ceiling (PCP)
Priority boost Only when actual blocking occurs Preemptive; based on resource ceilings
Deadlock prevention Does NOT prevent deadlocks Prevents deadlocks
Blocking bound May experience chained blocking At most one blocking per critical section
Complexity Simpler to implement Requires knowing all resource-task relationships at design time

Problem 3: Determining Maximum Computation Time

Given: Two tasks under RMS. T1: C1 = 2, p1 = 5. T2: p2 = 10. What is the maximum C2 such that the task set is guaranteed schedulable under RMS?

Solution: Using the Liu-Layland bound for n = 2:

U ≤ 2(2^(1/2) - 1) = 0.828

C1/p1 + C2/p2 ≤ 0.828
2/5 + C2/10 ≤ 0.828
0.4 + C2/10 ≤ 0.828
C2/10 ≤ 0.428
C2 ≤ 4.28

Since C2 must be an integer: C2_max = 4

Verification: U = 2/5 + 4/10 = 0.4 + 0.4 = 0.8 ≤ 0.828. Schedulable.

Problem 4: CFS CPU Share Calculation

Given: Three processes with nice values: A (nice = 0), B (nice = 5), C (nice = -5). Calculate each process's CPU share.

Solution: Using approximate weights (weight(n) ≈ 1024 / 1.25n):

weight(A) = 1024 / 1.25^0  = 1024
weight(B) = 1024 / 1.25^5  = 1024 / 3.052 = 335
weight(C) = 1024 / 1.25^-5 = 1024 * 3.052 = 3125

Total weight = 1024 + 335 + 3125 = 4484

CPU_share(A) = 1024 / 4484 = 22.8%
CPU_share(B) = 335 / 4484  = 7.5%
CPU_share(C) = 3125 / 4484 = 69.7%

Process C (nice = -5, highest priority) gets the lion's share of CPU time.

Problem 5: Priority Inversion Duration

Given: TH (high), TM1, TM2 (medium), TL (low). TL holds resource R for 5 time units. At t=1, TH arrives and needs R. At t=2, TM1 arrives (execution time 3). At t=4, TM2 arrives (execution time 2). Without priority inheritance, how long is TH blocked?

Solution (no priority inheritance):

  • t=0: TL starts, acquires R.
  • t=1: TH arrives, needs R, blocked.
  • t=2: TM1 arrives, preempts TL. TM1 runs t=2 to t=5.
  • t=4: TM2 arrives, but TM1 is running (same or higher priority among mediums; say same priority, FCFS). TM2 waits.
  • t=5: TM1 finishes. TM2 runs t=5 to t=7.
  • t=7: TL resumes. TL had run 2 units (t=0 to t=2), needs 3 more (if total critical section = 5 units total including initial 1 unit before TH arrived). TL was in critical section from t=0, ran 1 unit before TH at t=1, then preempted. Actually TL ran from 0 to 1 (1 unit done), then TH arrives and blocks, TL continues but TM1 preempts at t=2. TL ran 1 more unit (t=1 to t=2). So TL has done 2 of 5 units. Needs 3 more.
  • t=7 to t=10: TL completes, releases R.
  • t=10: TH finally gets R and runs.

TH was blocked from t=1 to t=10 = 9 time units.

With priority inheritance: TL inherits TH's priority at t=1. TM1 and TM2 cannot preempt. TL finishes R at t=0+5=5 (ran 1 unit before t=1, inherits at t=1, continues uninterrupted for 4 more = t=5). TH blocked from t=1 to t=5 = 4 time units. Much better!

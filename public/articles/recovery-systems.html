Databases

Recovery Systems and Log-Based Recovery

A database system must be prepared to handle failures at any time. Transactions may be interrupted, the system may crash, or storage media may fail. The recovery system is responsible for restoring the database to a consistent state after such failures. This article covers Chapter 19 of Korth/Silberschatz's Database System Concepts and is a core topic for GATE CSE.

1. Failure Classification

Failures in a database system can be classified into three broad categories based on their severity and the type of storage they affect:

1.1 Transaction Failure

A transaction fails to complete its execution. This can occur due to:

  • Logical errors: The transaction cannot complete due to some internal condition (e.g., bad input, overflow, resource limit exceeded).
  • System errors: The DBMS must abort the transaction due to an undesirable condition such as deadlock.

1.2 System Failure (System Crash)

A hardware malfunction, a bug in the database software or operating system, or a power failure causes the loss of the content of volatile storage (main memory). The contents of non-volatile storage (disk) survive.

  • The assumption is called the fail-stop assumption: the system halts and does not perform any further actions.
  • All data in memory buffers that were not yet written to disk are lost.

1.3 Media Failure (Disk Failure)

A disk block or an entire disk loses its contents. This is the most catastrophic failure type.

  • Handled by maintaining archival dumps (backups) and stable storage.
  • Recovery from media failure is typically slower and may require restoring from backup.

2. Storage Types

The recovery system's design is closely linked to the types of storage available:

Storage Type Survives Examples Characteristics
Volatile Storage Neither system crash nor media failure Main memory, cache Fast access, data lost on power failure
Non-volatile Storage System crash, but NOT media failure Hard disk, SSD, flash Survives power failure; can lose data on disk failure
Stable Storage All failure types (theoretically) RAID, mirrored disks, remote backup Approximated by replication; never truly achievable but practically reliable
Key Point: Stable storage is a theoretical concept. In practice, it is approximated using techniques like RAID (mirroring) and remote backup. The log is always stored on stable storage (or its approximation).

3. Recovery and Atomicity

The fundamental challenge of recovery is to guarantee the atomicity property of transactions: either all effects of a transaction are reflected in the database, or none are.

If a transaction Ti has committed, its effects must persist (durability). If it has aborted or was active at the time of failure, all its effects must be undone. The recovery system must be able to determine for each transaction whether to redo or undo its operations.

3.1 The Problem

Consider this scenario:

T1: Read(A)         -- A = 1000
    A = A - 50
    Write(A)        -- A = 950 (in buffer)
    Read(B)         -- B = 2000
    B = B + 50
    Write(B)        -- B = 2050 (in buffer)
    Commit

--- System crash here ---
--- Buffer for A was flushed to disk, but buffer for B was NOT ---

After the crash, A = 950 on disk and B = 2000 on disk. The sum has changed from 3000 to 2950, violating consistency. Without a recovery mechanism, we cannot fix this.

4. Log-Based Recovery

The most widely used technique for recovery. A log (also called a journal) is a sequence of log records that records all update activities in the database. The log is stored on stable storage.

4.1 Log Record Format

Each log record has one of the following forms:

Log Record Meaning
<Ti, start> Transaction Ti has started
<Ti, X, Vold, Vnew> Ti has written data item X; old value was Vold, new value is Vnew
<Ti, commit> Transaction Ti has committed
<Ti, abort> Transaction Ti has aborted
<checkpoint L> Checkpoint with list L of active transactions
Note: Some formulations use <Ti, X, Vold> for undo-only logging and <Ti, X, Vnew> for redo-only logging. The general form with both old and new values supports both undo and redo.

4.2 Write-Ahead Logging (WAL) Protocol

The Write-Ahead Logging (WAL) protocol is the cornerstone of log-based recovery. It mandates:

  1. Undo rule: Before a data item is written to the database on disk, the log record containing the old value must be flushed to stable storage.
  2. Redo rule: Before a transaction commits, all its log records (including the commit record) must be flushed to stable storage.
WAL Guarantee: WAL ensures that even if the system crashes at any point, the log on stable storage has enough information to undo uncommitted changes and redo committed changes.

Mnemonic: "Log before data, commit log before declaring committed."

4.3 Deferred Database Modification

In this scheme, the actual database writes are deferred until the transaction commits. During execution, all writes go to the log only.

  • When a transaction commits, the log records are used to perform the actual writes to the database.
  • If the transaction aborts or the system crashes before commit, no undo is needed because the database was never modified.
  • Recovery: Only redo is needed for committed transactions whose updates may not have reached disk.

Recovery Procedure (Deferred Modification)

redo(T_i):
    For each log record <T_i, X, V_new> in order:
        Write V_new to data item X on disk

Recovery:
    For each transaction T_i:
        if <T_i, commit> is in the log:
            redo(T_i)
        else:
            ignore T_i  (no undo needed)

4.4 Immediate Database Modification

In this scheme, database modifications are written to disk before the transaction commits (but after the log record is written, per WAL). This means uncommitted changes may be on disk.

  • If a transaction commits, its changes are already (partially or fully) on disk.
  • If a transaction aborts or crashes before commit, its changes may already be on disk and must be undone.
  • Recovery: Both undo and redo are needed.

Recovery Procedure (Immediate Modification)

undo(T_i):
    For each log record <T_i, X, V_old, V_new> in REVERSE order:
        Write V_old to data item X on disk
    Write <T_i, abort> to log

redo(T_i):
    For each log record <T_i, X, V_old, V_new> in FORWARD order:
        Write V_new to data item X on disk

Recovery:
    For each transaction T_i:
        if <T_i, commit> is in the log:
            redo(T_i)
        else:
            undo(T_i)
Key Difference:
Deferred: No undo needed, only redo for committed transactions.
Immediate: Both undo (for uncommitted) and redo (for committed) needed.

5. Undo and Redo Operations

Understanding when undo and redo are needed:

Transaction State at Crash Action Required Reason
Committed, changes on disk None Already durable
Committed, changes NOT fully on disk Redo Must ensure durability
Not committed, changes on disk Undo Must ensure atomicity
Not committed, changes NOT on disk None No effect to reverse
Idempotency: Both undo and redo operations must be idempotent — applying them multiple times produces the same result as applying them once. This is crucial because the system may crash during recovery, and recovery must be restartable.

5.1 Undo Operation Details

  • Scan the log backwards (from most recent to oldest).
  • For each write record of the transaction, restore the old value.
  • After all writes are undone, write an <Ti, abort> record to the log.
  • Compensation Log Records (CLRs) may be written during undo to track progress.

5.2 Redo Operation Details

  • Scan the log forwards (from oldest to most recent).
  • For each write record of the committed transaction, apply the new value.
  • Redo is applied regardless of whether the change is already on disk (idempotency).

6. Checkpointing

6.1 Why Checkpoints Are Needed

Without checkpoints, recovery would require scanning the entire log from the beginning, which could take an impractically long time. Checkpoints limit the amount of log that needs to be examined during recovery.

6.2 Checkpoint Procedure

A simple checkpointing procedure:

  1. Stop accepting new transactions temporarily (in simple checkpointing).
  2. Wait for all currently active transactions to either commit or abort (in simple checkpointing).
  3. Flush all modified buffer blocks to disk (force-write all dirty pages).
  4. Write a <checkpoint> record to the log and flush the log to stable storage.
  5. Resume normal transaction processing.
Simple vs Fuzzy Checkpoints: The simple checkpoint described above requires stopping all processing, which is impractical. In practice, fuzzy checkpoints (or non-quiescent checkpoints) are used, which allow transactions to continue during the checkpoint.

6.3 Non-Quiescent (Fuzzy) Checkpointing

A more practical approach that does not halt transaction processing:

  1. Write <checkpoint L> to the log, where L is the list of all currently active transactions.
  2. Flush the log to stable storage.
  3. Flush all dirty buffer pages to disk (can be done in the background).
  4. Write <checkpoint-end> when flushing is complete.

6.4 Recovery with Checkpoints

During recovery with checkpoints (immediate modification):

  1. Find the last checkpoint record in the log.
  2. The set of transactions to consider = transactions in list L at checkpoint + transactions that started after the checkpoint.
  3. Redo phase: Scan forward from the checkpoint, redo all updates of all transactions.
  4. Undo phase: Scan backward, undo updates of transactions that did not commit.
flowchart LR A["Log Start"] --> B["...older records..."] B --> C["<checkpoint {T1, T2}>"] C --> D["T1 writes"] D --> E["T3 start"] E --> F["T1 commit"] F --> G["T2 writes"] G --> H["T3 writes"] H --> I["CRASH"] style C fill:#d5f5e3 style I fill:#fdecea

In this scenario at crash time: T1 is committed (redo), T2 and T3 are not committed (undo). We only need to examine the log from the checkpoint record onwards.

7. ARIES Recovery Algorithm (Overview)

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) is the most widely used recovery algorithm in commercial database systems. It was developed at IBM Research.

7.1 Key Principles of ARIES

  1. Write-Ahead Logging (WAL): Log records written before data pages are flushed.
  2. Repeating History During Redo: During recovery, redo ALL actions (including those of uncommitted transactions) to bring the database to the exact state at the time of crash.
  3. Logging Changes During Undo: Compensation Log Records (CLRs) are written during undo to prevent repeated undo work if there is a crash during recovery.

7.2 ARIES Data Structures

Structure Purpose
Log Sequential record of all changes; each record has a unique LSN (Log Sequence Number)
Transaction Table Tracks active transactions and their last LSN
Dirty Page Table Tracks pages in buffer that have been modified but not yet written to disk. Records the recLSN (LSN of first log record that dirtied the page).
Page LSN Each page on disk stores the LSN of the last log record that modified it

7.3 ARIES Three-Phase Recovery

Phase 1: Analysis

Scan the log forward from the last checkpoint to determine:

  • Which transactions were active at the time of crash (need undo)
  • Which pages might be dirty (may need redo)
  • Rebuild the Transaction Table and Dirty Page Table

Phase 2: Redo

Scan the log forward from the smallest recLSN in the Dirty Page Table. For each update log record:

  • If the page is in the Dirty Page Table AND the record's LSN ≥ recLSN of the page AND the pageLSN on disk < the record's LSN, then redo the update.
  • This "repeats history" including updates of uncommitted transactions.

Phase 3: Undo

Scan the log backward, undoing all updates of transactions that were active at crash time. For each undo, write a Compensation Log Record (CLR).

  • CLRs have an undoNextLSN field pointing to the next record to be undone.
  • CLRs are never undone themselves (they are redo-only records).
  • This ensures that if the system crashes during recovery, undo work is not repeated.
flowchart LR A2["Last Checkpoint"] --> B2["Analysis Phase (forward scan)"] B2 --> C2["Redo Phase (forward from min recLSN)"] C2 --> D2["Undo Phase (backward scan)"] style B2 fill:#d5f5e3 style C2 fill:#fef9e7 style D2 fill:#fdecea
GATE Takeaway on ARIES: Remember the three phases (Analysis, Redo, Undo) and the key principle of "repeating history" during redo. Also remember that CLRs prevent repeated undo work.

8. Shadow Paging

Shadow paging is an alternative to log-based recovery that avoids writing a log altogether (in its simplest form).

8.1 How It Works

  • The database maintains two page tables: the current page table and the shadow page table.
  • At the start of a transaction, the shadow page table is a copy of the current page table. Both point to the same actual pages on disk.
  • When a page is modified, a new copy of the page is written to a different disk location, and the current page table is updated to point to the new page. The shadow page table still points to the old page.
  • To commit: The current page table becomes the new shadow page table (e.g., by writing its location to a fixed disk location). The old pages can be freed.
  • To abort/recover: Discard the current page table and revert to the shadow page table. The new pages can be freed.

8.2 Advantages and Disadvantages

Advantages Disadvantages
No log overhead (no WAL) Data fragmentation (pages scattered on disk)
Recovery is instant (just use shadow) Difficult to extend to concurrent transactions
Simple commit protocol Overhead of copying page table
No undo/redo needed Garbage collection of old pages needed
Practical Note: Shadow paging is rarely used in modern systems because of its limitations with concurrent transactions and data fragmentation. Log-based recovery (especially ARIES) is the dominant approach. However, shadow paging appears in GATE questions.

9. Recovery with Concurrent Transactions

In a real system, multiple transactions run concurrently, and the log is a single interleaved sequence of records from all transactions. Recovery must handle this correctly.

9.1 Classification During Recovery

At the time of a crash, transactions fall into categories:

  • Committed transactions: Have <Ti, commit> in the log. Must be redone.
  • Aborted transactions: Have <Ti, abort> in the log. Already handled.
  • Active (incomplete) transactions: Have <Ti, start> but neither commit nor abort. Must be undone.

9.2 Recovery Procedure with Concurrent Transactions

Recovery Algorithm:
1. Scan log forward from last checkpoint to end:
   - Build redo-list: all T_i with <T_i, commit> in log
   - Build undo-list: all T_i with <T_i, start> but no commit/abort

2. Redo phase (scan forward from checkpoint):
   For each log record <T_i, X, V_old, V_new>:
       if T_i is in redo-list:
           write V_new to X

3. Undo phase (scan backward from end of log):
   For each log record <T_i, X, V_old, V_new>:
       if T_i is in undo-list:
           write V_old to X
   Write <T_i, abort> for each T_i in undo-list

9.3 Interaction with Concurrency Control

If strict two-phase locking is used (locks held until commit/abort), then uncommitted changes are not visible to other transactions. This simplifies recovery because:

  • No cascading rollbacks are needed.
  • Undoing one transaction's changes does not affect other committed transactions.

10. Worked Examples: Crash Recovery Scenarios

Example 1: Simple Log Recovery

Given Log (Immediate Modification): 
<T1, start>
<T1, A, 100, 200>
<T2, start>
<T2, B, 300, 400>
<T1, C, 500, 600>
<T1, commit>
<T2, D, 700, 800>
--- CRASH ---
Analysis:
  • T1 has committed → Redo all of T1's updates
  • T2 has NOT committed → Undo all of T2's updates
Recovery Actions:
Redo (forward scan):
  • Set A = 200 (redo T1's write)
  • Set C = 600 (redo T1's write)
Undo (backward scan):
  • Set D = 700 (undo T2's write to D)
  • Set B = 300 (undo T2's write to B)
  • Write <T2, abort>
Final state: A=200, B=300, C=600, D=700

Example 2: Recovery with Checkpoint

Given Log: 
<T1, start>
<T1, A, 10, 20>
<T2, start>
<T2, B, 30, 40>
<checkpoint {T1, T2}>
<T3, start>
<T1, C, 50, 60>
<T1, commit>
<T3, D, 70, 80>
<T2, E, 90, 100>
<T2, commit>
--- CRASH ---
Analysis (from checkpoint):
  • Active at checkpoint: T1, T2
  • Started after checkpoint: T3
  • T1 committed → Redo
  • T2 committed → Redo
  • T3 NOT committed → Undo
Redo (forward from checkpoint):
  • Set C = 60 (T1)
  • Set D = 80 (T3 — we redo even uncommitted in ARIES; in simple recovery, skip this)
  • Set E = 100 (T2)
Note: In simple (non-ARIES) recovery, we only redo committed transactions. For T1 and T2, we must also redo records before the checkpoint since the checkpoint doesn't guarantee those pages were flushed.

Undo (backward from end):
  • Set D = 70 (undo T3)
  • Write <T3, abort>
Final state: A=20, B=40, C=60, D=70, E=100

Example 3: Deferred Modification Recovery

Given Log (Deferred Modification): 
<T1, start>
<T1, A, _, 500>
<T2, start>
<T1, B, _, 600>
<T2, C, _, 700>
<T1, commit>
<T2, D, _, 800>
--- CRASH ---
Analysis: With deferred modification, database writes are NOT performed until commit. So only T1's committed writes need to be redone. T2 never modified the database (its writes were deferred).

Recovery: Only redo T1.
  • Set A = 500
  • Set B = 600
T2 is simply ignored (no undo needed since database was never modified by T2).

Example 4: Multiple Crashes

Scenario: The system crashes during recovery. Is this a problem?

Answer: No! Because undo and redo are idempotent, recovery can simply be restarted from the beginning. In ARIES, Compensation Log Records (CLRs) written during the first recovery attempt ensure that completed undo work is not repeated, making repeated recovery efficient.

12. Summary of Key Points for GATE

  • WAL Protocol: Log record must be written to stable storage before the corresponding data page is written to disk. Commit record must be on stable storage before transaction is considered committed.
  • Deferred modification: Only redo needed (no undo). Database not modified until commit.
  • Immediate modification: Both undo and redo needed. Database modified before commit.
  • Checkpoint: Limits the extent of log scanning during recovery.
  • ARIES: Three phases (Analysis, Redo, Undo). Repeats history during redo. Uses CLRs during undo.
  • Idempotency: Undo and redo must be idempotent for crash-during-recovery safety.
  • Shadow paging: No log needed, but impractical for concurrent transactions.
  • Stable storage is theoretical; approximated by RAID/mirroring.

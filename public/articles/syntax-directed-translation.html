Syntax-Directed Translation and Intermediate Code

🔧 Section 7: Compiler Design

1. Syntax-Directed Definitions (SDD)

A Syntax-Directed Definition associates semantic rules (attribute computations) with grammar productions. Each grammar symbol has a set of attributes, and each production has a set of rules for computing attribute values.

1.1 Attributes

Attribute TypeDirectionComputed FromExample
SynthesizedBottom-upAttributes of children in parse treeE.val from T.val and E'.val
InheritedTop-down / SidewaysAttributes of parent and/or siblingsType info passed down from declaration

1.2 Example: Desk Calculator SDD

Production              Semantic Rule
-------------------------------------------
L -> E \n              print(E.val)
E -> E1 + T            E.val = E1.val + T.val
E -> T                 E.val = T.val
T -> T1 * F            T.val = T1.val * F.val
T -> F                 T.val = F.val
F -> ( E )             F.val = E.val
F -> digit             F.val = digit.lexval

All attributes here are synthesized — values flow from leaves to root.

2. S-Attributed and L-Attributed Definitions

2.1 S-Attributed Definition

  • Uses only synthesized attributes.
  • Can be evaluated in a single bottom-up pass (e.g., during LR parsing).
  • Semantic actions are placed at the end of productions.

2.2 L-Attributed Definition

  • Uses both synthesized and inherited attributes.
  • Restriction: inherited attributes of a symbol can depend only on attributes of symbols to its left in the production (or inherited attributes of the LHS).
  • Can be evaluated in a single left-to-right pass (e.g., during LL parsing).
  • Every S-attributed definition is also L-attributed.
GATE Tip: S-attributed SDDs are a subset of L-attributed SDDs. S-attributed works with bottom-up parsing; L-attributed works with top-down parsing. Every S-attributed SDD is L-attributed, but not vice versa.

3. Syntax-Directed Translation Schemes (SDT)

An SDT embeds semantic actions (program fragments) within the production bodies. Actions are enclosed in curly braces and execute when the parser reaches that position.

Production with embedded actions:
  E -> T { E'.inh = T.val } E' { E.val = E'.syn }
  E' -> + T { E1'.inh = E'.inh + T.val } E1' { E'.syn = E1'.syn }
  E' -> epsilon { E'.syn = E'.inh }

Postfix SDT (for S-attributed)

All actions are placed at the end of productions. Compatible with LR parsing — actions execute when reductions occur.

E -> E1 + T  { E.val = E1.val + T.val; }
T -> T1 * F  { T.val = T1.val * F.val; }
F -> digit   { F.val = digit.lexval; }

4. Runtime Environments

4.1 Storage Organization

At runtime, the program's memory is organized into regions:

  • Code area: The compiled machine code (fixed size, read-only).
  • Static area: Global variables, constants (fixed size, allocated at compile time).
  • Stack area: Activation records for function calls (grows and shrinks with calls/returns).
  • Heap area: Dynamically allocated memory (managed by malloc/free or garbage collector).

4.3 Stack Allocation

When a function is called, a new activation record is pushed onto the stack. When the function returns, its activation record is popped. This naturally supports recursion — each recursive call gets its own frame.

4.4 Heap Allocation

Used for data that must outlive the function call that creates it (e.g., objects in OOP, dynamically allocated arrays).

  • Fragmentation: External fragmentation occurs when free memory is scattered in small blocks.
  • Garbage collection: Automatically reclaims unreachable heap memory (mark-and-sweep, reference counting, generational GC).

5. Intermediate Code Representations

Intermediate code is a machine-independent representation that sits between the source language and target machine code. It facilitates optimization and portability.

5.1 Three-Address Code (TAC)

Each instruction has at most three operands (addresses). General form: x = y op z.

Source: a = b * c + d * e

Three-Address Code:
  t1 = b * c
  t2 = d * e
  t3 = t1 + t2
  a = t3

5.2 Quadruples

Four fields: (operator, arg1, arg2, result). Each instruction stored in a record.

Source: a = b * c + d

Quadruples:
  (0)  *, b, c, t1
  (1)  +, t1, d, t2
  (2)  =, t2, _, a

5.3 Triples

Three fields: (operator, arg1, arg2). The result is referenced by the instruction number (position).

Triples:
  (0)  *, b, c       // result is (0)
  (1)  +, (0), d     // result is (1)
  (2)  =, a, (1)     // a = (1)

5.4 Indirect Triples

A list of pointers to triples. This allows reordering instructions for optimization without renumbering references.

Comparison

6. Generating TAC for Control Flow

Source: if (a < b) { x = 1; } else { x = 2; }

Three-Address Code:
  if a < b goto L1
  goto L2
L1: x = 1
  goto L3
L2: x = 2
L3: ...
  
Source: while (a < b) { a = a + 1; }

Three-Address Code:
L1: if a < b goto L2
  goto L3
L2: t1 = a + 1
  a = t1
  goto L1
L3: ...

7. Key GATE Takeaways

  • SDDs attach semantic rules to grammar productions. Attributes are synthesized (bottom-up) or inherited (top-down).
  • S-attributed SDDs use only synthesized attributes and work with LR parsers.
  • L-attributed SDDs work with LL parsers and allow inherited attributes from left siblings only.
  • Activation records store function call information on the stack: parameters, locals, return address, links.
  • Three-address code is the most common intermediate representation. Each instruction has at most 3 operands.
  • Quadruples name results explicitly; triples use positional references; indirect triples allow easy reordering.
  • Know how to generate TAC for arithmetic expressions, if-else, and while loops.
